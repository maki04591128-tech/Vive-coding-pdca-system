## 📖 この文書について

> この文書は、AIサービス（LLM）が障害で使えなくなったときに、**別のAIサービスに自動で切り替える仕組み**の運用手順を定めたものです。  
> たとえるなら、「停電時の非常用電源への切替マニュアル」のようなものです。  
> メインのAI（クラウドLLM）が使えなくなったら、バックアップのAI（ローカルLLM）に自動で切り替わり、作業を続けられます。

### 🔰 この文書で使われる主要な用語

| 用語 | かんたんな説明 |
|------|-------------|
| **フォールバック** | メインのサービスが使えないとき、代わりのサービスに切り替えること |
| **クラウドLLM** | インターネット経由で使うAIサービス（ChatGPT、Claude等）。高性能だが料金がかかる |
| **ローカルLLM** | 自分のサーバーで動かすAI。インターネット不要だが性能はやや劣る |
| **サーキットブレーカー** | 障害が連鎖しないよう、問題のあるサービスへの接続を自動で遮断する仕組み |
| **CLI** | Command Line Interface。ターミナル（黒い画面）でコマンドを入力して操作する方式 |
| **モード** | AIサービスの利用方式。cloud（クラウド）/ local（ローカル）/ hybrid（併用）の3種類 |
| **vLLM** | ローカルでAIモデルを動かすサーバーソフトウェア |
| **Ollama** | 手元のPCで簡単にAIモデルを動かせるツール |

## 1. 概要

本ドキュメントは、クラウドLLM ↔ ローカルLLM の切替機構の運用手順を定義する。日常的なモード切替と障害発生時の対応手順の両方をカバーする。

---

## 2. モード切替（手動）

### 2.1 ローカルLLMモードへの切替

以下のいずれかの状況でローカルLLMモードへ切り替える。

- オフライン環境での作業
- コスト削減が必要な開発フェーズ
- クラウドLLMの計画メンテナンス

```bash
# CLI での切替
python -m vibe_pdca.cli mode set \
  --project-id <PROJECT_ID> \
  --llm-mode local \
  --reason "開発フェーズのためローカルLLMへ切替"
```

```python
# Python API での切替
from vibe_pdca.llm.models import ProviderType
gateway.set_mode(ProviderType.LOCAL, reason="開発フェーズのためローカルLLMへ切替")
```

### 2.2 クラウドLLMモードへの復帰

```bash
python -m vibe_pdca.cli mode set \
  --project-id <PROJECT_ID> \
  --llm-mode cloud \
  --reason "本番レビュー開始のためクラウドLLMへ復帰"
```

### 2.3 設定ファイルでの切替（再起動必要）

`config/default.yml` または環境別設定ファイルを編集する。

```yaml
llm:
  preferred_mode: "local"   # "cloud" → "local" に変更
```

## 3. 自動フォールバック

### 3.1 自動フォールバックの動作

自動フォールバックはデフォルトで有効であり、以下の流れで動作する。

1. クラウドLLMへのAPI呼び出しが失敗する
2. サーキットブレーカーが連続失敗を記録する
3. 連続失敗が `failure_threshold`（デフォルト3回）に達するとサーキットがOPENになる
4. 以降のリクエストは自動的にローカルLLMへルーティングされる
5. `recovery_timeout`（デフォルト60秒、本番推奨120秒 — ADR-008参照）経過後、サーキットがHALF_OPENになる
6. 試行リクエストが成功すれば、`success_threshold`回の成功後にサーキットがCLOSEDに戻る
7. 試行リクエストが失敗すれば、再びOPENに戻る

### 3.2 自動フォールバックの無効化

特定の状況で自動フォールバックを無効化する場合。

```python
gateway.set_auto_fallback(False)
```

無効化すると、クラウドLLM障害時は `CloudLLMUnavailableError` が発生しPDCAサイクルが停止する。

### 3.3 Discord通知

フォールバック発生時は `pdca-warning` チャネルに以下の通知が送信される。

```
【LLMフォールバック通知】
時刻        : 2026-03-03 14:30:22 UTC
対象プロバイダ: openai-gpt5.1
障害種別    : API接続タイムアウト
フォールバック先: ollama-llama3.3 (ローカル)
サーキット状態: OPEN
影響範囲    : PM役割のレビュー品質が低下する可能性あり
復旧見込み  : recovery_timeout=60秒後にHALF_OPENで試行
```

## 4. 障害対応手順

### 4.1 クラウドLLM単一プロバイダ障害

ADR-001 のフォールバック順に従い、同じ役割の代替クラウドプロバイダが自動的に使用される。ローカルLLMへのフォールバックは発生しない。

**対応**: 通知を確認し、障害プロバイダの復旧を監視する。サーキットブレーカーが自動的に復旧を試行する。

### 4.2 クラウドLLM全プロバイダ障害（インターネット障害含む）

全クラウドプロバイダのサーキットがOPENになると、ローカルLLMへ自動フォールバックする。

**対応**:

1. Discord通知で障害を確認する
2. インターネット接続を確認する
3. ローカルLLMが正常稼働していることを確認する

```bash
# ローカルLLMの稼働確認
curl http://localhost:11434/api/tags
```

4. 障害が長引く場合は手動で縮退モードへ移行する

```bash
python -m vibe_pdca.cli mode set \
  --project-id <PROJECT_ID> \
  --mode degraded \
  --reason "クラウドLLM全プロバイダ障害 - ローカルLLMで縮退運転中"
```

### 4.3 ローカルLLMも利用不可の場合

クラウド・ローカル共に利用不可の場合は `LocalLLMUnavailableError` が発生し、PDCAサイクルは自動停止する。

**対応**:

1. ローカルLLMサーバー（Ollama等）を起動する
2. モデルがダウンロード済みか確認する

```bash
ollama list
ollama pull llama3.3:70b
```

3. 復旧後、PDCAサイクルを再開する

```bash
python -m vibe_pdca.cli run \
  --project-id <PROJECT_ID> \
  --resume
```

## 5. サーキットブレーカーの手動制御

### 5.1 手動でサーキットを OPEN にする

特定のクラウドプロバイダに既知の障害がある場合、ヘルスチェックを待たずに即座にフォールバックさせる。

```python
from vibe_pdca.llm.circuit_breaker import CircuitState
cb = gateway._circuit_breakers["openai-gpt5.1"]
cb.force_open(reason="OpenAI 計画メンテナンス")
```

### 5.2 手動でサーキットを CLOSED にする

障害復旧が確認できた場合、自動復旧を待たずに即座にクラウドへ復帰させる。

```python
cb.force_close(reason="OpenAI 障害復旧確認済み")
```

## 6. モニタリング

### 6.1 ステータス確認

```python
status = gateway.get_status()
```

### 6.2 監視すべきメトリクス

| メトリクス | 閾値 | 対応 |
|-----------|------|------|
| サーキット状態 | OPEN | フォールバック発生中 - 障害原因を調査 |
| フォールバック回数 | 5回/時間超 | 障害が頻発 - プロバイダ変更を検討 |
| ローカルLLMレイテンシ | 30秒超 | モデルサイズの見直しを検討 |
| 日次コスト | 上限の80% | コスト上限の引き上げまたはローカル切替を検討 |

## 7. トラブルシューティング

| 症状 | 原因 | 対処 |
|------|------|------|
| フォールバックが発生しない | `auto_fallback: false` | 設定を `true` に変更 |
| ローカルLLMに接続できない | Ollama未起動 | `ollama serve` で起動 |
| フォールバック後に復帰しない | `recovery_timeout` が長すぎる | 設定値を短縮 |
| 誤ってOPENになる | `failure_threshold` が低すぎる | 設定値を引き上げ |
| コスト上限エラー | 日次/サイクル上限に到達 | `daily_limit_usd` を引き上げ |
