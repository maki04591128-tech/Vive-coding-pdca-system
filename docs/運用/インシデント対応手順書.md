## 📖 この文書について

> この文書は、システムに**障害や異常が発生したとき**にどう対応するかを定めた手順書です。  
> たとえるなら、「病院の救急対応マニュアル」のようなものです。  
> - **P0（重症）** = 救急搬送レベル。情報漏えいや権限の不正利用など、すぐに全停止して調査が必要
> - **P1（中症）** = 外来緊急対応レベル。運用に大きな影響があるが、縮退モードで凌ぎつつ対応
> - **P2（軽症）** = 経過観察レベル。次のサイクルで自動修正を試みる

### 🔰 この文書で使われる主要な用語

| 用語 | かんたんな説明 |
|------|-------------|
| **インシデント** | システムの異常事態・障害のこと |
| **P0/P1/P2** | 障害の深刻度ランク。数字が小さいほど深刻 |
| **縮退モード** | 一部の機能を制限して最低限の動作を続けるモード |
| **ロールバック** | 障害発生前の状態に戻すこと。「巻き戻し」のようなもの |
| **エスカレーション** | 自分では対処できない問題を上位の担当者に報告すること |
| **ポストモーテム** | 障害の事後分析。「何が起きたか」「なぜ起きたか」「再発防止策」をまとめる |
| **CI** | Continuous Integration。コードの自動テストの仕組み |
| **監査ログ** | 「誰が・いつ・何をしたか」の記録。改ざんできない仕組みになっています |

## 1. インシデント分類

| 優先度 | 定義 | 自動アクション | 人間対応期限 |
|--------|------|--------------|------------|
| **P0** | 情報漏えい疑い / 権限逸脱 / 重大破壊 | **即時自動停止** | 1時間以内 |
| **P1** | 運用継続に重大影響 | 縮退モード移行 + Discord重大通知 | 4時間以内 |
| **P2** | 軽微な異常 | Discord警告通知のみ | 次サイクルで是正 |

### 自動発火条件

| 条件 | 分類 |
|------|------|
| 秘密情報露出疑い | P0 |
| 権限昇格の試み検知 | P0 |
| 監査ログ書き込み失敗 | P0 |
| 処理スタック6時間超 | P1 |
| CI連続失敗3回以上 | P1 |
| コスト急増（3倍超 / 日次$30超） | P1 |
| LLM 3モデル以上同時障害 | P1 |
| GitHub障害 | P1 |
| Discord障害 | P1 |

## 2. P0 対応フロー

```
[自動] P0検知 → 即時全停止 → 監査ログ記録 → Discord重大通知
[人間] Owner確認（1h以内）→ 初期調査 → ロールバック（必要時）
[人間] 根本原因特定 → 再開条件設定 → 4/4承認 → 再開
```

### P0確認事項

- [ ] 秘密情報露出の有無
- [ ] 権限設定変更の有無
- [ ] データ整合性確認
- [ ] 想定外の外部API呼び出しの有無

---

## 3. P1 対応フロー

```
[自動] P1検知 → 縮退モード移行 → 監査ログ記録 → Discord重大通知
[人間] Owner/Maintainer確認（4h以内）→ 原因調査 → 対処 → 通常モード復帰
```

---

## 4. P2 対応フロー

P2は現在のサイクルを止めない。同じP2が3サイクル連続で発生した場合はP1に昇格。

## 5. ロールバック手順

```bash
# バックアップ一覧
python -m vibe_pdca.cli backup list --project-id <PROJECT_ID>

# ロールバック実行
python -m vibe_pdca.cli rollback \
  --project-id <PROJECT_ID> \
  --backup-id <BACKUP_ID> \
  --reason "P0インシデント対応"

# 整合性確認
python -m vibe_pdca.cli verify --project-id <PROJECT_ID>
```

## 7. 障害パターン別対応表

| 障害パターン | 分類 | 自動対応 | 人間対応 |
|------------|------|---------|---------|
| LLM 1モデル障害 | P2 | 代替モデルで継続 | 翌日確認 |
| LLM 3モデル以上同時障害 | P1 | DO停止 | 復旧確認・手動再開 |
| GitHub障害 | P1 | 全自動停止 | 復旧後に手動再開 |
| 秘密情報露出疑い | P0 | 即停止 | 調査・ローテーション・4/4承認後再開 |
| 監査ログ書き込み失敗 | P0 | 即停止 | ログストレージ修復後再開 |
| コスト急増 | P1 | 自動停止 | 原因調査・Owner承認後再開 |
