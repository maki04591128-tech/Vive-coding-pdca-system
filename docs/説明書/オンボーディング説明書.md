## 📖 この説明書で使われる用語

> コンピュータに詳しくない方でも理解できるよう、この説明書で使われる主要な用語をまとめました。  
> 初めて読む方は、まずここに目を通してから本文に進んでください。

| 用語 | かんたんな説明 |
|------|-------------|
| **PDCA** | Plan（計画）→ Do（実行）→ Check（確認）→ Act（改善）の4つのステップを繰り返す改善手法です。ビジネスでもよく使われる考え方で、本システムではこのサイクルをAIが自動で回します |
| **LLM（エルエルエム）** | Large Language Model（大規模言語モデル）の略で、ChatGPTやClaudeのような「文章を理解し、文章で回答するAI」のことです |
| **AI ペルソナ** | このシステムでは5つのAIがそれぞれ異なる「役割」を持ちます。PM（プロジェクトマネージャー）・書記・プログラマ・デザイナ・ユーザの5役です。人間のチームのように、それぞれの視点でプロジェクトを進めます |
| **GitHub（ギットハブ）** | プログラムのソースコード（設計図のようなもの）を保存・共有するためのWebサービスです。複数人での共同作業に使われます |
| **リポジトリ** | GitHub上にある「プロジェクトの保管庫」のことです。1つのプロジェクトに1つのリポジトリを作ります |
| **Issue（イシュー）** | GitHubの「やることリスト」機能です。タスクやバグの管理に使います |
| **PR（プルリクエスト）** | プログラムの変更を「確認してから取り込んでください」と依頼する仕組みです |
| **Discord（ディスコード）** | テキストや音声でやり取りできるチャットアプリです。本システムではAIからの通知や承認要求の送信先として使います |
| **Discord Bot（ボット）** | Discordに常駐する自動プログラムです。本システムの通知や承認フローを自動化します |
| **GitHub App** | GitHubの機能を外部のシステムと連携させるための仕組みです。本システムがGitHub上でIssueやPRを自動作成するために必要です |
| **Web UI（ウェブユーアイ）** | Webブラウザ（Chrome等）からアクセスする操作画面のことです |
| **CLI（シーエルアイ）** | Command Line Interface の略で、マウスではなくキーボードでコマンド（命令文）を入力して操作する方法です。「ターミナル」や「コマンドプロンプト」とも呼ばれます |
| **API キー** | 外部サービスを利用するための「合鍵」のようなものです。サービスごとに発行され、正当な利用者であることを証明します |
| **A操作 / B操作 / C操作** | 操作の重要度による分類です。A操作は最も重要で全員の承認が必要、B操作は中程度、C操作は軽微な変更で自動実行されます |

## 1. はじめに

本ガイドは、バイブコーディングPDCA自動開発システムを**初めて使用する方**向けの入門書です。  
コンピュータの専門知識がなくても理解できるよう、役割と権限の仕組み、初期セットアップの進め方、日常的な操作方法をわかりやすく説明します。

## 2. システム概要

### このシステムは何ができるの？

バイブコーディングPDCA自動開発システムは、**「こんなものを作りたい」と日本語で入力するだけで、AIが自動でプログラムを作ってくれるシステム**です。

具体的には、5つのAI（PM・書記・プログラマ・デザイナ・ユーザ）がチームのように協力して、計画を立て（Plan）→ コードを書き（Do）→ 品質を確認し（Check）→ 改善する（Act）…というサイクルを自動で繰り返します。

成果物（プログラムのソースコード等）はGitHub上に保存され、進捗はDiscordで通知されます。

### 2.1 基本的な流れ

1. **最終到達点の入力**：Webブラウザの操作画面（Web UI）から「何を作りたいか」を日本語で入力する
2. **要件レビュー**：5つのAIペルソナ（PM・書記・プログラマ・デザイナ・ユーザ）が入力内容を確認し、不足があれば質問してくる
3. **PDCA自動ループ**：内容が固まったら、AIが自動でプログラムの生成・レビュー・改善を繰り返す
4. **進捗報告**：ひとつのサイクルが終わるたびにDiscord（チャットアプリ）で通知が届く
5. **人間の介入**：重要な判断が必要な場合はDiscordで「承認してください」という通知が届く

## 3. 役割と権限

### 3.1 4つの役割

本システムでは、人間のチームメンバーに「役割（ロール）」が割り当てられます。役割によってできる操作が異なります。

> 💡 **なぜ役割を分けるの？**  
> 重要な操作を一人で勝手に実行できないようにするためです。たとえば、システムの設定を大きく変更する場合は、全員の承認が必要になります。

| 役割 | 誰がなるか | 主な責務 | わかりやすく言うと |
|------|-----------|---------|-----------------|
| **Owner** | プロジェクトの責任者 | 最終到達点の入力、重要な意思決定（A操作の承認）、モデル切替判断 | プロジェクトの「社長」。最終判断をする人 |
| **Maintainer** | 日常運用の担当者 | 停止/再開、運転モード切替、ポリシー変更（制限付き） | プロジェクトの「管理者」。日々の運用を担当する人 |
| **Reviewer** | レビュー参加者 | レビュー結果の閲覧・コメント | プロジェクトの「確認者」。成果物の品質をチェックする人 |
| **Auditor** | 監査担当者 | 監査ログの閲覧・エクスポート | プロジェクトの「監査役」。記録を確認して問題がないか検査する人 |

詳細はロール権限マトリクス（VCS-RBAC-001）を参照。

### 3.2 操作の分類

システムの操作は重要度によってA・B・Cの3段階に分けられています。

> 💡 **なぜ操作を分類するの？**  
> 影響の大きい操作ほど慎重に進めるためです。銀行で言えば、「残高照会（C操作）」は自由にできますが、「大口送金（A操作）」には複数人の承認が必要なのと同じです。

| 分類 | 内容 | 承認方法 | たとえるなら |
|------|------|---------|------------|
| **A操作** | 権限の拡大・ポリシー（ルール）の緩和など、影響が大きい操作 | Discordで全ロール（4人）の承認が必要 | 銀行の大口送金（複数人の承認が必要） |
| **B操作** | プログラム変更が大きい場合・外部ライブラリ更新など、中程度の操作 | バックアップ後に自動で進行（通知あり） | ATMでの送金（本人確認後に実行） |
| **C操作** | 軽微な変更 | 承認不要（自動で実行） | コンビニでの買い物（そのまま購入できる） |

詳細はロール権限マトリクス（VCS-RBAC-001）を参照。

## 4. 初期セットアップの概要

> 💡 **セットアップとは？**  
> システムを使い始めるための「初期準備」のことです。

### 4.1 前提条件

以下の3つのアカウントが必要です。まだお持ちでない方は、それぞれのサービスで無料アカウントを作成してください。

- **GitHubアカウント**（Organization所属）— プログラムの保管先。[github.com](https://github.com) で作成
- **Discordアカウント**（指定サーバーへの参加）— チャットと通知に使用。[discord.com](https://discord.com) で作成
- **本システムのWeb UIへのアクセス権** — 管理者から付与されます

### 4.2 セットアップの全体像

セットアップは、以下の5つの工程から成り立っています。

| ステップ | 内容 | 目的 |
|---------|------|------|
| **Step 1** | プロジェクトリポジトリの準備 | AIが作ったプログラムを保管する「倉庫」を用意する |
| **Step 2** | Discord環境の準備 | AIからの通知や承認要求を受け取る「連絡先」を用意する |
| **Step 3** | プロジェクト設定 | Web UI（操作画面）からプロジェクトの基本情報を入力する |
| **Step 4** | 最終到達点の入力 | 「何を作りたいか」というゴールをAIに伝える |
| **Step 5** | 初回動作確認 | AIがゴールを理解し、PDCAサイクルが正常に動くことを確認する |

> 📌 **具体的な手順はこちら**  
> - 本番サーバーを使う場合 → [02_システムセットアップ手順書](../手順書/02_システムセットアップ手順書.md)  
> - 手元のPCで始める場合 → [03_ローカルセットアップ手順書](../手順書/03_ローカルセットアップ手順書.md)  
> - どちらを選ぶべきか迷った場合 → [手順書ガイド（00_はじめに）](../手順書/00_はじめに.md)

### 4.3 セットアップで設定する主な項目

セットアップ時には以下の情報を入力します。事前に決めておくとスムーズです。

| 設定項目 | 意味 | 例 |
|---------|------|-----|
| プロジェクト名 | プロジェクトのわかりやすい名前 | 「勤怠管理システム」 |
| プロジェクト種別 | 作るものの種類 | Webアプリ / ライブラリ / CLI / インフラ/IaC / データパイプライン / モバイルアプリ |
| LLMコスト上限（USD） | AIの1日あたりの利用料金の上限 | 「10」（ドル） |
| 運転モード | AIの自動化レベル | 手動承認（初心者向け） / 半自動 / 全自動 |

> 💡 **運転モードの選び方**：初めてのプロジェクトでは「**手動承認**」がおすすめです。重要な操作のたびに確認が求められるので、AIが何をしているかを把握できます。

### 4.4 「最終到達点」とは？

> 💡 **最終到達点** = 「最終的にどんなものを作りたいか」というゴールの説明です。

セットアップの中で最も重要なのが「最終到達点」の入力です。以下の項目を記入します。

| 項目 | 内容 | 例 |
|------|------|-----|
| **目的** | 何を実現するか | 「社内の勤怠管理Webアプリを作りたい」 |
| **受入条件**（必須） | 完成とみなす条件 | 「従業員がブラウザから出退勤を打刻できること」 |
| **制約**（必須） | 技術・納期・運用の制約 | 「Python + Flaskを使用。2週間以内」 |
| **禁止事項**（必須） | やってはいけないこと | 「個人情報を外部APIに送信しないこと」 |
| **優先度**（できれば） | 各項目の優先順位 | 「打刻機能 > レポート機能 > 管理画面」 |

### 4.5 セットアップ完了の定義

以下の3つをすべて満たしていれば、セットアップは完了です 🎉

- ✅ 最終到達点（ゴール）が入力済み
- ✅ AIによる要件レビューで重大な問題（blocker）が解消済み
- ✅ 初回のPDCAサイクルが完了し、進捗レポートがDiscordに届いている

## 5. 典型操作

> ここでは、日常的によく使う操作の**概要**を説明します。  
> 各操作の具体的な手順は [04_運用保守手順書](../手順書/04_運用保守手順書.md) を参照してください。

### 5.1 PDCAの停止

**いつ使うか**：緊急事態が発生した、方針を大きく変更したい、長期間席を離れる場合

> 💡 「停止」は一時中断です。データは消えません。いつでも再開できます。

Web UIで「停止」ボタンを押すと（Owner または Maintainer のみ）、AIが即座にPDCAサイクルを停止します。停止時にシステムが自動で以下の3つを提示してくれます：

- **原因分析レポート**：何が起きたかの分析
- **再開条件**：何を解決すれば再開できるか
- **ロールバック候補**：問題の前の状態に戻すための候補

> 📌 **停止・再開の具体的な操作手順** → [04_運用保守手順書 §3](../手順書/04_運用保守手順書.md#3-システムの起動停止再開)

### 5.2 PDCAの再開

**いつ使うか**：停止中のシステムを再び動かしたい場合

停止理由と再開条件を確認した上で、Web UIの「再開」ボタンを押します（Owner または Maintainer のみ）。

> 📌 **再開の具体的な操作手順** → [04_運用保守手順書 §3](../手順書/04_運用保守手順書.md#3-システムの起動停止再開)

### 5.3 A操作の承認

**いつ使うか**：Discordに「承認してください」という通知が届いた場合

> 💡 A操作は影響が大きい操作なので、チーム全員（4人）の承認が必要です。通知を見逃さないようにしましょう。

Discord通知には以下が記載されています：
- **何をするか**：実行される操作の内容
- **なぜ必要か**：操作が必要な理由
- **影響範囲**：どこに影響があるか
- **代替案**：別の方法はないか

回答は3種類あります：
- **YES（承認）**：問題なければ承認
- **NO（却下）**：問題があれば却下（**必ず理由を記入してください**）
- **HOLD（保留）**：判断を保留する場合

全ロール（4人）の承認が揃うと操作が実行されます。12時間以内に揃わない場合は自動的に保留になります。

> 📌 **A操作承認の詳細な手順** → [04_運用保守手順書 §4](../手順書/04_運用保守手順書.md#4-a操作の承認フロー)

### 5.4 運転モードの切替

**いつ使うか**：AIの自動化レベルを変更したい場合

> 💡 **運転モード** = AIがどこまで自動で判断するかの設定です。  
> 初心者のうちは「手動承認」モードで始めて、慣れてきたら「半自動」→「全自動」と段階的に切り替えるのがおすすめです。

| モード | 推奨する場面 | B操作（中程度の変更）の扱い | たとえるなら |
|--------|------------|--------------------------|------------|
| 手動承認 | 初期段階・重要なフェーズ | Owner（責任者）が承認してから進行 | 車の教習所（教官が隣にいる状態） |
| 半自動 | 安定している運用期 | バックアップを取ってから自動で進行 | 一般道の運転（自分で判断するが、安全装置あり） |
| 全自動 | リスクが低い改善フェーズ | バックアップを取ってから自動で進行 | 高速道路のクルーズコントロール |

> 運転モード別の詳細な権限差分はロール権限マトリクス（VCS-RBAC-001）§7 を参照。

> 📌 **運転モード切替の具体的な手順** → [04_運用保守手順書 §5](../手順書/04_運用保守手順書.md#5-運転モードの管理)

### 5.5 ドライラン（テスト実行）

**いつ使うか**：本番に影響を与えずに「どう動くか」を確認したい場合

> 💡 **ドライラン** = 実際のプロジェクトには一切影響を与えずに、AIの動作だけを確認するモードです。「試し運転」と考えてください。

> 📌 **ドライランの実行手順** → [04_運用保守手順書](../手順書/04_運用保守手順書.md)

## 6. よくある問題と対処法

> 💡 問題が発生した場合は、まず以下のよくある事例を確認してください。ここにない問題は管理者に相談しましょう。  
> 📌 **詳しいトラブルシューティング手順** → [04_運用保守手順書 §16](../手順書/04_運用保守手順書.md#16-トラブルシューティング集)

### 6.1 GitHub App権限不足

**症状**：Issue（やることリスト）やPR（変更依頼）が自動で作成されない、「権限エラー」が表示される

**原因**：システムがGitHubにアクセスするための権限が不足しています

**確認すべきこと**：
- GitHub Appがリポジトリ単位でインストールされているか（Organization全体への付与は禁止）
- Issues / Pull Requests / Contents の Read/Write 権限が付与されているか

> 📌 **GitHub Appの設定手順** → [02_システムセットアップ手順書 §7](../手順書/02_システムセットアップ手順書.md#7-phase-5github-app-の作成と設定) または [03_ローカルセットアップ手順書 §7](../手順書/03_ローカルセットアップ手順書.md#7-step-5github-app-の作成と設定)

### 6.2 Discord Bot接続失敗

**症状**：Discord通知が届かない、承認要求が送信されない

**原因**：Discord Botの設定が正しくないか、Botがチャンネルに追加されていません

**確認すべきこと**：
- Botの認証キー（`DISCORD_BOT_TOKEN`）が正しいか
- Botがプロジェクト専用チャンネルに追加されているか
- Botにメッセージ送信・リアクション追加の権限があるか

> 📌 **Discord Botの設定手順** → [02_システムセットアップ手順書 §8](../手順書/02_システムセットアップ手順書.md#8-phase-6discord-bot-とチャンネルの設定) または [03_ローカルセットアップ手順書 §8](../手順書/03_ローカルセットアップ手順書.md#8-step-6discord-botチャンネルの設定)

### 6.3 初回サイクルが停止する

**症状**：PDCAが開始直後に停止してしまう

**原因**：入力内容の不足か、AIサービスへの接続に問題があります

**確認すべきこと**：
- 最終到達点に「受入条件」「制約」「禁止事項」の3つが含まれているか
- AI（LLM）のAPIキーが有効か

> 📌 **障害対応の手順** → [04_運用保守手順書 §8](../手順書/04_運用保守手順書.md#8-障害対応インシデント管理)

### 6.4 LLMコスト上限超過

**症状**：「コスト上限を超えました」というメッセージが表示され、自動停止する

**原因**：AIの利用料金が設定した上限額を超えました

**確認すべきこと**：
- 現在のコスト内訳を確認し、どのAIモデルが多く費用を使っているか把握する
- 必要に応じてOwnerがコスト上限を引き上げてPDCAを再開する

> 📌 **コスト管理の手順** → [04_運用保守手順書 §6](../手順書/04_運用保守手順書.md#6-コスト管理)
