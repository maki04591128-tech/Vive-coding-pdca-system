## 1. はじめに

本ガイドは、バイブコーディングPDCA自動開発システムを初めて使用するユーザーおよびチームメンバー向けに、役割と権限の理解、初期セットアップ手順、典型操作の実行方法を説明する。

---

## 2. システム概要

バイブコーディングPDCA自動開発システムは、ユーザーが「最終到達点（ゴール）」を自然言語で入力するだけで、複数のAI（LLM）が自律的にPLAN → DO → CHECK → ACTのサイクルを回し、GitHub上で成果物を継続的に生み出すシステムである。

### 2.1 基本的な流れ

1. **最終到達点の入力**：Web UIから「何を作りたいか」を入力する
2. **要件レビュー**：5つのAIペルソナが入力内容をレビューし、不足があれば質問する
3. **PDCA自動ループ**：要件確定後、AIが自動でコード生成・レビュー・改善を繰り返す
4. **進捗報告**：サイクルごとにDiscordで通知が届く
5. **人間の介入**：重要な判断が必要な場合はDiscordで承認要求が届く

---

## 3. 役割と権限

### 3.1 4つの役割

| 役割 | 誰がなるか | 主な責務 |
|------|-----------|---------|
| **Owner** | プロジェクトの責任者 | 最終到達点の入力、重要な意思決定（A操作の承認）、モデル切替判断 |
| **Maintainer** | 日常運用の担当者 | 停止/再開、運転モード切替、ポリシー変更（制限付き） |
| **Reviewer** | レビュー参加者 | レビュー結果の閲覧・コメント |
| **Auditor** | 監査担当者 | 監査ログの閲覧・エクスポート |

### 3.2 操作の分類

| 分類 | 内容 | 承認方法 |
|------|------|---------|
| **A操作** | 権限拡大・ポリシー緩和など重要操作 | Discord上で全ロール（4人）の承認が必要 |
| **B操作** | diff超過・依存更新など中程度の操作 | バックアップ後に自動進行（通知あり） |
| **C操作** | 軽微な変更 | 承認不要（自動） |

詳細はロール権限マトリクス（VCS-RBAC-001）を参照。

## 4. 初期セットアップ手順

### 4.1 前提条件

- GitHubアカウント（Organization所属）
- Discordアカウント（指定サーバーへの参加）
- 本システムのWeb UIへのアクセス権

### 4.2 セットアップ手順

#### Step 1：プロジェクトリポジトリの準備

1. GitHubでプロジェクト用の空リポジトリを作成する
2. GitHub Appをリポジトリにインストールする（Organization全体ではなくリポジトリ単位）
3. リポジトリのBranch Protectionルールを設定する

#### Step 2：Discord環境の準備

1. プロジェクト専用のDiscordチャンネルを作成する
2. システムのDiscord Bot をチャンネルに追加する
3. 通知設定を確認する（A操作通知は必ず受信する設定にする）

#### Step 3：プロジェクト設定

1. Web UIにログインする
2. 「新規プロジェクト」を選択する
3. 以下を設定する：
   - プロジェクト名
   - GitHubリポジトリURL
   - Discordチャンネル
   - プロジェクト種別（Webアプリ / ライブラリ / CLI / インフラ/IaC / データパイプライン / モバイルアプリ）
   - LLMコスト上限（USD）
   - 運転モード（手動承認 / 半自動 / 全自動）

#### Step 4：最終到達点の入力

1. Web UIで「最終到達点入力」画面を開く
2. 以下の項目を記入する：
   - **目的**：何を実現するか
   - **受入条件**：達成状態の判定条件（必須）
   - **制約**：技術・納期・運用・互換性の制約（必須）
   - **禁止事項**：やってはいけない変更（必須）
   - **優先度**：各項目の優先順位（望ましい）
3. 「送信」ボタンで内容を送信する
4. システムからの質問に回答する（必要な回数だけ往復する）
5. 内容が整ったら「要件定義完了」ボタンを押す

#### Step 5：初回動作確認

1. 5ペルソナによる要件レビューが実施される
2. blockerがなければPDCAサイクルが開始される
3. 初回のサイクル完了通知がDiscordに届くことを確認する

### 4.3 初期セットアップ完了の定義

以下をすべて満たした状態をセットアップ完了とみなす：

- 最終到達点が入力済み
- 要件レビューでblockerが解消済み
- 初回PDCAサイクルが完了し、進捗レポートが生成されている

## 5. 典型操作

### 5.1 PDCAの停止

**いつ使うか**：緊急時、方針変更時、長期不在時

1. Web UIで「停止」ボタンを押す（Owner / Maintainer のみ）
2. 即座にPDCAが停止する
3. システムが「原因分析レポート」「再開条件」「ロールバック候補」を提示する

### 5.2 PDCAの再開

**いつ使うか**：停止状態から運用を再開する場合

1. 停止理由と再開条件を確認する
2. 再開条件が満たされていることを確認する
3. Web UIで「再開」ボタンを押す（Owner / Maintainer のみ）

### 5.3 A操作の承認

**いつ使うか**：Discordに承認要求通知が届いた場合

1. Discord通知の内容を確認する（何をするか、なぜ必要か、影響範囲、代替案）
2. 承認（YES）/ 却下（NO）/ 保留（HOLD）のいずれかをDiscordで回答する
3. NOの場合は理由を必ず記入する
4. 全ロール（4人）の承認が揃うと実行される
5. 12時間以内に承認が揃わない場合は保留となる

### 5.4 運転モードの切替

**いつ使うか**：システムの自動化レベルを変更したい場合

| モード | 推奨場面 | B操作の扱い |
|--------|---------|------------|
| 手動承認 | 初期・重要フェーズ | Owner承認後に進行 |
| 半自動 | 安定運用期 | バックアップ後に自動進行 |
| 全自動 | 低リスク改善フェーズ | バックアップ後に自動進行 |

```bash
python -m vibe_pdca.cli mode set \
  --project-id <PROJECT_ID> \
  --mode manual|semi-auto|full-auto \
  --reason "理由"
```

### 5.5 ドライラン

**いつ使うか**：本番に影響を与えずに動作確認したい場合

```bash
python -m vibe_pdca.cli run \
  --project-id <PROJECT_ID> \
  --dry-run \
  --cycles 1
```

## 6. よくある問題と対処法

### 6.1 GitHub App権限不足

**症状**：Issue / PR が作成されない、権限エラーが発生する

**対処法**：
1. GitHub Organization設定 → GitHub Apps → 当該Appの権限を確認
2. リポジトリ単位でインストールされているか確認（Organization全体付与は禁止）
3. 必要な権限：Issues(Read/Write), Pull Requests(Read/Write), Contents(Read/Write)

### 6.2 Discord Bot接続失敗

**症状**：Discord通知が届かない、承認要求が送信されない

**対処法**：
1. `.env` の `DISCORD_BOT_TOKEN` が正しいか確認
2. Botがプロジェクト専用チャンネルに追加されているか確認
3. Botの権限（メッセージ送信・リアクション追加）を確認

### 6.3 初回サイクルが停止する

**症状**：PDCA開始直後に停止する

**対処法**：
1. 停止理由を確認する（Web UI「制御」画面 または `python -m vibe_pdca.cli status`）
2. 最終到達点に受入条件・制約・禁止事項が含まれているか確認
3. LLM APIキーが有効か確認（`python -m vibe_pdca.cli health`）

### 6.4 LLMコスト上限超過

**症状**：コスト上限超過で自動停止

**対処法**：
1. `python -m vibe_pdca.cli cost --period today --breakdown model` でコスト内訳を確認
2. Ownerがコスト上限を引き上げる場合：`python -m vibe_pdca.cli config set --key daily_cost_limit_usd --value <新上限>`
3. Ownerの承認後に再開
