# コントリビューション説明書

バイブコーディングPDCA自動開発システムへの貢献を歓迎します。

> 💡 **コントリビューション（Contribution）とは？**  
> ソフトウェア開発の世界では、プロジェクトに対して改善やバグ修正、新機能の追加などで「貢献する」ことをコントリビューションと呼びます。  
> この説明書では、本プロジェクトに貢献するための手順を説明します。

---

## 📖 この説明書で使われる用語

| 用語 | かんたんな説明 |
|------|-------------|
| **リポジトリ** | プロジェクトのファイルをすべて保管する「倉庫」のこと。GitHubというサービス上にあります |
| **クローン** | リポジトリをネット上から自分のパソコンにコピーすること |
| **ブランチ** | 「枝」のこと。メインのコードに影響を与えずに、別の作業スペースで変更を行う仕組みです |
| **コミット** | 変更内容を「記録」すること。「セーブポイント」のようなものです |
| **プルリクエスト（PR）** | 自分の変更を「確認してください。問題なければ取り込んでください」とチームに依頼する仕組みです |
| **リント（Lint）** | プログラムの書き方ルールに違反していないかを自動でチェックすること。「スペルチェック」のプログラム版です |
| **型チェック** | プログラムのデータの種類（数値、文字列など）が正しく使われているかを検査すること |
| **テスト** | プログラムが正しく動くかを自動で確認すること。人間がひとつずつ確認する代わりに、コンピュータが検証します |
| **仮想環境（venv）** | プロジェクトごとにPythonの設定を分離する仕組み。他のプロジェクトと設定が干渉しないようにします |
| **pip** | Pythonのパッケージ（追加機能）をインストールするためのツール |

## 開発環境のセットアップ

> 💡 **セットアップ** = 開発を始めるための「初期準備」です。

開発環境のセットアップでは、以下の作業を行います：

| 作業 | 内容 |
|------|------|
| リポジトリのクローン | プロジェクトをネット上から自分のPCにコピーする |
| Python仮想環境の作成 | プロジェクト専用のPython環境を用意する |
| 開発用パッケージのインストール | テストやリントに必要なツールを導入する |
| 環境変数の設定 | APIキーなどの秘密情報を設定ファイルに記入する |

> 📌 **セットアップの具体的な手順** → [03_ローカルセットアップ手順書](../手順書/03_ローカルセットアップ手順書.md)  
> 03_ローカルセットアップ手順書では、各コマンドの意味も含めて初心者向けに解説しています。

## 開発ワークフロー

> 💡 以下の5ステップが、変更を行う基本的な流れです。

### 1. ブランチの作成

> 💡 **ブランチ** = メインのコードに影響を与えずに作業するための「分岐」です。作業が完了したらメインに合流させます。

新しい機能を追加したりバグを修正したりするときは、まず「ブランチ」を作成します。これにより、メインのコードを壊すことなく安全に作業できます。

### 2. コードの変更

- `src/vibe_pdca/` フォルダ配下にモジュール（プログラムファイル）を配置
- 既存のコードスタイル（書き方のルール）に合わせる（`from __future__ import annotations` を使用）
- 1行の長さは 100 文字以内

### 3. テストの作成・実行

> 💡 **テスト** = プログラムが正しく動くことを自動で検証する仕組みです。変更後は必ずテストを実行してください。

テストを実行して、変更がプログラムを壊していないことを確認します。テスト結果が全件成功（パス）であることを確認してください。

### 4. リントと型チェック

> 💡 **リント** = コードの書き方ルール違反を自動検出するツールです。提出前に必ず実行してください。

リントと型チェックを実行し、コードスタイルの問題がないことを確認します。

### 5. プルリクエストの作成

> 💡 **プルリクエスト（PR）** = 変更を「チームに確認してもらい、問題なければメインのコードに取り込んでもらう」ための仕組みです。

- 変更内容を明確に記述する（何をなぜ変更したか）
- 関連する Issue（やることリスト）の番号を参照する
- テストが全件パス（成功）していることを確認してから提出する

> 📌 **各ステップの具体的なコマンドと実行方法** → [03_ローカルセットアップ手順書](../手順書/03_ローカルセットアップ手順書.md)

## コーディング規約

> 💡 **コーディング規約** = チーム全員が同じスタイルでコードを書くためのルールです。統一されたスタイルにすることで、誰が書いたコードでも読みやすくなります。

### Python スタイル

- **Python 3.12+** を対象
- **ruff** によるリント（設定は `pyproject.toml` に定義）— コードスタイルの自動チェックツール
- **mypy** による型チェック（strict モード）— データ型の正しさを検査するツール
- `from __future__ import annotations` を全ファイルの先頭に記述
- docstring（関数やクラスの説明文）は日本語で記述する（提案番号への参照を含む）
- セクションコメント（区切り線）は `# ── ClassName ──` 形式

### モジュール構成

> 💡 以下はプログラムの主要なフォルダ構成です。それぞれのフォルダが異なる機能を担当しています。

```
src/vibe_pdca/
├── engine/          # コアエンジン（PDCAサイクルの中心部分・各種機能）
├── models/          # データモデル（データの形を定義するファイル群）
├── config/          # 設定管理（システムの設定を読み込む部分）
├── llm/             # LLMゲートウェイ（AIとの通信を担当する部分）
├── github/          # GitHub連携（GitHubとの連携を担当する部分）
├── gui/             # GUI（操作画面。Fletというツールで構築）
├── audit/           # 監査ログ（操作記録を保存する部分）
├── monitoring/      # モニタリング（システムの状態を監視する部分）
├── governance/      # ガバナンス（ルールの管理を担当する部分）
├── glossary/        # 用語集（専門用語の管理）
├── prompts/         # プロンプトテンプレート（AIへの指示文のひな形）
└── plugins/         # プラグインシステム（機能を追加するための仕組み）
```

### テスト規約

- テストファイルは `tests/test_<モジュール名>.py` に配置
- pytest クラスベース（`class TestXxx:`）の形式を使用
- 外部依存（AI APIやGitHub API）はモック化する（実際のサービスに接続せず、模擬データで検証する）
- 標準ライブラリのみ使用する（テスト専用の外部パッケージは追加しない）
