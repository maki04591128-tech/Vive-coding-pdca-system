## 1. ゴールとスコープ

### 1.1 ゴール

**「バイブコーディング」**とは、ユーザーが高レベルな「最終到達点（ゴール）」を自然言語で与えるだけで、AI（LLM群）が自律的に設計・実装・レビュー・改善のサイクルを回し、成果物を継続的に生み出す開発スタイルを指す。

本システムはその基盤として、以下を実現する：

- **最終到達点**を入力すると、複数LLMを役割分担させて PLAN → DO → CHECK → ACT を自動ループ
- マイルストーン（Definition of Done）単位で進捗報告
- GitHub上で Issue / PR / CI を回し、成果物（コード / ドキュメント / UI）を継続的に更新

### 1.2 使用モデル

> **【確定】定義例B採用：実装着手時に利用可能な最新安定版を能力軸で選定する**

**役割別の必須能力軸（今確定）：**

| 役割 | 必須能力軸 | 重視事項 |
|------|-----------|---------|
| **PM** | 長文の構造化・タスク分解・リスク列挙 | 日本語での論理的説明 |
| **書記** | ドキュメント間の矛盾検出・要約精度 | 長コンテキスト処理（ADR・仕様書全体の把握） |
| **プログラマ** | コードの欠陥検出・セキュリティ観点・型安全性 | 対象言語の精度 |
| **デザイナ** | UI導線の問題検出・アクセシビリティ・情報設計 | 非エンジニア視点での説明能力 |
| **ユーザ** | 混乱・不便・誤操作の発見 | 非エンジニア視点での指摘 |
| **DO担当（実装）** | 指示に忠実な実装・テスト生成・エラー修正 | 対象言語の精度・長いコードへの対応 |

**モデル選定プロセス（実装着手時に実施）：**

1. 各ベンダーの最新安定版APIの提供状況・レート制限・コスト単価を確認する
2. 上記の能力軸に最も適合するモデルを役割ごとに選定する
3. 選定結果と選定理由をADRに記録し、初期値として固定する
4. フォールバック順（障害時の代替モデル）も合わせて記録する

> **DO担当は確定：Claude（Anthropic API経由）**
> その他の役割のモデル名は実装着手日に決定する。

### 1.3 スコープ

**含む：**
- 自動開発システムの要件定義（機能・非機能・運用・受入基準）
- マルチLLM役割分担（PM / 書記 / 実装 / デザイン / ユーザ観点）
- PDCAの状態機械（停止条件、エスカレーション、報告）
- GitHubを中心とした成果物管理（Issue / PR / CI / マイルストーン）
- 監査ログ、権限、暴走防止、安全設計
- Discord連携（承認・通知）

**含まない：**
- 実装手順（コード作成・設定ファイル例・具体コマンド・インフラIaC）
- 特定プロダクト（暗号通貨等）固有の要件

### 1.4 非目標

本システムは次を目標としない：

- 任意のプロジェクトを無条件に全自動で完成させること
- 人間の意思決定を完全に不要にすること

安全・監査・説明可能性を優先し、必要時に人間介入を要求する。

| ステークホルダー | 責務 |
|----------------|------|
| ユーザー（Owner） | 最終到達点の提示 / 重要な意思決定（危険操作の承認・停止解除など） |

| Reviewer | レビュー閲覧・コメント |
| Auditor | 監査ログ閲覧・エクスポート |
| システム（自動開発基盤） | PDCAの自動運転 / マイルストーン進捗報告 / 変更の追跡・説明可能性の担保 |

## 3. 用語定義

| 用語 | 定義 |
|------|------|
| **最終到達点** | ユーザーが与える「完成状態の定義」。受入条件・制約・禁止事項を含む |
| **マイルストーン** | 最終到達点へ向けた中間成果。各マイルストーンに DoD を持つ |
| **DoD（Definition of Done）** | 機械的に判定可能な達成条件（例：テスト合格、ドキュメント更新、レビュー解決） |
| **PDCAサイクル** | PLAN / DO / CHECK / ACT の1周 |
| **ペルソナレビュー** | 同一差分に対して観点固定の複数レビューを実施する方式 |
| **バイブコーディング** | 自然言語のゴール入力から AI が自律的に開発サイクルを回す開発スタイル |
| **不信入力** | Issue本文・PR本文・PRコメント・README・CIログ等、外部からシステムに流入するすべてのテキスト |
| **ADR（Architecture Decision Record）** | アーキテクチャ上の意思決定を記録する文書 |
| **SLO** | Service Level Objective。性能・信頼性の目標値 |
| **縮退モード** | 外部障害時に機能を限定して運用継続する状態 |
| **blocker / major / minor** | レビュー指摘の重大度区分（→ 8.1参照） |

## 4. システム構成（論理アーキテクチャ）

### 4.1 構成要素

```
┌──────────────────────────────────────────────────────────┐
│  ユーザー / Owner  →  最終到達点の入力・承認操作           │
└────────────────────┬─────────────────────────────────────┘
                     ↓
┌────────────────────────────────────────────────────────────┐
│  オーケストレータ（PDCA制御）                               │
│  ・状態機械（PLAN/DO/CHECK/ACT）                           │
│  ・タスク分解・依存管理                                    │
│  ・停止条件監視                                            │
└────┬──────────┬──────────┬──────────┬─────────────────────┘
     ↓          ↓          ↓          ↓
┌─────────┐ ┌─────────┐ ┌─────────┐ ┌──────────────────────┐
│LLMゲート│ │実行サン │ │成果物基盤│ │永続ストレージ         │
│ウェイ   │ │ドボックス│ │(GitHub) │ │・PDCA状態            │
│(モデル統│ │(ビルド/ │ │Issue/PR │ │・監査ログ            │
│一I/F,鍵 │ │テスト/  │ │CI/Mile  │ │・レビューJSON        │
│管理,ログ│ │静的解析)│ │stone    │ │・メモリ/RAG          │
└─────────┘ └─────────┘ └─────────┘ └──────────────────────┘
                                           ↓
                              ┌────────────────────────────┐
                              │可視化・通知                 │
                              │Dashboard / Discord / GitHub│
                              └────────────────────────────┘
```

### 4.2 各コンポーネントの要件概要

**LLMゲートウェイ：** 複数モデルの統一呼び出し・鍵管理・レート制限・コスト可視化・ログ。OpenAI互換API相当のインターフェースを提供する。プロジェクト単位の仮想キー（権限分離）を持ち、障害時はフォールバック（別モデル or 縮退）を行う。

**オーケストレータ：** PDCA状態機械の中核。タスク分解・レビュー統合・進捗更新・停止条件監視を担う。

> **【確定】実装仕様**
> - 実装基盤：**Pythonスクリプト**
> - LLM呼び出し方式：**同期（逐次）**
> - 1サイクルのタイムアウト上限：**6時間**

**実行サンドボックス：** Allowlist制。権限最小化（書込範囲・ネットワーク・OS権限）。監査ログ必須。危険コマンド・秘密情報の出力・外部送信を禁止する。

> **【確定】サンドボックス実装方式：Dockerコンテナ + ホワイトリスト管理**
>
> **コンテナ設計要件：**
> - Claudeが生成したコードのビルド・テスト・静的解析はすべてDockerコンテナ内で実行する
> - コンテナはプロジェクト・サイクルごとに起動し、終了後に破棄する（ステートレス）
> - コンテナはrootで実行しない（非rootユーザーを必須とする）
> - ホストのファイルシステムへのマウントは読み取り専用かつ最小範囲に限定する
>
> **ネットワーク制御（ホワイトリスト管理）：**
>
> | 許可対象 | 用途 |
> |---------|------|
> | GitHub API（api.github.com） | PR作成・CI取得 |
> | パッケージレジストリ（npm / PyPI / 各言語標準） | 依存解決 |
> | LLM API（Anthropic / OpenAI / Google / xAI） | モデル呼び出し |
>
> - 上記以外の外部通信はすべて遮断する（デフォルト拒否）
> - ホワイトリストの変更はA操作（人間承認必須）とする
>
> **イメージ管理：**
> - ベースイメージはプロジェクトのビルド要件に応じて定義し、Dockerfileをリポジトリで版管理する
> - イメージの更新はPRとして提出し、通常のPDCAフローでレビューする
> - 脆弱性スキャン（例：Trivy）をイメージビルド時のDoDに含める

**成果物基盤（GitHub）：** Issue / PR / CI / マイルストーンの一元管理。

**永続ストレージ：** PDCA状態・監査ログ・レビュー原本・RAGメモリを保持する。

> **【確定】状態の永続化方式：GitHub Issues をステートストアとして使用する**
> - PDCAの状態はすべてGitHub Issues / PR / Milestone のデータとして表現・管理する
> - 外部DBは持たない。リカバリは GitHub上の最新状態を読み込むことで実現する
> - 監査ログ・レビュー原本はGitHubリポジトリ内のファイルとして保存する（→ 16章参照）

**可視化・通知：** ダッシュボード（現在の目標・進捗・未解決ブロッカー）とDiscord通知を提供する。

## 5. マルチLLM役割分担

### 5.1 役割定義

| 役割 | 計画モデル | 主な責務 | 評価軸 |
|------|-----------|---------|--------|
| **PM** | ADR-001参照 | ロードマップ化 / ACT採否・優先度・スコープ調整 / 停止継続判断 / 進捗報告 | スコープ判断・リスク評価・説明可能性 |
| **サブPM/書記** | ADR-001参照 | 要件・仕様・ADR・Runbook作成更新 / 変更履歴記録 / 整合性監査 | 整合性・抜け漏れ検出・版管理品質 |
| **プログラマ** | ADR-001参照 | 実装品質観点レビュー（正しさ / 保守性 / 性能 / セキュリティ） | 欠陥検出率・セキュリティ観点 |
| **デザイナ** | ADR-001参照 | UI/UX観点レビュー（導線 / 情報設計 / アクセシビリティ / 一貫性） | UX観点の網羅性・矛盾検出 |
| **ユーザ** | ADR-001参照 | 利用者視点レビュー（混乱 / 不便 / 誤操作 / 学習コスト / 期待とのズレ） | 不満・混乱点の発見率 |

### 5.2 DOフェーズの成果物

DOフェーズで更新可能な成果物は以下に限定する：ソースコード、テスト、ドキュメント（PRD / SRS / ADR / Runbook）、UI仕様（画面仕様・コンポーネント仕様）。すべての変更は差分として追跡可能（GitHub PR）であること。

> **【確定】DOフェーズの実装担当**
> - コード生成・編集の実行主体：**Claude**（Anthropic API経由）
> - PLANで定義されたタスクをClaudeに指示し、生成されたコードをサンドボックスでビルド・テスト検証後にPRとして提出する
> - Claudeへの指示プロンプトは書記が仕様書から自動生成し、オーケストレータが実行する

### 5.3 モデル劣化検知

> **【確定】以下の閾値を採用する**

以下の条件が**連続10サイクル**続いた場合、当該ペルソナを縮退・切替候補として扱う：

| 指標 | 劣化と判定する閾値 |
|------|-----------------|
| blocker検出率 | 全ペルソナ平均の **50%以下** |
| 再提出率（JSON不正・形式エラー） | **20%超** |
| PMとのNO判定率（対立解消でPMに却下される率） | **40%超** |

**切替時のルール：**
- 切替判断は **Ownerが行い、ADRに記録する**
- 自動切替は行わない（人間判断を必須とする）
- 切替候補として検出された時点でDiscord通知（警告）を送信する
- 切替後は10サイクルで新モデルの同指標を再評価する

## 6. PDCA状態機械

### 6.1 状態遷移概要

```
[最終到達点の入力]
        ↓
   PLAN（計画立案）
        ↓
    DO（成果物更新）
        ↓
   CHECK（レビュー・CI検証）
        ↓
    ACT（採否・方針決定）
        ↓
  ┌── DoD達成？ ──┐
  | Yes              | No（継続）
  ↓                 ↓
マイルストーン更新  次サイクルPLANへ
  ↓
  最終到達点達成？
  | Yes
  ↓
  完了報告
```

停止条件が発火した場合は任意のフェーズから即座に停止する（→ 6.5参照）。

### 6.2 PLAN要件

**入力：** 最終到達点・現マイルストーン・既知の制約・直近のCHECK結果
**出力：** 直近1サイクルのタスク（最大 **7件**）、各タスクのDoD（機械判定できる形式）、リスクと回避策、依存関係（ブロッカー）

### 6.3 DO要件

- PLANで定義したタスクに基づき、第5.2節に列挙した成果物を更新すること
- 変更はすべてGitHub PRとして追跡可能であること
- 変更種別（ソースコード / テスト / ドキュメント / 設定 / 依存関係 / バイナリ）を分類し、種別ごとのゲートを適用すること（→ 9.3参照）

### 6.4 CHECK要件

**入力：** PR差分・CI結果・DoD・関連ドキュメント差分
**実施：** 自動チェック（CI結果の収集と要約）、5ペルソナレビュー、指摘の重複排除と優先度付け
**出力：** 統合レビューサマリ（重大度別の指摘一覧）、DoD達成判定（達成 / 未達と理由）

### 6.5 ACT要件

**入力：** 統合レビュー・DoD判定・失敗履歴
**出力：** 指摘の採否（理由付き）、次サイクルの方針（優先度・スコープ調整）、マイルストーン進捗の更新、進捗報告の生成と配信
**決定ログに必須記録する項目：** 決定（採否 / 延期 / 中止 / 縮退）・理由・影響範囲・再検討条件

### 6.6 停止条件（必須）

以下のいずれかが発生した場合、PDCAを即停止する：

| 条件 | 既定閾値 |
|------|---------|
| 重大インシデント（秘密情報露出疑い・危険操作検知・権限逸脱） | 即停止 |
| CI連続失敗 | **5回** |
| 変更量（diff）超過 | 追加/変更行合計 **2,000行** or 単一ファイル **600行** |
| 監査ログ不整合 | 即停止 |
| 同一エラーの連続リトライ | **2回**（3回目は人間承認待ち） |
| **1サイクルタイムアウト** | **6時間**（処理停滞状態が6h継続） |
| ユーザーの停止指示 | 即停止 |

### 6.7 スコープ逸脱の扱い

変更が最終到達点の禁止事項またはマイルストーンDoDと無関係な場合は「逸脱」と判定し、ACTで差し戻し / 別マイルストーン化 / スコープ変更（人間承認）のいずれかに分類する。

## 7. マイルストーン・DoD管理

### 7.1 マイルストーン生成原則

マイルストーンは以下を必ず満たすこと：成果物が明確、DoDが機械判定可能、依存関係が明示されている。

**粒度：** 1マイルストーンあたり最大タスク数 **30件**。1サイクルで扱うタスク数は最大 **7件**。巨大タスクは分割を強制する。

**依存ブロッカー：** ブロッカーが残る場合は代替案またはスコープ調整案を必ず併記する。

### 7.2 マイルストーン標準テンプレ

| マイルストーン | 目的 |
|--------------|------|
| M0 基盤 | 開発環境・CI・リポジトリ整備 |
| M1 仕様と骨格 | 要件定義・基本設計・スケルトン |
| M2 主要機能 | コア機能の実装とテスト |
| M3 品質と運用 | 品質改善・セキュリティ・監視・Runbook |
| M4 リリース | 最終受入・リリース準備 |

### 7.3 最終到達点テンプレ

ユーザー入力に必須項目として以下を含めること。欠落があればシステムが差分質問として提示する。

- 目的（何を実現するか）
- 受入条件（達成状態の判定条件） ※必須
- 制約（技術・納期・運用・互換性） ※必須
- 禁止事項（やってはいけない変更・触れてはいけない領域） ※必須
- 優先度（望ましい）

## 8. ペルソナレビュー仕様

### 8.1 重大度分類

| 重大度 | 定義 | 処置 |
|--------|------|------|
| **blocker** | 安全性・正しさ・機密・運用停止に直結 | 未解決なら原則マージ禁止 |
| **major** | 信頼性 / UX / 性能 / 整合性に大きく影響 | 原則次サイクルで解決 |
| **minor** | 軽微改善・リファイン・スタイル | 解決は任意・バックログ化可 |

### 8.2 共通出力形式

全レビュアは**機械処理可能なJSON**で出力する。各指摘は severity, category, basis, suggestion, confidence を含む。

### 8.3 統合ルール

- **JSON検証：** 不一致は再提出要求（最大1回）
- **重複排除：** 近似一致でクラスタリング。統合後も「どのレビュアが言ったか」「根拠」を追跡可能とする
- **優先度：** `重大度 × 確信度 × ペルソナ重み` で算出
- **対立解消：** 矛盾はPMにエスカレーション。採否理由をADR/変更履歴に記録必須

> **【確定】PMも判断できない場合のエスカレーション**
> - 人間（Owner）へDiscord通知でエスカレーションする

**ペルソナ重み（初期値）：**

| ペルソナ | 重み |
|---------|------|
| programmer | 1.00 |
| pm | 0.95 |
| designer | 0.85 |
| user | 0.85 |
| scribe | 0.80 |

> **【確定】ペルソナ重みの調整方針**
> - 調整トリガー：10サイクル連続の観測パターン
> - 変更幅は1回につき **±0.05 以内**
> - 変更はB操作（ペルソナ3承認）として処理する
> - いずれのペルソナの重みも **0.50 を下回らない**（最低保証）

## 9. GitHub連携要件

### 9.1 マイルストーン管理

作成 / 更新 / クローズが可能。DoDを紐付けること。

### 9.2 Issue / PR運用

PLANのタスクはIssueとして生成可能。ラベル（優先度・領域・リスク）を付与。DOの変更はPRとして提出。CHECK結果をPRにコメント。blockerが残る場合は原則マージ禁止。

### 9.3 変更種別ごとのゲート

| 変更種別 | 追加要件 |
|---------|---------|
| 依存関係変更 | 目的と影響範囲を明記 / 脆弱性スキャン・互換性確認をDoDに含める |
| 設定変更（権限 / CI / ポリシー） | 重要変更扱い（原則人間承認）/ 変更理由とロールバック方針を記録 |
| バイナリ/大型ファイル | 原則禁止。例外は人間承認 |

### 9.4 例外マージ（緊急時）

> **【確定】定義例A採用：条件を列挙型（厳格）**

blockerが残る状態でのマージは原則禁止。以下の**4条件をすべて満たす**場合のみ例外マージを許可する：

**条件1（事業影響）：** 本番障害の復旧 or 法的期限が24時間以内
**条件2（リスク限定）：** blockerの影響範囲が特定モジュールに限定
**条件3（追跡保証）：** 残存blockerを追跡するIssueが自動生成
**条件4（承認）：** Owner（Discord）の承認を得ていること（A操作 → 4/4承認）

### 9.5 CI連携

CI必須ゲート（常時）：lint / unit test / secrets混入検知。CI条件付きゲート（プロジェクト種別でON）：型チェック / 統合テスト / E2E / 依存脆弱性 / SBOM / アクセシビリティ。

> **【確定】CI失敗の分類と対処（Claude一任）**

| 分類 | 判断基準の例 | 対処 |
|------|------------|------|
| コード起因 | 毎回同じ箇所で確実に失敗 | DO で修正を優先 |
| インフラ起因 | ネットワーク / 外部サービス / タイムアウト系エラー | 自動リトライ（最大2回）→ 解消しなければ停止・報告 |
| フレーキー | 断続的に失敗 | 再実行で確認→ 継続失敗ならコード起因として扱う |

### 9.6 GitHub App 権限

> **【確定】認証方式：GitHub App**

| 権限 | スコープ | 用途 |
|------|---------|------|
| Contents | Read & Write | コードのPR作成・ブランチ操作 |
| Issues | Read & Write | Issue生成・ラベル付与・クローズ |
| Pull requests | Read & Write | PRの作成・コメント投稿 |
| Actions | Read | CIステータス収集 |
| Checks | Read & Write | CHECKフェーズのステータス書き込み |
| Metadata | Read | リポジトリ情報取得（必須） |
| Milestones | Read & Write | マイルストーン管理 |
| Workflows | Read | CI設定の読み取り（書き込みはA操作扱い） |

権限はリポジトリ単位で付与し、Organization全体への付与は行わない。

## 10. ユーザーフロー（操作要件）

### 10.1 最終到達点の入力

> **【確定】入力UI：Web UI**（※ Web UIでのゴール入力が前提条件）

要件定義確定フロー：ユーザーが入力 → システムが欠落検出・差分質問 → ユーザーが「要件定義完了」ボタン → 5ペルソナレビュー → blockerなければPDCA開始

### 10.2 運転モード

| モード | 定義 |
|--------|------|
| **手動承認モード** | 重要操作に人間承認が必要 |
| **半自動モード** | 一定条件を満たす場合のみ自動進行 |
| **全自動モード** | 事前定義した範囲内で自動進行。逸脱検知時は即停止 |

### 10.3 進捗閲覧

現在のマイルストーンとDoDを常に可視化できること。直近サイクルのPLAN / CHECK / ACTを一画面で追えること。

### 10.4 介入操作

いつでも停止できること。停止後に「原因分析レポート」「再開条件」「ロールバック候補」を提示できること。ユーザーが採否の上書き / 優先度変更 / マイルストーンの再編を行えること。

## 12. セキュリティ要件

### 12.1 想定脅威

プロンプト注入、依存関係汚染、権限昇格、データ流出。

### 12.2 不信入力の処理

Issue本文 / PR本文等はすべて「不信入力」として扱い、命令ではなく素材として処理する。プロンプトの指示階層が不信入力によって上書きされないこと。

### 12.3 秘密情報管理

リポジトリ・ログ・レポートに平文保存を禁止。LLM入力へ Confidential データを送らない。

### 12.4 権限分離

GitHub権限を最小化。LLMキーは役割・環境ごとに分離できること。

### 12.5 依存関係・SBOM

依存ライブラリのSBOM生成と脆弱性スキャンをDoDに含める。

### 12.6 データ分類

| 分類 | 定義 | 取り扱い |
|------|------|---------|
| Public | 外部公開可能 | 制限なし |
| Internal | 社内 / プロジェクト内限定 | 外部送信に注意 |
| Confidential | 鍵 / 個人情報 / 機密仕様 | ログ不可・LLM入力マスク・最小権限閲覧 |

### 12.7 プロンプト注入レッドチーム

代表攻撃パターンに対して指示階層が維持され、禁止事項が破られないことを検証できること。

## 13. 信頼性・SLO要件

### 13.1 SLO目標値

> **【確定】初期運用30サイクル後に実績ベースで見直す**

| 指標 | 目標値 | 上限 |

### 13.2 縮退モード

> **【確定】定義例A採用：人間への報告を最優先とする縮退方針**

| 優先度 | 機能 | 障害時の扱い |
|--------|------|------------|

| 3 | CHECKレポートの生成 | 代替モデルで補完 |

### 13.3 復旧

失敗原因の分類・記録。自動リトライ上限：同一失敗に対して **2回**。

### 13.4 環境区分

dev / stg / prod の3環境。重要ポリシー変更は stg で検証後に prod へ昇格する。
