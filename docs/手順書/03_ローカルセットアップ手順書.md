## 1. このドキュメントについて

このガイドは、バイブコーディングPDCA自動開発システムを **ゼロからローカル環境に構築し、本番稼働させるまで** の全手順を、初心者でもわかるように解説したものです。

### 誰のためのドキュメント？

- 本システムを初めてセットアップする開発者
- コードの中身は詳しくないが、環境構築を任された運用担当者
- ローカルLLMを使った開発環境を用意したいチームメンバー

### 前提条件

| 項目 | 必要なもの |
|------|-----------|
| OS | macOS / Linux / Windows（WSL2推奨） |
| Python | 3.12 以上 |
| Git | インストール済み |
| メモリ | 最低 16GB（ローカルLLM使用時は 32GB 以上推奨） |
| ディスク | 最低 50GB 空き（ローカルLLMモデル含む） |
| GPU | ローカルLLM使用時は NVIDIA GPU 推奨（8GB VRAM 以上） |

> **💡 ヒント**: クラウドLLMのみ使う場合は GPU 不要です。ローカルLLMを使う場合のみ GPU が必要になります。

## 3. Step 1：Python 環境の構築

### 3.1 Python のバージョン確認

ターミナル（コマンドプロンプト）を開いて、以下のコマンドを入力してください。

```bash
python3 --version
```

**期待される出力**：

```
Python 3.12.x
```

> `3.12` 以上の数字が表示されればOKです。`3.11` 以下の場合は、次の「3.2 Python のインストール」を実施してください。

### 3.2 Python のインストール（未インストール/バージョンが古い場合）

#### macOS の場合

```bash
# Homebrew がない場合はまずインストール
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Python 3.12 のインストール
brew install python@3.12
```

#### Ubuntu / Debian の場合

```bash
sudo apt update
sudo apt install -y python3.12 python3.12-venv python3-pip
```

#### Windows の場合

1. [python.org](https://www.python.org/downloads/) から Python 3.12 以上をダウンロード
2. インストーラーを実行（**「Add Python to PATH」にチェックを入れる**）
3. WSL2 を使う場合は、WSL2 内で Ubuntu の手順を実施

### 3.3 仮想環境の作成（推奨）

プロジェクト専用の Python 環境を作ることで、他のプロジェクトと依存パッケージが混ざるのを防ぎます。

```bash
# 仮想環境を作成する（.venv というフォルダが作られます）
python3 -m venv .venv

# 仮想環境を有効にする
# macOS / Linux の場合:
source .venv/bin/activate

# Windows の場合:
.venv\Scripts\activate
```

> **確認方法**: コマンドラインの先頭に `(.venv)` と表示されていればOKです。

### ✅ 確認ポイント

```bash
python3 --version
# → Python 3.12.x が表示されること
```

```bash
python -c "import vibe_pdca; print('OK')"
```

**期待される出力**：

```
OK
```

```bash
python -c "import vibe_pdca; print('OK')"

pytest --co -q --ignore=tests/test_gui.py 2>&1 | tail -1
```

## 5. Step 3：API キーの設定（.env ファイル）

### 5.1 .env ファイルの作成

```bash
# テンプレートをコピー
cp .env.example .env
```

### 5.2 .env ファイルの編集

お好みのテキストエディタで `.env` ファイルを開きます。

```bash
# nano（ターミナル内エディタ）の場合
nano .env

# VS Code の場合
code .env
```

### 5.3 各APIキーの取得方法と設定

#### OpenAI（PM担当：GPT-5.1）

1. [OpenAI Platform](https://platform.openai.com/) にアクセスしてログイン
2. 右上のアイコン → 「API keys」をクリック
3. 「Create new secret key」をクリック
4. 名前を入力して「Create secret key」をクリック
5. 表示されたキーをコピー

```env
OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

> ⚠️ キーは一度しか表示されません。必ずコピーして安全な場所に保管してください。

#### Anthropic（プログラマ/DO担当：Claude）

1. [Anthropic Console](https://console.anthropic.com/) にアクセスしてログイン
2. 左メニュー「API Keys」をクリック
3. 「Create Key」をクリック
4. 名前を入力して作成
5. 表示されたキーをコピー

```env
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

#### Google（書記/デザイナ担当：Gemini）

1. [Google AI Studio](https://aistudio.google.com/) にアクセスしてログイン
2. 「Get API key」をクリック
3. 「Create API key」をクリック
4. プロジェクトを選択（または新規作成）
5. 表示されたキーをコピー

```env
GOOGLE_API_KEY=AIzaxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

#### xAI（ユーザ担当：Grok）

1. [xAI Console](https://console.x.ai/) にアクセスしてログイン
2. APIキーを生成
3. 表示されたキーをコピー

```env
XAI_API_KEY=xai-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

### 5.4 環境の選択

`.env` ファイルで動作環境を設定できます。

```env
# 開発環境（デフォルト）
VIBE_PDCA_ENV=dev

# ローカルLLMを優先使用する場合（クラウドAPIキーがなくても動作）
# VIBE_PDCA_LLM_MODE=local
```

### 5.5 設定の確認

```bash
# .env ファイルが正しく読み込めるか確認
python -c "
from dotenv import load_dotenv
import os
load_dotenv()
for key in ['OPENAI_API_KEY', 'ANTHROPIC_API_KEY', 'GOOGLE_API_KEY', 'XAI_API_KEY']:
    val = os.getenv(key, '未設定')
    masked = val[:8] + '...' if len(val) > 8 else val
    print(f'{key}: {masked}')
"
```

> **💡 ヒント**: ローカルLLMのみ使う場合は、クラウドAPIキーの設定は省略できます。`.env` で `VIBE_PDCA_LLM_MODE=local` を設定してください。

### ✅ 確認ポイント

- `.env` ファイルが存在すること
- 少なくとも1つ以上のAPIキーが設定されていること（またはローカルLLMモードに設定済み）
- `.env` ファイルが `.gitignore` に含まれていること（APIキーをGitにコミットしない）

## 6. Step 4：ローカル LLM（Ollama）のセットアップ

> 💡 **ローカル LLM とは？**  
> ChatGPTやClaudeのようなAIを、インターネットを経由せずに自分のパソコンで動かす仕組みです。  
> **Ollama** はそのためのツールで、簡単にAIモデルをダウンロード・起動できます。  
> 💡 詳しくは → [AI推論サーバー説明書](../説明書/AI推論サーバー説明書.md#-はじめに--この説明書を読む前に)

> **⚡ クラウドLLMのみ使う場合はこのステップを省略できます。**  
> ただし、クラウドLLMの障害時にローカルLLMへ自動フォールバックする機能を使う場合はセットアップが必要です。

### 6.1 Ollama のインストール

#### macOS / Linux の場合

```bash
curl -fsSL https://ollama.com/install.sh | sh
```

#### Windows の場合

1. [Ollama 公式サイト](https://ollama.com/download) からインストーラーをダウンロード
2. インストーラーを実行

### 6.2 Ollama の起動確認

```bash
# Ollama サーバーの起動（バックグラウンドで動作します）
ollama serve &

# 起動確認
curl http://localhost:11434/api/tags
```

**期待される出力**（モデル未ダウンロード時）：

```json
{"models":[]}
```

### 6.3 役割別モデルのダウンロード

本システムでは、AIの役割ごとに最適化されたモデルを使い分けます。

```bash
# PM・書記用（日本語・推論に強いモデル / 約 40GB）
ollama pull qwen3:72b

# プログラマ・DO用（コード生成に強いモデル / 約 12GB）
ollama pull codestral:22b

# デザイナ用（汎用・高品質モデル / 約 40GB）
ollama pull llama3.3:70b

# ユーザ用（軽量モデル / 約 15GB）
ollama pull gemma3:27b
```

> **⚠️ ダウンロードサイズについて**: 全モデルをダウンロードすると約 100GB 以上必要です。ディスク空き容量を確認してください。

> **💡 軽量構成にする場合**: まず `codestral:22b`（12GB程度）だけダウンロードして動作確認し、後から他のモデルを追加できます。

### 6.4 ダウンロードしたモデルの確認

```bash
ollama list
```

**期待される出力**：

```
NAME              ID           SIZE    MODIFIED
qwen3:72b         xxxx         40 GB   1 minute ago
codestral:22b     xxxx         12 GB   2 minutes ago
llama3.3:70b      xxxx         40 GB   3 minutes ago
gemma3:27b        xxxx         15 GB   4 minutes ago
```

### 6.5 動作テスト

```bash
# 簡単なプロンプトでモデルの動作確認
ollama run codestral:22b "Hello, write a Python hello world function."
```

応答が返ってくればOKです。`Ctrl+D` で対話モードを終了できます。

### 6.6 モデルの差し替え（オプション）

特定のモデルを使いたい場合は、`.env` ファイルで上書きできます。

```env
# 全役割を同じモデルにする場合
VIBE_PDCA_LOCAL_LLM_MODEL=llama3.3:70b

# 役割別に個別指定する場合（上の一括指定より優先）
VIBE_PDCA_LOCAL_LLM_MODEL_PM=qwen3:72b
VIBE_PDCA_LOCAL_LLM_MODEL_PROGRAMMER=codestral:22b
```

### ✅ 確認ポイント

```bash
# Ollama が動作していること
curl -s http://localhost:11434/api/tags | python3 -c "
import json, sys
data = json.load(sys.stdin)
models = [m['name'] for m in data.get('models', [])]
print(f'ダウンロード済みモデル: {len(models)}個')
for m in models:
    print(f'  - {m}')
"
```

## 7. Step 5：GitHub App の作成と設定

### 7.1 GitHub App とは？

GitHub App は、本システムがGitHub上でIssue（やることリスト）作成・PR（変更依頼）作成・CI取得などの操作を行うための「専用アカウント」のようなものです。Personal Access Token（個人トークン）よりも安全で、リポジトリ単位で権限を管理できます。

> 💡 詳しくは → [オンボーディング説明書](../説明書/オンボーディング説明書.md#-この説明書で使われる用語) の用語集「GitHub App」を参照

### 7.2 GitHub App の作成手順

1. GitHub にログインし、右上のアイコン → 「Settings」をクリック

2. 左メニューの一番下「Developer settings」をクリック

3. 「GitHub Apps」→「New GitHub App」をクリック

4. 以下の項目を入力：

   | 項目 | 入力内容 |
   |------|---------|
   | GitHub App name | `vibe-pdca-system`（一意な名前） |
   | Homepage URL | `https://github.com/your-org/your-repo` |
   | Webhook Active | チェックを**外す**（Webhookは使わない） |

5. **Permissions**（権限）を以下の通り設定：

   **Repository permissions（リポジトリ権限）**:

   | 権限 | レベル | 用途 |
   |------|--------|------|
   | Contents | **Read & write** | コードのPR作成・ブランチ操作 |
   | Issues | **Read & write** | Issue生成・ラベル付与・クローズ |
   | Pull requests | **Read & write** | PRの作成・コメント投稿 |
   | Actions | **Read-only** | CIステータス収集 |
   | Checks | **Read & write** | CHECKフェーズのステータス書き込み |
   | Metadata | **Read-only** | リポジトリ情報取得（必須） |
   | Milestones | **Read & write** | マイルストーン管理 |
   | Workflows | **Read-only** | CI設定の読み取り |

   > ⚠️ **重要**: 「Workflows」の「Write」権限は**付与しないでください**。CI設定変更はA操作（人間承認必須）です。

6. 「Where can this GitHub App be installed?」は「**Only on this account**」を選択

7. 「Create GitHub App」をクリック

### 7.3 秘密鍵の生成

1. 作成された GitHub App の設定ページを開く
2. 一番下の「Private keys」セクションで「Generate a private key」をクリック
3. `.pem` ファイルが自動ダウンロードされる
4. このファイルを安全な場所に保存する

```bash
# 秘密鍵をプロジェクト外の安全な場所に移動
mkdir -p ~/.vibe-pdca
mv ~/Downloads/vibe-pdca-system.*.private-key.pem ~/.vibe-pdca/github-app.pem
chmod 600 ~/.vibe-pdca/github-app.pem
```

### 7.4 App ID の確認

GitHub App の設定ページの一番上に「App ID」が表示されています。この番号をメモしてください。

### 7.5 リポジトリへのインストール

1. GitHub App の設定ページで「Install App」タブをクリック
2. 対象のOrganization（またはアカウント）の「Install」をクリック
3. **「Only select repositories」を選択**し、対象リポジトリのみ選択

> ⚠️ **重要**: 「All repositories」は**選択しないでください**。リポジトリ単位でインストールすることがセキュリティ要件です（ADR-005）。

4. 「Install」をクリック

### 7.6 環境変数への設定

`.env` ファイルに以下を追加：

```env
# GitHub App 設定
GITHUB_APP_ID=123456
GITHUB_APP_PRIVATE_KEY_PATH=~/.vibe-pdca/github-app.pem
GITHUB_APP_INSTALLATION_ID=78901234
```

> **Installation ID の確認方法**: インストール後のURLに含まれています。  
> `https://github.com/settings/installations/78901234` ← この数字

### ✅ 確認ポイント

- GitHub App が作成済みであること
- 秘密鍵（`.pem` ファイル）が安全な場所に保存されていること
- 対象リポジトリにのみインストールされていること（Organization全体への付与は禁止）
- `.env` に `GITHUB_APP_ID`, `GITHUB_APP_PRIVATE_KEY_PATH`, `GITHUB_APP_INSTALLATION_ID` が設定されていること

## 8. Step 6：Discord Bot・チャンネルの設定

> 💡 **Discord とは？**  
> テキストや音声でやり取りできるチャットアプリです。本システムでは AIからの通知や「承認してください」という要求の送信先として使います。  
> 💡 詳しくは → [オンボーディング説明書](../説明書/オンボーディング説明書.md#-この説明書で使われる用語) の用語集「Discord」「Discord Bot」を参照

### 8.1 Discord Bot を作る理由

本システムは以下の目的でDiscordを使用します：

- **A操作の承認要求**: 重要な変更（権限拡大・ポリシー変更等）の承認をDiscordで行う
- **B操作の通知**: 中程度の変更のバックアップ完了通知
- **サイクル完了通知**: PDCAサイクルの進捗報告
- **障害通知**: LLMフォールバック発生やインシデントの通知

### 8.2 Discord Bot の作成手順

1. [Discord Developer Portal](https://discord.com/developers/applications) にアクセス

2. 「New Application」をクリック

3. アプリ名（例：`vibe-pdca-bot`）を入力して「Create」

4. 左メニュー「Bot」をクリック

5. 「Add Bot」→「Yes, do it!」をクリック

6. 以下の設定を行う：
   - **PUBLIC BOT**: オフにする（招待リンクでの追加を防ぐ）
   - **MESSAGE CONTENT INTENT**: **オンにする**（メッセージ内容の読み取りに必要）

7. 「Reset Token」をクリックしてBot Tokenを取得

```env
# .env に追加
DISCORD_BOT_TOKEN=MTxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

> ⚠️ Bot Token は一度しか表示されません。必ずコピーして安全に保管してください。

## 9. Step 7：Docker サンドボックスの構築

> 💡 **Docker とは？**  
> プログラムを隔離された安全な空間（「コンテナ」と呼びます）で実行するための仕組みです。  
> たとえるなら、アプリを動かす「小さな仮想パソコン」を作るツールです。  
> 💡 詳しくは → [AI推論サーバー説明書 §7](../説明書/AI推論サーバー説明書.md#7-ソフトウェアスタック構成)

### 9.1 Docker サンドボックスとは？

DOフェーズ（コード生成・テスト実行）は、安全のためDocker コンテナ内で実行されます。これにより：

- 生成されたコードがホストシステムに直接影響を与えない
- ネットワークアクセスがホワイトリスト制御される
- サイクルごとにクリーンな環境で実行される

### 9.2 Docker のインストール

#### macOS の場合

1. [Docker Desktop for Mac](https://docs.docker.com/desktop/install/mac-install/) をダウンロード・インストール

#### Ubuntu の場合

```bash
# Docker Engine のインストール
sudo apt-get update
sudo apt-get install -y docker.io docker-compose-v2

# 現在のユーザーを docker グループに追加（再ログインが必要）
sudo usermod -aG docker $USER
newgrp docker
```

#### Windows の場合

1. [Docker Desktop for Windows](https://docs.docker.com/desktop/install/windows-install/) をダウンロード・インストール
2. WSL2 バックエンドを有効にする

### 9.3 Docker の動作確認

```bash
docker --version
docker run hello-world
```

### 9.4 サンドボックスイメージのビルド

```bash
# プロジェクトのルートディレクトリで実行
docker build -f docker/Dockerfile.sandbox -t vibe-pdca-sandbox .
```

> **ビルドにかかる時間**: 初回は 2〜5 分程度です。

### 9.5 サンドボックスの動作テスト


**期待される出力**：


## 14. トラブルシューティング

### よくある問題と解決策

#### Q1: `pip install -e ".[dev]"` でエラーになる

**考えられる原因**: Python バージョンが 3.12 未満

```bash
# バージョン確認
python3 --version

# バージョンが古い場合は Step 1 の手順でアップデート
```

#### Q2: `ollama pull` でダウンロードが遅い/失敗する

**考えられる原因**: ネットワーク接続の問題、ディスク容量不足

```bash
# ディスク空き容量の確認
df -h .

# 小さいモデルから試す
ollama pull gemma3:27b  # 15GB程度
```

#### Q3: Docker ビルドでエラーになる

**考えられる原因**: Docker デーモンが起動していない

```bash
# Docker の状態確認
docker info

# macOS: Docker Desktop を起動
# Linux:
sudo systemctl start docker
```

#### Q4: テストが失敗する

**考えられる原因**: 依存パッケージのバージョン不整合

```bash
# 依存関係を再インストール
pip install -e ".[dev]" --force-reinstall

# 特定のテストだけ実行して原因を特定
pytest tests/test_config.py -v
```

#### Q5: クラウドLLMに接続できない

**考えられる原因**: APIキーの設定ミス、ネットワーク制限

```bash
# APIキーが正しく設定されているか確認
grep -E "^(OPENAI|ANTHROPIC|GOOGLE|XAI)" .env

# OpenAI への疎通確認
curl -s -o /dev/null -w "%{http_code}" \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  https://api.openai.com/v1/models
# → 200 が返ればOK
```

#### Q6: ローカルLLMの応答が遅い

**考えられる原因**: GPUが使われていない、モデルが大きすぎる

```bash
# GPU の認識確認
nvidia-smi

# より小さいモデルに変更
VIBE_PDCA_LOCAL_LLM_MODEL=gemma3:27b
```

#### Q7: Discord通知が届かない

**考えられる原因**: Webhook URLの設定ミス、Botの権限不足

```bash
# Webhook の疎通テスト
curl -H "Content-Type: application/json" \
  -d '{"content": "テスト通知"}' \
  "$DISCORD_WEBHOOK_INFO"
```

## 15. 用語集

| 用語 | 説明 |
|------|------|
| **PDCA** | Plan-Do-Check-Act の略。計画→実行→検証→改善のサイクル |
| **LLM** | Large Language Model（大規模言語モデル）の略。ChatGPTやClaudeのようなAI |
| **Ollama** | ローカルでLLMを実行するためのソフトウェア |
| **サーキットブレーカー** | クラウドLLMの障害を検知し、ローカルLLMに自動切替する仕組み |
| **A操作** | 権限拡大やポリシー変更など、4人全員のDiscord承認が必要な操作 |
| **B操作** | diff超過や依存更新など、バックアップ後に自動実行される操作 |
| **C操作** | 軽微な変更。承認不要で自動実行される |
| **GitHub App** | GitHubのIssue/PR操作を安全に行うための仕組み |
| **Webhook** | 外部サービスにイベントを自動通知する仕組み |
| **Docker** | アプリケーションをコンテナ（隔離環境）で実行する仕組み |
| **サンドボックス** | 安全にコードを実行するための隔離された環境 |
| **ドライラン** | 外部への書き込みを行わないシミュレーション実行 |
| **stg環境** | ステージング環境。本番と同じ構成でテストを行う環境 |
| **フォールバック** | 主要な方法が使えない場合に代替手段に切り替えること |
| **VRAM** | GPUが持つメモリ。ローカルLLMの実行に必要 |
