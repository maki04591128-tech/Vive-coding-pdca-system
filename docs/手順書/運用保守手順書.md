## 1. この手順書について

### 誰のための手順書？

- システムの日常運用を担当する Owner / Maintainer
- 障害発生時の対応を行う方
- 定期メンテナンスを実施する方

### この手順書でできるようになること

```
✅ 毎日のシステム確認が滞りなく行える
✅ 障害が発生した場合の対応手順がわかる
✅ 定期メンテナンス（週次/月次/四半期）の作業がわかる
✅ ハードウェアの保守点検ができる
✅ セキュリティパッチの適用手順がわかる
✅ システムのアップデート手順がわかる
```

### ロール別の責務範囲

| ロール | 日常運用 | 障害対応 | メンテナンス | セキュリティ |
|--------|---------|---------|------------|------------|
| **Owner** | ✅ 全操作 | ✅ P0/P1 対応 | ✅ リリース承認 | ✅ ポリシー決定 |
| **Maintainer** | ✅ 停止/再開/モード切替 | ✅ P1/P2 対応 | ✅ パッチ適用 | ✅ 実施 |
| **Reviewer** | 読み取りのみ | 通知受信 | — | — |
| **Auditor** | 監査ログ閲覧 | 通知受信 | — | ✅ 監査 |

## 2. 日常運用チェックリスト

毎日（または始業時）に以下を確認してください。所要時間：約 10 分。

```bash

python -m vibe_pdca.cli status --project-id <PROJECT_ID>

curl -s http://localhost:8000/health | jq .

# → サンドボックスコンテナが表示されること（実行中の場合）
```

```

□ hw-critical：ハードウェア重大障害が発生していないか

```

## 3. システムの起動・停止・再開

### 3.1 通常起動

```bash
# .env が設定済みであること
cd ~/Vive-coding-pdca-system
source .venv/bin/activate

# 起動
python -m vibe_pdca.orchestrator start \
  --project-id <PROJECT_ID> \
  --env prod
```

起動後の確認：
```
□ Discord pdca-info に起動通知が届いている
□ GitHub に Issue / Milestone が作成されている
□ 監査ログに起動イベントが記録されている
```

### 3.2 通常停止（Graceful Stop）

**ロール**：Owner / Maintainer

```bash
python -m vibe_pdca.cli stop \
  --project-id <PROJECT_ID> \
  --graceful \
  --reason "理由を記載（例：週次メンテナンスのため）"
```

> 通常停止は現在実行中のフェーズが完了してから停止します。

### 3.3 緊急停止（Emergency Stop）

**ロール**：Owner / Maintainer

```bash
python -m vibe_pdca.cli stop \
  --project-id <PROJECT_ID> \
  --emergency \
  --reason "理由を記載（例：P0 インシデント疑い）"
```

> 緊急停止後は「[8. 障害対応](#8-障害対応インシデント管理)」に従いインシデント対応を開始してください。

### 3.4 再開（Resume）

```bash
# 通常再開（Owner / Maintainer）
python -m vibe_pdca.cli resume \
  --project-id <PROJECT_ID> \
  --reason "理由を記載（例：メンテナンス完了）"

# コスト超過後の再開（Owner のみ）
python -m vibe_pdca.cli resume \
  --project-id <PROJECT_ID> \
  --force-cost-override \
  --reason "集中実装フェーズのため上限を一時的に引き上げ"

# P0/P1 後の再開（Owner のみ、Discord 4/4 承認必須）
python -m vibe_pdca.cli resume \
  --project-id <PROJECT_ID> \
  --post-incident \
  --reason "根本原因を特定し再発防止策を実施済み"
```

## 4. A操作の承認フロー

### 4.1 A操作とは

権限拡大・ポリシー緩和・禁止領域変更など、重要な変更操作です。  
Discord 上で全ロール（4 人）の承認が必要です。タイムアウトは 12 時間です。

### 4.2 承認手順

```
1. Discord pdca-critical チャンネルのスレッド内に承認要求通知が届く
2. 内容を確認する：
   - 何をするか
   - なぜ必要か
   - 影響範囲
   - 代替案
   - ロールバック方針
3. 以下のいずれかで回答する：

   承認する場合：
     /approve A-2026-001

   却下する場合（理由必須）：
     /reject A-2026-001 reason:"セキュリティリスクが高いため"

   保留する場合（条件・期限必須）：
     /hold A-2026-001 condition:"テスト結果を確認後" deadline:"2026-03-10"
```

### 4.3 タイムアウト時の対応

12 時間以内に 4/4 承認が揃わない場合は自動的に「保留」になります。  
保留中の A 操作は PDCA サイクルの該当操作をスキップして進行します。

## 5. 運転モードの管理

### 5.1 運転モード一覧

| モード | 推奨場面 | B 操作の扱い |
|--------|---------|------------|
| **手動承認** | 初期・重要フェーズ | Owner 承認後に進行 |
| **半自動** | 安定運用期 | バックアップ後に自動進行 |
| **全自動** | 低リスク改善フェーズ | バックアップ後に自動進行 |
| **縮退** | 障害発生時 | 機能制限あり |

### 5.2 モード切替

**ロール**：Owner / Maintainer

```bash
# 運転モードの切替
python -m vibe_pdca.cli mode set \
  --project-id <PROJECT_ID> \
  --mode manual|semi-auto|full-auto \
  --reason "理由を記載"

# 縮退モードへの移行
python -m vibe_pdca.cli mode set \
  --project-id <PROJECT_ID> \
  --mode degraded \
  --reason "クラウド LLM 全プロバイダ障害"

# 通常モードへの復帰
python -m vibe_pdca.cli mode set \
  --project-id <PROJECT_ID> \
  --mode normal \
  --reason "クラウド LLM 復旧確認済み"
```

### 5.3 LLM モードの切替

```bash
# ローカル LLM モードへ切替
python -m vibe_pdca.cli mode set \
  --project-id <PROJECT_ID> \
  --llm-mode local \
  --reason "コスト削減のためローカルへ切替"

# クラウド LLM モードへ復帰
python -m vibe_pdca.cli mode set \
  --project-id <PROJECT_ID> \
  --llm-mode cloud \
  --reason "レビュー品質向上のためクラウドへ復帰"
```

## 6. コスト管理

### 6.1 コスト確認

```bash
# 本日のコスト
python -m vibe_pdca.cli cost --period today --breakdown model

# 今週のコスト
python -m vibe_pdca.cli cost --period week --breakdown model

# 今月のコスト
python -m vibe_pdca.cli cost --period month --breakdown model
```

### 6.2 コスト上限の変更

**ロール**：Owner のみ

```bash
python -m vibe_pdca.cli config set \
  --project-id <PROJECT_ID> \
  --key daily_cost_limit_usd \
  --value 50.00 \
  --reason "集中実装フェーズのため"
```

### 6.3 コスト削減のアクション

| 状況 | アクション |
|------|-----------|
| 日次コストが上限の 80% | ローカル LLM への切替を検討 |
| 日次コストが上限に到達 | 自動停止。Owner が再開を判断 |
| 月次コストが予算超過 | 運転モードを「手動承認」に変更 |

## 7. 監視とアラート対応

### 7.1 Grafana ダッシュボードで確認する項目

```
常時表示推奨のメトリクス：
  - GPU 温度（80℃ 超えでアラート）
  - GPU VRAM 使用率（95% 超えでアラート）
  - GPU 使用率（推論中: 80〜100% が正常）
  - ディスク使用率（85% 超えでアラート）
  - CPU 使用率（90% 超えが継続でアラート）
  - LLM レイテンシ（30 秒超でアラート）
  - サーキットブレーカー状態（OPEN でアラート）
```

### 7.2 Discord アラート対応表

| アラートレベル | 通知先チャンネル | 対応者 | 対応期限 |
|-------------|----------------|--------|---------|
| ⚠️ 警告（P2） | pdca-warning | Maintainer | 次サイクル |
| 🔴 重大（P1） | pdca-critical | Owner / Maintainer | 4 時間以内 |
| 🚨 緊急（P0） | pdca-critical | Owner | 1 時間以内 |
| ⚙️ HW 警告 | hw-warning | Maintainer | 当日中 |
| 🔥 HW 緊急 | hw-critical | Owner / Maintainer | 1 時間以内 |

### 7.3 アラート対応フロー

```
1. アラート通知を受信
2. アラートの種類を特定（SW 障害 / HW 障害 / コスト / セキュリティ）
3. 以下の該当セクションに従い対応：
   - SW 障害 → 「8. 障害対応」
   - HW 障害 → 「13. ハードウェア保守」
   - コスト  → 「6. コスト管理」
   - LLM    → 「9. LLM フォールバック運用」
4. 対応結果を Discord で報告
5. 必要に応じてインシデントレポートを作成
```

## 8. 障害対応（インシデント管理）

### 8.1 インシデント分類

| 優先度 | 定義 | 自動アクション | 人間対応期限 |
|--------|------|--------------|------------|
| **P0** | 情報漏えい疑い / 権限逸脱 / 重大破壊 | **即時自動停止** | 1 時間以内 |
| **P1** | 運用継続に重大影響 | 縮退モード移行 | 4 時間以内 |
| **P2** | 軽微な異常 | 通知のみ | 次サイクル |

### 8.2 P0 対応手順

```
Step 1：即時確認（1 時間以内）
  □ Discord pdca-critical の P0 通知を確認
  □ 秘密情報の露出がないか確認
  □ 権限設定の変更がないか確認

Step 2：初期調査
  □ 監査ログの確認
    python -m vibe_pdca.cli audit-log --last 50
  □ システムステータスの確認
    python -m vibe_pdca.cli status --project-id <PROJECT_ID>

Step 3：必要に応じてロールバック
  python -m vibe_pdca.cli backup list --project-id <PROJECT_ID>
  python -m vibe_pdca.cli rollback \
    --project-id <PROJECT_ID> \
    --backup-id <BACKUP_ID> \
    --reason "P0 インシデント対応"

Step 4：根本原因の特定
  □ タイムラインの整理
  □ 影響範囲の特定
  □ 根本原因の特定

Step 5：再開
  → 4/4 承認必須
  python -m vibe_pdca.cli resume \
    --project-id <PROJECT_ID> \
    --post-incident \
    --reason "根本原因特定・再発防止策実施済み"

Step 6：インシデントレポートの作成（24 時間以内）
```

### 8.3 P1 対応手順

```
Step 1：確認（4 時間以内）
  □ Discord pdca-critical の P1 通知を確認
  □ 縮退モードに自動移行していることを確認

Step 2：原因調査
  python -m vibe_pdca.cli audit-log --last 20
  python -m vibe_pdca.cli cost --period today

Step 3：対処
  → 原因に応じた対処を実施

Step 4：通常モードへ復帰
  python -m vibe_pdca.cli mode set \
    --project-id <PROJECT_ID> \
    --mode normal \
    --reason "P1 対応完了"

Step 5：インシデントレポートの作成（72 時間以内）
```

## 9. LLM フォールバック運用

### 9.1 フォールバックの仕組み

```
クラウド LLM（通常運用）
  ↓ 障害発生
代替クラウド LLM（フォールバック 1）
  ↓ こちらも障害
ローカル LLM（最終フォールバック）
  ↓ ローカルも障害
PDCA サイクル自動停止
```

### 9.2 フォールバック発生時の対応

```
1. Discord pdca-warning のフォールバック通知を確認
2. フォールバック先を確認（クラウド代替 or ローカル）
3. ローカル LLM の場合はレビュー品質が低下する可能性あり
4. 障害が長引く場合は手動で縮退モードに移行：

   python -m vibe_pdca.cli mode set \
     --project-id <PROJECT_ID> \
     --mode degraded \
     --reason "クラウド LLM 全プロバイダ障害"
```

### 9.3 ローカル LLM のトラブル

```bash
# vLLM の状態確認
systemctl --user status vllm

# vLLM が停止している場合の再起動
systemctl --user restart vllm

# モデルが破損している可能性がある場合
rm -rf ~/.cache/huggingface/hub/models--Qwen--Qwen2.5-72B-Instruct
systemctl --user restart vllm
# → モデルが再ダウンロードされる
```

## 11. システムアップデート・リリース

### 11.1 アップデート前の準備

```
□ リリースノートを確認
□ バックアップを作成（手動）
□ A 操作として 4/4 承認を取得
□ メンテナンス時間を Discord で周知
```

### 11.2 アップデート手順

```bash
# Step 1：システムを停止
python -m vibe_pdca.cli stop --env prod --graceful \
  --reason "リリース作業"

# Step 2：バックアップの確認
python -m vibe_pdca.cli backup list --project-id <PROJECT_ID>

# Step 3：コードの更新
cd ~/Vive-coding-pdca-system
git pull origin main

# Step 4：依存関係の更新
source .venv/bin/activate
pip install -e ".[dev]" --upgrade

# Step 5：マイグレーション（必要な場合）
python -m vibe_pdca.cli migrate --env prod

# Step 6：設定の検証
python -m vibe_pdca.cli config validate --env prod

# Step 7：再起動
python -m vibe_pdca.orchestrator start \
  --project-id <PROJECT_ID> \
  --env prod
```

### 11.3 アップデート後の確認

```
□ システムステータスが running
□ Discord に起動通知が届いている
□ 監査ログにエラーがない
□ CI が正常に動作している
□ 1 サイクル正常完了を確認
□ P0 / P1 が発生していない
```

### 11.4 ロールバック判断基準

以下のいずれかに該当する場合は即ロールバックしてください。

```
・システムが起動しない
・監査ログの書き込みが失敗する
・A 操作フローが動作しない
・P0 / P1 インシデントが発生する
```

## 12. 定期メンテナンス

### 12.1 週次メンテナンス（毎週月曜日推奨）

所要時間：約 30 分

```
□ 承認待ち渋滞の棚卸し
    → 12 時間以上保留の A 操作がないか確認
□ Suppress List の確認
    → 有効期限切れのエントリがないか確認
□ コスト推移の確認
    → 今週のコスト合計が予算内か確認
□ CI 成功率の確認
    → 連続失敗パターンがないか確認
□ Discord 通知の健全性確認
    → 通知が正常に届いているか確認
```

### 12.2 月次メンテナンス（毎月第 1 月曜日推奨）

所要時間：約 1〜2 時間

```
□ セキュリティパッチの適用
    sudo apt update && sudo apt upgrade -y
□ Docker イメージの更新
    docker pull nvidia/cuda:12.4.0-base-ubuntu22.04
    docker build -t vibe-sandbox -f docker/Dockerfile.sandbox .
□ vLLM の更新
    source ~/vllm-env/bin/activate
    pip install --upgrade vllm
□ 不要な Docker イメージ・コンテナの削除
    docker system prune -a --volumes
□ ディスク使用量の棚卸し
    du -sh ~/.cache/huggingface/
    du -sh /opt/prometheus/data/
□ バックアップの棚卸し
    → 古いバックアップの削除（保持期間: 監査ログ 365日 / 状態 90日 / メモリ 180日）
□ Prometheus / Grafana のデータ量確認
□ UPS バッテリーのステータス確認
□ GPU ECC エラーの確認
    nvidia-smi --query-gpu=ecc.errors.corrected.aggregate.total --format=csv
```

### 12.3 四半期メンテナンス（3ヶ月ごと推奨）

所要時間：約半日

```
□ ドキュメントの棚卸し（必要なドキュメント一覧.md を参照）
□ Suppress List の全件棚卸し

    → 不要な権限がないか確認
□ API キー・トークンの棚卸し
    → 不要なキーがないか、有効期限が近いものがないか
□ LLM モデルの評価
    → 新しいモデルへの更新を検討
□ コスト分析レポートの作成
□ GPU 性能テスト（劣化がないか確認）
    → nvidia-smi -q | grep "Temperature\|Power\|ECC"
□ UPS バッテリーテスト（負荷テスト）
```

## 13. ハードウェア保守

### 13.1 日常的な HW 確認

```bash
# GPU の状態確認
nvidia-smi

# 確認項目：
#   温度：80℃ 以下（アイドル時は 30〜50℃）
#   電力：TDP 以内（A100 → 250W 以下）
#   ECC エラー：0 であること
#   Fan：N/A（サーバー管理）または 0〜100%

# ディスクの健康状態（SMART）
sudo apt install -y smartmontools
sudo smartctl -a /dev/nvme0n1
# → SMART overall-health: PASSED であること

# メモリの ECC エラー確認
sudo edac-util -s
# → No errors であること
```

### 13.2 HW 障害対応

| 障害 | 症状 | 対応 |
|------|------|------|
| **GPU 温度過熱** | 温度 80℃ 超え | エアフロー確認 / 電力制限 / ファン清掃 |
| **GPU ECC エラー** | nvidia-smi にエラー表示 | 軽度：経過観察 / 重度：GPU 交換 |
| **ディスク SMART 異常** | SMART 警告 | 即バックアップ → ディスク交換 |
| **メモリ ECC エラー** | edac-util にエラー | メモリモジュール交換 |
| **電源ユニット故障** | 冗長電源の片系故障 | PSU 交換（冗長なので稼働は継続可） |
| **UPS バッテリー劣化** | バッテリー保持時間低下 | バッテリー交換 |

### 13.3 清掃（半年ごと推奨）

```
□ サーバーの電源を停止（UPS も OFF）
□ エアダスターで内部のホコリを除去
  → 特に CPU ヒートシンク・GPU ファン周り
□ ラックのフィルター（ある場合）を清掃
□ ケーブル類の緩みがないか確認
□ 電源を再投入し、正常起動を確認
```

## 16. トラブルシューティング集

### Q: PDCA サイクルが停止し、再開できない

```
1. 停止理由を確認：
   python -m vibe_pdca.cli status --project-id <PROJECT_ID>
2. 監査ログで原因を調査：
   python -m vibe_pdca.cli audit-log --last 20
3. 一般的な原因と対処：
   - コスト上限超過 → Owner がコスト上限を引き上げ後に再開
   - P0/P1 → 4/4 承認後に再開
   - LLM 全プロバイダ障害 → LLM 復旧後に再開
```

### Q: Discord 通知が来ない

```
1. Bot がサーバーに参加しているか確認
2. Bot Token が正しいか確認（.env）
3. チャンネルの権限設定を確認
4. Discord API の障害を確認（https://discordstatus.com/）
```

### Q: GPU の性能が低下している

```
1. GPU 温度を確認（80℃ 超えはサーマルスロットリング発生）
2. GPU 使用率を確認（他のプロセスが GPU を使っていないか）
3. vLLM の設定を確認（--gpu-memory-utilization）
4. nvidia-smi で ECC エラーを確認
```

### Q: ディスク容量が不足している

```
1. 何がディスクを圧迫しているか確認：
   du -sh /home/vibe-admin/* | sort -hr | head -20
2. Docker 不要リソースの削除：
   docker system prune -a --volumes
3. 古いログの削除：
   sudo journalctl --vacuum-time=30d
4. HuggingFace キャッシュの確認：
   du -sh ~/.cache/huggingface/
```

### Q: サーバーが突然シャットダウンした

```
1. UPS のステータスを確認（停電発生の有無）
2. iDRAC / iLO のログを確認（HW エラーの有無）
3. /var/log/kern.log を確認（カーネルパニックの有無）
4. GPU 温度が過熱していた可能性（nvidia-smi のログ確認）
5. 電源容量が不足している可能性（PSU の W 数を確認）
```

## 18. 用語集

| 用語 | 説明 |
|------|------|
| **Graceful Stop** | 現在の処理が完了してから停止する方法 |
| **Emergency Stop** | 即座に停止する方法（処理は中断される） |
| **A 操作** | 権限拡大等の重要操作。全ロールの承認が必要 |
| **B 操作** | 中程度の操作。バックアップ後に自動進行 |
| **C 操作** | 軽微な操作。承認不要 |
| **P0 / P1 / P2** | インシデントの優先度分類 |
| **縮退モード** | 機能を制限して運用を継続するモード |
| **フォールバック** | 障害時に代替手段に切り替えること |
| **サーキットブレーカー** | 連続失敗を検知して自動的にフォールバックする仕組み |
| **Suppress List** | 既知の誤検知パターンを除外するリスト |
| **RBAC** | Role-Based Access Control。役割に基づくアクセス制御 |
| **OpenClaw** | 本システムのハードウェア監視層（第 1 層） |
| **ECC** | Error Correcting Code。メモリのビット化けを自動修正する機能 |
| **SMART** | Self-Monitoring, Analysis and Reporting Technology。ディスクの健康状態を監視する技術 |
| **UPS** | 無停電電源装置 |
| **Prometheus** | メトリクス収集・監視ツール |
| **Grafana** | メトリクスの可視化ダッシュボードツール |
