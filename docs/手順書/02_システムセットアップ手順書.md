## 3. Phase 1：Docker（rootless モード）のインストール

> **Docker とは？**  
> プログラムを隔離された安全な空間（「コンテナ」と呼びます）で実行するための仕組みです。  
> たとえるなら、アプリを動かす「小さな仮想パソコン」を作るツールです。  
> 💡 詳しくは → [AI推論サーバー説明書 §7](../説明書/AI推論サーバー説明書.md#7-ソフトウェアスタック構成)

> **rootless Docker とは？**  
> 通常の Docker は root 権限（管理者権限）で動作しますが、rootless モードは一般ユーザー権限で動作します。  
> セキュリティ設計（ADR-007）により、本システムでは rootless モードが必須です。

### 3.1 Docker のインストール

```bash
# Docker をインストール
curl -fsSL https://get.docker.com -o get-docker.sh
sh get-docker.sh

# rootless モードをセットアップ
dockerd-rootless-setuptool.sh install

# 環境変数の設定（rootless Docker 用）
echo 'export PATH=$HOME/bin:$PATH' >> ~/.bashrc
echo 'export DOCKER_HOST=unix:///run/user/$(id -u)/docker.sock' >> ~/.bashrc
source ~/.bashrc
```

### 3.2 NVIDIA Container Toolkit のインストール

Docker コンテナ内から GPU を使えるようにします。

```bash
# NVIDIA Container Toolkit のインストール
distribution=$(. /etc/os-release;echo $ID$VERSION_ID)
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
curl -s -L https://nvidia.github.io/libnvidia-container/$distribution/libnvidia-container.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

sudo apt update
sudo apt install -y nvidia-container-toolkit

# Docker のランタイム設定（rootless 対応）
nvidia-ctk runtime configure --runtime=docker --config=$HOME/.config/docker/daemon.json
systemctl --user restart docker
```

### 3.3 動作確認

```bash
# Docker が rootless で動作しているか確認
docker info --format '{{.SecurityOptions}}'
# → 出力に "name=rootless" が含まれていること

# GPU が Docker コンテナ内から見えるか確認
docker run --rm --gpus all nvidia/cuda:12.4.0-base-ubuntu22.04 nvidia-smi
# → コンテナ内で nvidia-smi が表示されれば成功
```

### ✅ 確認ポイント

- [ ] `docker info` に "rootless" が表示される
- [ ] Docker コンテナ内から `nvidia-smi` で GPU が見える
- [ ] `docker ps` がエラーなく実行できる

## 4. Phase 2：ローカル LLM（vLLM）のセットアップ

> **vLLM とは？**  
> 高性能な LLM 推論サーバーです。OpenAI 互換の API（通信窓口）を提供するため、クラウド API と同じコードでアクセスできます。  
> たとえるなら、AI の「エンジン」にあたるソフトウェアです。  
> 💡 詳しくは → [AI推論サーバー説明書 §7](../説明書/AI推論サーバー説明書.md#7-ソフトウェアスタック構成)

### 4.1 vLLM のインストール

```bash
# vLLM 専用の Python 仮想環境を作成
python3.12 -m venv ~/vllm-env
source ~/vllm-env/bin/activate

# vLLM のインストール
pip install vllm
```

### 4.2 モデルのダウンロードと起動

#### 入門構成（A30 × 1、24GB VRAM）の場合

```bash
source ~/vllm-env/bin/activate

python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen2.5-32B-Instruct-GPTQ-Int4 \
    --gpu-memory-utilization 0.90 \
    --dtype auto \
    --max-model-len 16384 \
    --served-model-name "local-llm" \
    --port 8000
```

#### 推奨構成（A100 40GB × 2）の場合

```bash
source ~/vllm-env/bin/activate

python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen2.5-72B-Instruct \
    --tensor-parallel-size 2 \
    --gpu-memory-utilization 0.90 \
    --dtype bfloat16 \
    --max-model-len 32768 \
    --served-model-name "local-llm" \
    --port 8000
```

> **💡 初回起動時のモデルダウンロード**  
> 初回はモデルのダウンロードに 30 分〜2 時間かかります（モデルサイズにより異なります）。  
> ダウンロード後は `~/.cache/huggingface/` にキャッシュされるため、2 回目以降は高速に起動します。

### 4.3 vLLM の systemd サービス化（自動起動設定）

サーバー再起動時に vLLM が自動で立ち上がるようにします。

```bash
# サービスファイルを作成
mkdir -p ~/.config/systemd/user/

cat > ~/.config/systemd/user/vllm.service << 'EOF'
[Unit]
Description=vLLM Local LLM Server
After=network.target

[Service]
Type=simple
WorkingDirectory=%h
ExecStart=%h/vllm-env/bin/python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen2.5-72B-Instruct \
    --tensor-parallel-size 2 \
    --gpu-memory-utilization 0.90 \
    --dtype bfloat16 \
    --max-model-len 32768 \
    --served-model-name "local-llm" \
    --port 8000
Restart=on-failure
RestartSec=30
Environment=CUDA_VISIBLE_DEVICES=0,1

[Install]
WantedBy=default.target
EOF

# 入門構成（A30 × 1）の場合は上記の ExecStart を書き換えてください

# サービスの有効化・起動
systemctl --user enable vllm
systemctl --user start vllm

# ユーザーサービスがログアウト後も動くように設定
loginctl enable-linger $USER
```

### 4.4 動作確認

```bash
# vLLM のヘルスチェック
curl http://localhost:8000/health
# → {"status":"ok"} が返ればOK

# 推論テスト
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "local-llm",
    "messages": [{"role": "user", "content": "Hello, world!"}],
    "max_tokens": 100
  }'
# → AI の応答が JSON で返ってくれば成功
```

### ✅ 確認ポイント

- [ ] vLLM が起動し、ポート 8000 でリクエストを受け付ける
- [ ] `/health` が `ok` を返す
- [ ] 推論テストで応答が返る
- [ ] systemd サービスが有効化されている（`systemctl --user status vllm`）

## 7. Phase 5：GitHub App の作成と設定

> 💡 **GitHub App とは？**  
> GitHubの機能を外部のシステムと連携させるための仕組みです。本システムがGitHub上でIssue（やることリスト）やPR（変更依頼）を自動作成するために必要です。  
> 💡 詳しくは → [オンボーディング説明書 §2](../説明書/オンボーディング説明書.md#2-システム概要)

### 7.1 GitHub App の作成

```
Step 1：GitHub（https://github.com）にログイン
Step 2：右上のアイコン → Settings → Developer settings → GitHub Apps → New GitHub App

設定項目：
  App name：          VibePDCA-Bot（任意）
  Homepage URL：      https://github.com/maki04591128-tech/Vive-coding-pdca-system
  Webhook：           Active のチェックを外す（後で設定可）

権限設定（Repository permissions）：
  Contents：          Read & Write
  Issues：            Read & Write
  Pull requests：     Read & Write
  Metadata：          Read-only

権限設定（Organization permissions）：
  Members：           Read-only（任意）

Where can this GitHub App be installed？：
  → 「Only on this account」を選択

Step 3：「Create GitHub App」をクリック
Step 4：App ID をメモ → .env の GITHUB_APP_ID に設定
Step 5：「Generate a private key」をクリック
        → ダウンロードされた .pem ファイルをサーバーに転送
        → .env の GITHUB_APP_PRIVATE_KEY_PATH にパスを設定
```

### 7.2 GitHub App のインストール

```
Step 1：作成した GitHub App のページ → 「Install App」
Step 2：対象の Organization またはアカウントを選択
Step 3：「Only select repositories」→ 対象リポジトリを選択
        → ⚠️ 「All repositories」は選択しないこと（セキュリティ要件）
Step 4：「Install」をクリック
```

### ✅ 確認ポイント

- [ ] GitHub App が作成されている
- [ ] App ID と Private Key が `.env` に設定されている
- [ ] 対象リポジトリに App がインストールされている
- [ ] 権限が Contents / Issues / Pull requests のみ

├── 📁 ハードウェア関連
│   ├── 🔴 hw-critical            ← GPU・ディスク・電源の重大障害
│   ├── 🟡 hw-warning             ← ハードウェア警告
│   ├── 🟢 hw-info                ← 解消通知・定期チェック結果

    ├── #general                  ← 一般会話

## 10. Phase 8：OpenClaw（ハードウェア監視層）の導入

> **OpenClaw とは？**  
> 三層独立運用体制の「第1層：ハードウェア監視層」です。  
> GPU 温度・ディスク容量・電源状態などを監視し、異常があれば Discord に通知します。  
> PDCAシステムとは完全に独立して動作します。

### 10.1 OpenClaw の基本セットアップ

```bash
# 監視スクリプトの配置
sudo mkdir -p /opt/openclaw
sudo cp ~/Vive-coding-pdca-system/config/openclaw/* /opt/openclaw/ 2>/dev/null || true

# GPU 温度監視スクリプトの作成
cat > /opt/openclaw/gpu_monitor.sh << 'SCRIPT'
#!/bin/bash
# OpenClaw GPU 温度監視（5分間隔で実行）

THRESHOLD=80  # 警告閾値（℃）
DISCORD_WEBHOOK="${OPENCLAW_DISCORD_WEBHOOK}"

gpu_temp=$(nvidia-smi --query-gpu=temperature.gpu --format=csv,noheader,nounits | head -1)

if [ "$gpu_temp" -gt "$THRESHOLD" ]; then
    curl -H "Content-Type: application/json" \
         -d "{\"content\": \"⚠️ **GPU温度警告**: ${gpu_temp}℃ （閾値: ${THRESHOLD}℃）\"}" \
         "$DISCORD_WEBHOOK"
fi
SCRIPT

chmod +x /opt/openclaw/gpu_monitor.sh

# ディスク容量監視スクリプトの作成
cat > /opt/openclaw/disk_monitor.sh << 'SCRIPT'
#!/bin/bash
# OpenClaw ディスク容量監視

THRESHOLD=85  # 使用率警告閾値（%）
DISCORD_WEBHOOK="${OPENCLAW_DISCORD_WEBHOOK}"

usage=$(df / --output=pcent | tail -1 | tr -d ' %')

if [ "$usage" -gt "$THRESHOLD" ]; then
    curl -H "Content-Type: application/json" \
         -d "{\"content\": \"⚠️ **ディスク容量警告**: 使用率 ${usage}% （閾値: ${THRESHOLD}%）\"}" \
         "$DISCORD_WEBHOOK"
fi
SCRIPT

chmod +x /opt/openclaw/disk_monitor.sh
```

### 10.2 cron による定期実行の設定

```bash
# crontab を編集
crontab -e

# 以下を追加（5分間隔で GPU 監視、1時間間隔でディスク監視）
*/5 * * * * OPENCLAW_DISCORD_WEBHOOK="https://discord.com/api/webhooks/xxxxx/xxxxx" /opt/openclaw/gpu_monitor.sh
0 * * * *   OPENCLAW_DISCORD_WEBHOOK="https://discord.com/api/webhooks/xxxxx/xxxxx" /opt/openclaw/disk_monitor.sh
```

> **💡 Discord Webhook URL の取得方法**  
> Discord サーバー → `hw-critical` チャンネル → 設定（歯車アイコン）→ 連携サービス → ウェブフック → 新しいウェブフック

### ✅ 確認ポイント

- [ ] GPU 監視スクリプトが動作する（手動実行テスト）
- [ ] ディスク監視スクリプトが動作する
- [ ] cron が設定されている（`crontab -l` で確認）
- [ ] Discord Webhook URL が正しく設定されている

## 11. Phase 9：監視基盤（Prometheus + Grafana）の構築

### 11.1 Prometheus のインストール

```bash
# Prometheus のダウンロード
cd /tmp
wget https://github.com/prometheus/prometheus/releases/download/v2.50.0/prometheus-2.50.0.linux-amd64.tar.gz
tar xzf prometheus-2.50.0.linux-amd64.tar.gz
sudo mv prometheus-2.50.0.linux-amd64 /opt/prometheus

# 設定ファイルの作成
cat > /opt/prometheus/prometheus.yml << 'EOF'
global:
  scrape_interval: 15s

scrape_configs:
  - job_name: 'node'
    static_configs:
      - targets: ['localhost:9100']

  - job_name: 'nvidia-gpu'
    static_configs:
      - targets: ['localhost:9400']
EOF

# systemd サービスの作成
sudo cat > /etc/systemd/system/prometheus.service << 'EOF'
[Unit]
Description=Prometheus
After=network.target

[Service]
Type=simple
User=vibe-admin
ExecStart=/opt/prometheus/prometheus --config.file=/opt/prometheus/prometheus.yml --storage.tsdb.path=/opt/prometheus/data
Restart=on-failure

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable prometheus
sudo systemctl start prometheus
```

### 11.3 Grafana のインストール

## 16. トラブルシューティング

### Q: Docker で GPU が認識されない

```
対処：
  1. nvidia-container-toolkit が正しくインストールされているか確認
  2. Docker デーモンを再起動：systemctl --user restart docker
  3. nvidia-smi が動くか確認（ホスト側で）
```

### Q: vLLM が起動しない / Out of Memory

```
対処：
  1. GPU の VRAM が不足していないか確認（nvidia-smi）
  2. --gpu-memory-utilization を 0.85 に下げる
  3. --max-model-len を短くする（例: 16384 → 8192）
  4. より小さいモデルを使用する
```

### Q: GitHub App で権限エラーが出る

```
対処：
  1. App ID と Private Key が正しいか確認
  2. 対象リポジトリに App がインストールされているか確認
  3. 権限（Contents / Issues / Pull requests）が設定されているか確認
```

### Q: Discord Bot が応答しない

```
対処：
  1. Bot Token が正しいか確認
  2. Bot がサーバーに参加しているか確認
  3. MESSAGE CONTENT INTENT が ON か確認
  4. Bot に必要な権限（メッセージ送信等）があるか確認
```

### Q: テストが失敗する

```
対処：
  1. 依存関係が最新か確認：pip install -e ".[dev]" --upgrade
  2. .env ファイルが存在し、必要な値が設定されているか確認
  3. 特定のテストのみ失敗する場合：pytest tests/test_xxx.py -v で詳細確認
```
