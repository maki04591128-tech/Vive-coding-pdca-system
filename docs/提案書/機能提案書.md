## 📋 提案概要

現在のシステム設計は非常に包括的であり、PDCAサイクルの自動化・ガバナンス・監査・コスト管理・マルチプロバイダLLM対応など、主要機能は網羅されています。以下は、現在の設計に**含まれていない**、または**強化が望ましい**機能の提案です。

## 🔴 優先度：高（運用品質に直結）

### 提案 1: Webhook / イベント駆動アーキテクチャの導入

**現状の課題**:
現在のシステムはCLI/GUI起動による同期的な処理フローが中心です。GitHub上のイベント（Issue作成、PR更新、CIステータス変更等）に対するリアルティムの反応機構が設計に含まれていません。

**提案内容**:
- GitHub Webhookの受信サーバー（FastAPIベース）を追加し、以下のイベントでPDCAサイクルを自動トリガーする仕組みを構築
  - `issues.opened` → 新規Goal検出 → PLANフェーズ自動開始
  - `pull_request.review_submitted` → CHECKフェーズへの外部レビュー取り込み
  - `check_suite.completed` → CI結果のリアルタイム反映
  - `issue_comment.created` → 人間からの介入指示の自動検出
- イベントキューイング（Redis/インメモリキュー）によるバックプレッシャー制御

**期待効果**: 手動トリガーの削減、リアルタイム性の向上、GitHubネイティブなワークフロー統合

---

### 提案 2: セッション永続化とクラッシュリカバリ機構

**現状の課題**:
PDCAステートマシンはインメモリで動作しており、プロセス再起動時にサイクルの途中状態が失われる可能性があります。GitHub Issueをステートストアとして使う設計は記載されていますが、ローカル状態の定期的なチェックポイント保存やクラッシュリカバリの具体的な実装方針が不明確です。

**提案内容**:
- PDCAステートの定期チェックポイント保存（JSON/YAML形式、設定可能な間隔）
- プロセス起動時の自動リカバリ：最新チェックポイントからのサイクル再開
- 「ダーティシャットダウン」検出フラグによる不正終了時の自動通知
- チェックポイントデータの整合性検証（既存のchain-hashメカニズムの活用）

**期待効果**: 運用安定性の大幅向上、長時間サイクル実行中の障害耐性

---

### 提案 3: レート制限・スロットリングの統合管理

**現状の課題**:
コスト管理（$30/日、80コール/サイクル）は実装されていますが、各LLMプロバイダ固有のAPIレート制限（RPM/TPM）への対応が設計に含まれていません。GitHub API のレート制限（5000リクエスト/時間、Search API 30リクエスト/分）への対応も不十分です。

**提案内容**:
- プロバイダごとのレート制限トラッカー（トークンバケットアルゴリズム）
- レート制限到達時の自動待機（exponential backoff）とフォールバック
- GitHub API呼び出しの集約とキャッシュ（特にRAG検索用）
- レート制限状況のダッシュボード表示

**期待効果**: API制限によるサイクル中断の防止、安定した長時間運用

---

### 提案 4: テスト戦略の拡充 — E2Eテストとシナリオテスト

**現状の課題**:
単体テスト（28ファイル）は充実していますが、PDCAサイクル全体を通したEnd-to-Endテスト、およびシナリオベースのテスト（異常系・境界値・マルチサイクル）が不足しています。

**提案内容**:
- **E2Eシナリオテスト**: Goal入力→Milestone生成→Task実行→Review→Decision の完全サイクルをモックLLMで自動テスト
- **異常系シナリオ**: LLM全プロバイダダウン、GitHub API障害、コスト上限到達、6時間スタック等のシミュレーション
- **マルチサイクルテスト**: 10サイクル連続実行による学習フィードバック・劣化検出の検証
- **GUI統合テスト**: Fletコンポーネントの状態遷移テスト（現在テストなし）
- テスト用のLLMモックサーバー（固定レスポンスの提供）

**期待効果**: リグレッション防止、リリース品質の保証

### 提案 5: ダッシュボードの高度化 — 可視化とインタラクション

**現状の課題**:
現在のGUI（Flet）はステータス表示とログビューアが中心で、PDCAサイクルの進行状況やコスト推移を視覚的に把握することが困難です。

**提案内容**:

- **アラート通知パネル**: コスト警告、ストップ条件接近、ヘルスチェック異常をリアルタイム表示

**期待効果**: 意思決定の迅速化、システム状態の直感的把握

### 提案 6: プロンプトバージョニングとA/Bテスト機能

**現状の課題**:
プロンプトテンプレートのバージョン管理は設計に含まれていますが、複数バージョンのプロンプトを同時に運用して効果を比較するA/Bテスト機能がありません。学習フィードバック（10サイクルごと）でプロンプトを改善する仕組みはありますが、改善の効果測定が定量的に行えません。

**提案内容**:
- プロンプトテンプレートの複数バージョン同時運用
- サイクル単位でのランダム割り当て（A/Bスプリット）
- 成功率・レビュー品質・コストなどのKPIによるバージョン比較レポート
- 統計的有意差検定による自動バージョン選択

**期待効果**: プロンプト品質の継続的・定量的な改善

- **通知ルーティングエンジン**: イベント種別×重要度でチャネルを自動選択
- プラグインシステム（既存の`PluginCategory.NOTIFICATION`）を活用した拡張可能な設計

**期待効果**: 多様なチーム環境への対応、重要通知の見逃し防止

### 提案 15: セキュリティ強化 — 監査ログの外部転送と改ざん検知

**現状の課題**:
監査ログはchain-hashによる改ざん検知を備えていますが、ログ自体がローカルファイルに保存されるため、ホストOSレベルの改ざんには脆弱です。また、ADRにて「監査ログの単一障害点」がリスクとして認識されています。

**提案内容**:
- **外部ログ転送**: 監査ログを外部サービス（CloudWatch Logs、Datadog、Elasticsearch等）にリアルタイム転送
- **イミュータブルストレージ**: S3 Object Lock / Azure Immutable Blob による改ざん不可能なバックアップ
- **ログ署名**: デジタル署名による出所証明
- **定期的な整合性監査**: ローカルログとリモートログの自動照合
- **SIEM連携**: セキュリティイベントの自動相関分析

**期待効果**: コンプライアンス要件への対応、監査ログの信頼性向上

### 提案 16: ロールバック戦略の体系化とワンクリックロールバック

**現状の課題**:
`intervention.py`にロールバック機能の基盤はありますが、ロールバックの粒度（タスク単位/サイクル単位/マイルストーン単位）、ロールバック後のステート整合性保証、ロールバック履歴の追跡が体系化されていません。ACTフェーズで`reject`判定された場合のリカバリフローも、具体的な自動化手順が不明確です。

**提案内容**:
- **多粒度ロールバック**: タスク単位（PRのrevert）、サイクル単位（ステートの巻き戻し）、マイルストーン単位（複数サイクルの一括取消）の3レベルを定義
- **ロールバックプレビュー**: 実行前に影響範囲（対象PR、変更ファイル、依存タスク）を一覧表示
- **ステート整合性チェッカー**: ロールバック後のPDCAステートマシン・監査ログ・GitHub Issue の整合性を自動検証
- **ロールバック連鎖の検出**: あるタスクのロールバックが他のタスクに波及する場合の自動警告
- **GUIワンクリックロールバック**: ダッシュボードから操作可能なロールバックUI

**期待効果**: 障害復旧時間の短縮、ロールバック操作のヒューマンエラー防止

### 提案 17: 設定変更のホットリロードとバージョン管理

**現状の課題**:
`config/loader.py`による階層的な設定読み込みは実装されていますが、設定変更時にシステム再起動が必要です。また、設定変更の履歴追跡、変更前後の差分表示、設定変更のロールバック機能がありません。運用中のコスト上限やLLMプロバイダの切替を、サイクル中断なく行う必要があります。

**提案内容**:
- **ホットリロード**: ファイルシステム監視（`watchdog`）またはシグナル（`SIGHUP`）による設定の動的再読み込み
- **設定バージョニング**: 変更ごとにバージョン番号を付与し、監査ログに記録
- **差分表示**: 変更前後の設定値をdiff形式で表示
- **段階的適用**: 次のサイクル開始時に新設定を適用する「遅延適用モード」
- **バリデーション**: 新設定の整合性チェック（コスト上限の最小値、プロバイダの有効性等）を適用前に実行
- **ロールバック**: 過去の設定バージョンへの即座の復帰

**期待効果**: 運用中のダウンタイムゼロでの設定調整、設定変更の監査追跡

---

### 提案 18: PDCAサイクルのタイムアウト戦略の精緻化

**現状の課題**:
現在、6時間の一律タイムアウトが設定されていますが（`stop_conditions.py`）、タスクの複雑度やフェーズごとの所要時間は大きく異なります。シンプルなバグ修正は30分で完了しうる一方、大規模リファクタリングでは6時間でも不足する可能性があります。

**提案内容**:
- **フェーズ別タイムアウト**: PLAN（30分）、DO（3時間）、CHECK（1時間）、ACT（30分）等、フェーズごとに個別設定
- **タスク複雑度ベースの動的タイムアウト**: PLANフェーズで推定される複雑度に基づいてタイムアウト値を自動調整
- **プログレスベースの延長**: 進捗が確認できる場合（例：CIジョブが実行中）は自動延長
- **段階的エスカレーション**: タイムアウト50%到達で警告通知、75%で介入要求、100%でストップ
- **タイムアウト統計**: 過去のサイクルの所要時間データに基づくタイムアウト値の自動最適化

**期待効果**: 不必要な早期中断の防止、長時間スタックの早期検出

### 提案 19: サンドボックスのリソース制限と監視の強化

**現状の課題**:
Docker rootlessコンテナによるサンドボックス（ADR-007）は設計されていますが、コンテナ内のリソース消費（CPU、メモリ、ディスクI/O）の上限設定と監視、および悪意のあるコードによるリソース枯渇攻撃への対策が具体化されていません。

**提案内容**:
- **リソース制限の明示的設定**: `--memory`, `--cpus`, `--pids-limit`, `--storage-opt` によるコンテナリソース上限
- **リアルタイムリソース監視**: コンテナ実行中のCPU/メモリ/ディスク使用率をメトリクスとして収集
- **リソース超過時の自動終了**: OOMKillの検出とグレースフルなエラーハンドリング
- **実行時間制限**: コンテナ単位のタイムアウト（`--stop-timeout`）
- **ネットワーク帯域制限**: 大量のパッケージダウンロードによる帯域枯渇の防止
- **ファイルシステムサイズ制限**: 一時ファイルの無制限生成の防止

**期待効果**: DOフェーズの安全性向上、ホストシステムの保護

### 提案 29: PDCAサイクルのリプレイとデバッグモード

**現状の課題**:

**提案内容**:
- **サイクルスナップショット**: 各フェーズの入出力（LLMプロンプト、レスポンス、決定、CI結果等）を完全に記録

- **ステップ実行**: フェーズごとに一時停止し、中間状態を確認できるデバッグモード
- **What-If分析**: 特定のLLMレスポンスやCI結果を差し替えて「もしこうだったら」のシミュレーション
- **比較ビュー**: 2つのサイクルの入出力を並べて比較し、差分を特定

**期待効果**: 問題の根本原因分析の効率化、システムの透明性向上

- **エッジケースレポート**: 発見された問題をCHECKフェーズのレビュー結果に自動統合

**期待効果**: 生成コードの堅牢性向上、見落とされやすいバグの早期発見

### 実装内容サマリ

#### 提案16: ロールバック戦略の体系化
- `RollbackLevel` 列挙型（TASK / CYCLE / MILESTONE の3粒度）
- `RollbackPreview` — 実行前の影響範囲プレビュー（対象PR、変更ファイル、依存タスク）
- `StateConsistencyChecker` — ロールバック後のPDCAステート整合性検証
- `RollbackChainDetector` — ロールバック連鎖の自動検出・警告

#### 提案17: 設定ホットリロード・バージョン管理
- `ConfigVersion` / `ConfigVersionStore` — 設定のバージョン管理と履歴保持
- `ConfigDiff` — 変更前後の差分表示（追加/削除/変更キーの可視化）
- `ConfigValidator` — 設定適用前のバリデーション（必須フィールド/数値範囲/リスト検証）
- `HotReloadManager` — ファイルmtime監視、即時/遅延適用モード、コールバック通知

#### 提案18: タイムアウト戦略の精緻化
- `PhaseTimeout` — フェーズ別タイムアウト（PLAN:30分、DO:3時間、CHECK:1時間、ACT:30分）
- `TimeoutEscalation` — 段階的エスカレーション（50%:警告、75%:介入要求、100%:停止）
- `ComplexityBasedTimeout` — タスク複雑度に基づく動的タイムアウト調整
- `ProgressBasedExtension` — 進捗検出時の自動延長（最大回数制限付き）
- `TimeoutManager` — 統合管理・タイムアウト統計の追跡

#### 提案19: サンドボックスリソース制限・監視
- `ResourceLimit` / `ResourceUsage` — コンテナリソース制限と使用量のモデル
- `DockerResourceConfig` — Docker run引数 / Docker Compose設定の自動生成
- `ResourceMonitor` — リアルタイムリソース監視（80%警告/95%クリティカル閾値）
- `OOMHandler` — OOMKill検出（exit code 137）と診断レポート生成

#### 提案20: GitHub Appトークン管理強化
- `TokenRotationManager` — トークン有効期限監視と自動ローテーション（5分前バッファ）
- `PhaseScope` / `ScopeManager` — PDCAフェーズ別の最小権限スコープ管理
- `TokenLeakDetector` — GitHub トークンパターンの正規表現検出（ghp_/ghs_/github_pat_等）
- `AccessLogger` — GitHub API アクセスログの記録と統計

#### 提案21: タスク依存関係グラフ・クリティカルパス分析
- `TaskNode` / `DependencyGraph` — タスク間の有向非巡回グラフ（DAG）管理
- トポロジカルソートによる実行順序決定・循環依存検出
- `CriticalPathAnalyzer` — 最長パス分析・サイクル完了時間予測
- `BlockerDetector` — ボトルネックタスクの早期警告（閾値ベース）
- `get_parallel_groups()` — 並列実行可能なタスクグループの自動特定

#### 提案22: LLMレスポンス品質スコアリング
- `QualityDimension` — 4次元評価（構造的妥当性/完全性/一貫性/ハルシネーション）
- `StructuralValidator` / `CompletenessChecker` / `HallucinationDetector` — 個別バリデータ
- `QualityAwareRetrier` — 品質スコア閾値未達時の自動リトライ（最大3回、エラーフィードバック付き）
- `ModelQualityTracker` — モデル別/ロール別の品質スコア推移追跡

#### 提案23: 変更影響分析の自動化
- `StaticDependencyAnalyzer` — Python import文の解析・依存関係マップ構築
- `BreakingChangeDetector` — パブリックAPI変更の検出（関数/クラス定義の差分）
- `TestTargetFinder` — 変更に関連するテストファイルの自動特定（命名規則ベース）
- `ImpactAnalyzer` — 統合オーケストレータ・影響スコア（0.0〜1.0）算出
- Markdown形式のインパクトレポート自動生成

#### 提案24: ユーザーフィードバック収集・満足度トラッキング
- `FeedbackCollector` — サイクル完了後のフィードバック収集（1-5点、カテゴリ別）
