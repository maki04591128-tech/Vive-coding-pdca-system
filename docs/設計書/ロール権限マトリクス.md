## 📖 この文書について

> この文書は、バイブコーディングPDCAシステムを使う**人間の役割（ロール）**ごとに「誰が何をできるか」を一覧で示したものです。  
> たとえるなら、会社の「権限表」のようなものです。社長だけが決裁できること、部長でもできること、一般社員は閲覧だけ…といった区分を明確にしています。

### 🔰 この文書で使われる主要な用語

| 用語 | かんたんな説明 |
|------|-------------|
| **ロール** | 人間に割り当てられる「役割」のこと。権限の範囲が異なります |
| **Owner（オーナー）** | プロジェクトの最終責任者。すべての操作が可能です |
| **Maintainer（メンテナー）** | 日常の運用管理を担当する人。停止・再開などの操作ができます |
| **Reviewer（レビューアー）** | 成果物の品質を確認する人。閲覧とコメントが主な役割です |
| **Auditor（オーディター）** | 監査担当者。操作記録の閲覧・出力ができます |
| **A操作** | 影響が大きく、必ず人間全員の承認が必要な操作（例：権限の拡大） |
| **B操作** | 中程度の影響。条件付きで自動実行される操作（例：大きなコード変更） |
| **C操作** | 影響が小さく、自動で実行される操作（例：軽微な修正） |
| **4/4承認** | 4つのロール区分すべてから承認を得ること。全員一致が必要です |
| **ポリシー** | システムの動作ルール。上位のルールは下位で緩和できません |
| **運転モード** | AIの自動化レベル。手動承認／半自動／全自動の3段階があります |

## 1. ロール定義

> 💡 **かんたんに言うと**：Owner（社長）> Maintainer（管理者）> Reviewer（確認者）> Auditor（監査役）の順に権限が広いです。

| ロール | 責務 |
|--------|------|
| **Owner** | 最終到達点の提示 / 重要な意思決定（A操作の承認・停止解除等） / モデル切替判断 |
| **Maintainer** | 日常運用（停止 / 再開 / モード切替） / ポリシー変更（制限付き） |
| **Reviewer** | レビュー閲覧・コメント |
| **Auditor** | 監査ログ閲覧・エクスポート |

## 2. 操作分類別の権限マトリクス

> 💡 **操作の分類**：影響の大きさで A（重大）> B（中程度）> C（軽微）に分かれています。A操作は銀行の「大口送金」のようなもので、全員の承認印が必要です。

### 2.1 A操作（必ず人間承認）

A操作はDiscord上で4/4承認（全ロール区分から各1名以上）が必要。

| 操作 | Owner | Maintainer | Reviewer | Auditor |
|------|:-----:|:----------:|:--------:|:-------:|
| 権限拡大 | ✅ 承認 | ✅ 承認 | ✅ 承認 | ✅ 承認 |
| ポリシー緩和 | ✅ 承認 | ✅ 承認 | ✅ 承認 | ✅ 承認 |
| 禁止領域の変更 | ✅ 承認 | ✅ 承認 | ✅ 承認 | ✅ 承認 |
| 例外マージ | ✅ 承認 | ✅ 承認 | ✅ 承認 | ✅ 承認 |
| バイナリ例外追加 | ✅ 承認 | ✅ 承認 | ✅ 承認 | ✅ 承認 |
| ログ削除例外 | ✅ 承認 | ✅ 承認 | ✅ 承認 | ✅ 承認 |
| ホワイトリスト変更 | ✅ 承認 | ✅ 承認 | ✅ 承認 | ✅ 承認 |
| Suppress List登録 | ✅ 承認 | ✅ 承認 | ✅ 承認 | ✅ 承認 |
| Ownerが指定した操作 | ✅ 承認 | ✅ 承認 | ✅ 承認 | ✅ 承認 |

### 2.2 B操作（条件付き自動）

B操作はバックアップ作成後に自動進行。CI設定変更のみペルソナ3承認を追加要求。

| 操作 | Owner | Maintainer | Reviewer | Auditor |
|------|:-----:|:----------:|:--------:|:-------:|
| diff閾値超え変更 | 通知 | 通知 | ― | ― |
| 依存関係の大規模更新 | 通知 | 通知 | ― | ― |
| CI設定変更 | 通知 | 通知 | ― | ― |

### 2.3 C操作（自動）

| 操作 | Owner | Maintainer | Reviewer | Auditor |
|------|:-----:|:----------:|:--------:|:-------:|
| 既定ポリシー内の軽微変更 | ― | ― | ― | ― |

## 4. プロジェクト種別固有のA操作権限

プロジェクト種別固有の追加A操作（§17.6）は、共通A操作と同じく全ロール4/4承認を要する。種別ごとの具体的な追加A操作リストはADR-002を参照。

## 5. Discord承認の要件

> 💡 **たとえるなら**：Discordでの承認は「稟議書の回覧」に似ています。全員がハンコ（承認）を押さないと実行されません。12時間以内に集まらなければ保留になります。

| 項目 | 仕様 |
|------|------|
| A操作の必要承認数 | 4/4（全ロール区分から各1名以上） |
| B操作（CI設定変更）のペルソナ承認 | PM + 書記 + 他1名 = 3承認 |
| タイムアウト | 12時間（期限内に全承認が揃わない場合は保留） |
| NOの場合 | 理由必須、別案提示、ACTで記録 |
| HOLDの場合 | 条件・不足情報を明示、条件未達なら保留継続 |
| 通知失敗の場合 | 保留（承認待ちとして継続しない） |

## 6. ポリシー階層と権限

| ポリシー階層 | 変更権限 |
|------------|---------|
| グローバルポリシー | Owner（A操作） |
| プロジェクトポリシー | Owner（A操作） |
| マイルストーンポリシー | Owner / Maintainer（B操作） |
| サイクルポリシー | 自動（C操作） |

下位ポリシーは上位ポリシーを緩和できない。緩和が必要な場合はA操作として人間承認を要する。

---

## 7. 運転モード別の権限差分

要件定義書で定義されている3つの運転モードにより、B操作の承認要否が変わる。

| 操作 | 手動承認モード | 半自動モード | 全自動モード |
|------|:----------:|:--------:|:--------:|
| A操作（権限拡大等） | 4/4 承認必須 | 4/4 承認必須 | 4/4 承認必須 |
| B操作（diff超過等） | Owner承認後に進行 | バックアップ後に自動進行（通知あり） | バックアップ後に自動進行（通知あり） |
| B操作（CI設定変更） | Owner承認 + ペルソナ3承認 | ペルソナ3承認のみ | ペルソナ3承認のみ |
| C操作（軽微変更） | 自動（承認不要） | 自動（承認不要） | 自動（承認不要） |

> **注意**：A操作はいかなる運転モードでも4/4承認が必須であり、緩和できない。  
> モード切替自体の権限は §3 のシステム操作権限マトリクスに従う（Owner / Maintainer のみ）。
