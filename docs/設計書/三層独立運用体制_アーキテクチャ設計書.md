## 概要

本システムは**完全独立した3つの層**で構成される。各層は互いに干渉せず、
それぞれの責務範囲内でのみ動作する。

```

╠══════════════════════════════════════════════════════════════════╣
║                                                                  ║
```

## 1. 各層の責務定義

### 第1層：OpenClaw（ハードウェア監視層）

#### 役割の定義

OpenClawは**外部の独立した監査官・メンテナンススタッフ**として機能する。
PDCAシステムの内部ロジックや状態に一切関与せず、
**物理インフラの健全性のみを監視・保守する。**

#### 担当範囲（ハードウェア面のみ）

```
✅ OpenClawが担当する領域
─────────────────────────────────────────────────────
ハードウェア監視:
  - GPU温度・VRAM使用量・GPU使用率（nvidia-smi）
  - CPU温度・コア使用率・負荷平均
  - メモリ使用量・スワップ使用量
  - ディスク使用量・I/Oエラー
  - 電源状態・UPS残量・電圧異常
  - NIC リンク状態・パケットエラー率

OS・基盤層の保守:
  - Ubuntu セキュリティパッチ確認（通知のみ）
  - カーネルログ（kern.log）のハードウェアエラー検知
  - ファイルシステムエラー（SMART異常・fsck結果）
  - rootlessDockerデーモンの死活確認（プロセス存在確認のみ）
  - vLLMプロセスの死活確認（ポート疎通のみ）

物理的な自動修復（安全なもののみ）:
  - GPU温度過熱時のファン制御コマンド実行
  - ハングしたvLLMプロセスの再起動
  - ディスク容量警告時の古いログの自動アーカイブ

障害通知:
  - Discord #hw-incidents チャンネルへの通知
  - Discordに通知するだけで、PDCAシステムへの直接介入はしない

❌ OpenClawが担当しない領域（明示的な禁止事項）
─────────────────────────────────────────────────────
- PDCAシステムの設定ファイルへのアクセス・変更
- LLM APIキー・GitHub Appトークンへのアクセス
- Dockerコンテナの内容（PDCAシステムのコンテナ）への介入
- GitHub リポジトリへのアクセス
- PDCAシステムのデータベース・ログファイルへのアクセス
- PDCAシステムの再起動・強制停止
- Python依存パッケージの変更
- ネットワークホワイトリストの変更（ADR-007: A操作）
```

#### OpenClawの動作原則

```
原則1: 読み取り優先
  → ハードウェアメトリクスは読み取り専用で収集
  → 変更を伴う操作は最小限かつ安全なものに限定

原則2: 通知 → 人間承認 → 実行
  → システム変更を伴う修復は必ずDiscordで人間に確認を取る
  → 安全な操作（プロセス再起動等）は自律実行し事後通知

原則3: PDCAシステムとの完全独立
  → OpenClawはPDCAシステムのAPIを呼ばない
  → OpenClawはPDCAシステムのファイルシステムに触れない
  → 連携は「Discordへの通知」という一方向のみ
```

### 第2層：PDCAシステム（ソフトウェア・システム層）

#### 役割の定義

PDCAシステムは**ソフトウェアおよびシステム層の自動補完と自律運転**を担う。
ハードウェア層の物理的な状態には関知せず、
**ソフトウェアスタック内の問題を自律的に検知・補完・修復する。**

#### 担当範囲（ソフトウェア・システム面のみ）

```
✅ PDCAシステムが担当する領域
─────────────────────────────────────────────────────
ソフトウェア自動補完:
  - LLM API障害時のフォールバック（ADR-008: サーキットブレーカー）
  - クラウドLLM全断時のローカルLLM（vLLM）への自動切替
  - CI失敗の自動分類・リトライ（§9.5: コード起因/インフラ起因/フレーキー）
  - Dockerコンテナ再起動（PDCAシステム自身のコンテナのみ）

システム監視:
  - LLM APIレイテンシ・エラー率（LLMゲートウェイ経由）
  - PDCA状態機械の異常検知（停止条件監視）
  - コスト急増検知（§15.2: 日次上限・急増率）
  - 監査ログ整合性確認

GitHub・開発インフラの自動補完:
  - PR / Issue / CI の自動生成・管理
  - マイルストーン進捗の自動更新
  - RAGメモリの定期要約（10サイクルごと）

❌ PDCAシステムが担当しない領域（明示的な禁止事項）
─────────────────────────────────────────────────────
- ハードウェアへの直接コマンド（nvidia-smi 等）
- OS・カーネルレベルの操作
- ネットワーク設定の変更
- 電源管理・UPS制御
- OpenClawプロセスへの介入
```

#### 障害時の自動補完フロー（ソフト層）

```
【シナリオA: クラウドLLM障害】

  ADR-008 サーキットブレーカー検知
       ↓
  フォールバック順で自動切替（ADR-001）
       ↓
  全クラウドLLM障害 → ローカルvLLMへ自動切替
       ↓
  Discord #pdca-notifications に通知（情報レベル）
       ↓
  PDCAサイクル継続（縮退モード §13.2）


【シナリオB: PDCA停止条件発火（CI連続失敗・コスト急増等）】

  停止条件を自動検知
       ↓
  PDCA状態機械 → STOPPED状態へ遷移
       ↓
  原因分析レポート自動生成
       ↓
  Discord #pdca-incidents にP1通知
       ↓
  [人間] Owner/Maintainerが確認・判断
       ↓
  [人間] 再開承認 → PDCAサイクル再開
```

### 第3層：人間による監督・判断層

#### 役割の定義

人間は**両層の最終的な意思決定者・承認者**として機能する。
第1層（HW）・第2層（SW）の両方からDiscord経由で通知を受け取り、
**重大操作の承認・方針判断・A操作承認**を行う。

#### Discordチャンネルの分離設計

```
Discord サーバー構成
│
├── 【HW監視チャンネル】（OpenClaw専用）
│   ├── #hw-incidents    → GPU過熱・ディスク異常・電源警告等
│   ├── #hw-maintenance  → 定期メンテナンス報告・パッチ通知
│   └── #hw-status       → 定期ヘルスレポート（1時間ごと）
│
└── 【PDCAシステムチャンネル】（既存）
    ├── #pdca-incidents       → P0/P1インシデント（§19）
    ├── #pdca-notifications   → P2警告・サイクル完了・B操作通知
    └── #pdca-approvals       → A操作承認要求（4/4承認）
```

#### 人間が対応するエスカレーション

```
【HW層から来る通知への対応】

  OpenClaw通知                    人間の対応
  ─────────────────────────────────────────────────────
  GPU温度 > 85℃（警告）           確認・室温チェック
  GPU温度 > 95℃（危険）           PDCAシステムを手動で一時停止
                                  → 物理的な冷却対処
  VRAM使用率 > 98%（継続）         vLLMモデル設定の見直し指示
  ディスク残量 < 10%              データアーカイブ・追加ストレージ対処
  UPS残量 < 20%                  電源復旧・シャットダウン判断
  NVMe S.M.A.R.T.警告            ディスク交換計画
  GPU VRAM ECC エラー増加         GPU交換計画

【HW障害がPDCAに影響する場合の調整フロー】

  OpenClaw → #hw-incidents に通知
       ↓
  [人間] HW障害内容を確認
       ↓
  HW障害がPDCAサイクルに影響すると判断した場合
       ↓
  [人間] PDCAシステムを手動停止（Web UIまたはCLI）
       ↓
  [人間] OpenClawによるHW修復を待機
       ↓
  [人間] HW復旧確認後、PDCAサイクルを再開
       ↓
  Discord #pdca-approvals で再開承認

  ※ OpenClawとPDCAシステムは直接通信しない
  ※ 必ず「人間」が間に入って判断・調整を行う
```

## 2. インターフェース仕様（層間の通信ルール）

### 2.1 OpenClaw → 人間（通知のみ・一方向）

```python
# OpenClawのスキル実装例（ハードウェア監視専用）
# このスクリプトはPDCAシステムとは完全に独立して動作する

import subprocess
import httpx

DISCORD_WEBHOOK_HW = "https://discord.com/api/webhooks/xxx/hw-incidents"

def monitor_gpu():
    """GPUヘルスチェック（読み取り専用）"""
    result = subprocess.run(
        ["nvidia-smi",
         "--query-gpu=temperature.gpu,memory.used,memory.total,"
                       "ecc.errors.uncorrected.aggregate.total",
         "--format=csv,noheader,nounits"],
        capture_output=True, text=True
    )
    for i, line in enumerate(result.stdout.strip().split("\n")):
        temp, mem_used, mem_total, ecc_err = line.split(", ")
        temp, mem_used, mem_total, ecc_err = (
            int(temp), int(mem_used), int(mem_total), int(ecc_err)
        )

        # 危険レベル判定
        if temp >= 95:
            notify_discord(
                level="🚨 CRITICAL",
                channel=DISCORD_WEBHOOK_HW,
                message=(
                    f"GPU{i} 温度が危険域に達しました: {temp}℃\n"
                    f"**即座に物理的な冷却対処が必要です。**\n"
                    f"PDCAシステムの手動停止を検討してください。"
                )
            )
        elif temp >= 85:
            notify_discord(
                level="⚠️ WARNING",
                channel=DISCORD_WEBHOOK_HW,
                message=f"GPU{i} 温度警告: {temp}℃（上限85℃）"
            )

        # ECC UncorrectedエラーはGPU交換の前兆
        if ecc_err > 0:
            notify_discord(
                level="🔴 ECC ERROR",
                channel=DISCORD_WEBHOOK_HW,
                message=(
                    f"GPU{i} ECC Uncorrectedエラー検出: {ecc_err}件\n"
                    "GPU交換計画を開始してください。"
                )
            )

def notify_discord(level: str, channel: str, message: str):
    """Discordへの一方向通知（PDCAシステムとは別チャンネル）"""
    httpx.post(channel, json={
        "content": f"**[OpenClaw HW監視] {level}**\n{message}"
    })
```

### 2.2 OpenClaw → PDCAシステム（直接通信なし）

```
【禁止されている通信パターン】

  OpenClaw ──→ PDCAシステムのAPI           ❌ 禁止
  OpenClaw ──→ PDCAシステムのDB/ログ        ❌ 禁止
  OpenClaw ──→ PDCAシステムのDocker API     ❌ 禁止
  OpenClaw ──→ PDCAシステムの設定ファイル   ❌ 禁止

【許可されている通信パターン】

  OpenClaw ──→ Discord #hw-incidents       ✅ 通知のみ
  OpenClaw ──→ nvidia-smi（読み取り）       ✅ 読み取りのみ
  OpenClaw ──→ /proc, /sys（読み取り）      ✅ 読み取りのみ
  OpenClaw ──→ vLLM :8000/health（疎通確認）✅ GETのみ
```

### 2.3 PDCAシステム → OpenClaw（通信なし）

```
PDCAシステムはOpenClawの存在を知らない。
PDCAシステムはOpenClawのAPIを呼ばない。
PDCAシステムはHW監視結果を参照しない。

これが「独立した外部監査官」としての設計の核心である。
```

## 3. 障害シナリオ別の対応フロー

### シナリオ1：GPU温度異常（HW層の問題）

```
発生: GPU温度が90℃を超える

[OpenClaw] nvidia-smi で検知（5分ごとのcron）
       ↓
[OpenClaw] #hw-incidents に ⚠️ WARNING 通知
       ↓
[人間] 通知を確認
       ↓
[人間] 室温・エアフローを物理確認
       ↓
判断分岐:
  → 室温が高い場合: エアコン強化・ファンの追加
  → ファン故障の場合: PDCAを手動停止 → 部品交換 → PDCAを再開
  → 問題なしの場合: 経過監視（OpenClawが継続監視）

[PDCAシステム] この間、何も知らない（通知を受け取らない）
              ただし人間がPDCAを手動停止した場合はDiscordで承認フロー
```

### シナリオ2：vLLMプロセスのクラッシュ（HW/SW境界の問題）

```
発生: vLLMサーバーがOOMでクラッシュ

【OpenClawが検知する部分】
[OpenClaw] localhost:8000/health への疎通失敗を検知
[OpenClaw] #hw-incidents に通知
           「vLLMプロセスが応答しません。自動再起動を試みます。」
[OpenClaw] systemctl restart vllm（安全な操作として許可）
[OpenClaw] 再起動成功/失敗を #hw-incidents に事後報告

【PDCAシステムが検知する部分（独立して）】
[PDCAシステム] ローカルLLM呼び出し失敗を検知（LLMゲートウェイ）
[PDCAシステム] ADR-008: サーキットブレーカーがOPENに
[PDCAシステム] クラウドLLMへ自動フォールバック
[PDCAシステム] #pdca-notifications に通知
              「ローカルLLMへの接続失敗。クラウドLLMへフォールバック中」

【独立した2つの系統が並列して動作】
両者が独立して検知・対処し、Discordで人間が状況を把握できる
OpenClawの再起動成功後、PDCAシステムは次のサイクルで
ローカルLLMへの復帰を自動的に試みる（ADR-008 HALF_OPEN）
```

### シナリオ3：ディスク容量不足（HW層の問題）

```
発生: データ用SSDの空き容量が10%を下回る

[OpenClaw] df コマンドで検知（1時間ごとのcron）
       ↓
[OpenClaw] #hw-incidents に ⚠️ WARNING 通知
           使用量: 3.8TB / 4TB（残り200GB）
           最大消費者: /var/log/vllm（vLLMの推論ログ）
       ↓
[OpenClaw] 古いvLLMログを自動アーカイブ（安全な操作として許可）
           /var/log/vllm/*.log.* → /backup/logs/ へ移動
       ↓
[OpenClaw] アーカイブ後の空き容量を #hw-incidents に報告

[PDCAシステム] PDCAの監査ログ・レビューJSON等の保存は継続
              ディスク残量の監視はPDCAシステムの役割外
```

### シナリオ4：PDCA停止条件発火（SW層の問題）

```
発生: CI連続5回失敗（§6.6 停止条件）

[PDCAシステム] 停止条件を自動検知
[PDCAシステム] STOPPED状態へ遷移
[PDCAシステム] 原因分析レポートを自動生成
[PDCAシステム] #pdca-incidents に P1通知
       ↓
[人間] #pdca-incidents を確認
[人間] 原因調査（CIログ・コード確認）
[人間] 修正方針を決定
[人間] Discordで再開承認
       ↓
[PDCAシステム] 自動再開

[OpenClaw] このシナリオでは何もしない
           （ソフトウェアの問題であり、HW監視の対象外）
```

## 4. OpenClawの設定・権限設計

### 4.1 実行ユーザーの分離

```bash
# OpenClaw専用のLinuxユーザーを作成（最小権限）
sudo useradd -m -s /bin/bash openclaw-monitor
sudo usermod -aG video openclaw-monitor  # nvidia-smiのためGPUアクセスのみ

# PDCAシステムのファイルへのアクセスを明示的に禁止
# PDCAシステムは vibe-pdca ユーザーで実行
# openclaw-monitor は vibe-pdca の所有ファイルにアクセスできない
```

### 4.2 OpenClawに許可するコマンドリスト

```yaml
# ~/.openclaw/allowed_commands.yml（OpenClaw設定ファイル）

allowed_commands:
  # ハードウェア監視（読み取り専用）
  - "nvidia-smi"
  - "nvidia-smi --query-gpu=* --format=csv"
  - "sensors"          # CPU温度
  - "df -h"            # ディスク使用量
  - "free -h"          # メモリ使用量
  - "uptime"           # 負荷平均
  - "ip link show"     # NIC状態確認

  # 死活確認（GETのみ）
  - "curl -s http://localhost:8000/health"  # vLLM疎通確認

  # 安全な自動修復（事後通知必須）
  - "systemctl restart vllm"               # vLLM再起動のみ
  - "find /var/log/vllm -name '*.log.*' -mtime +7 -exec mv {} /backup/logs/ \\;"

denied_paths:
  # PDCAシステムの領域には絶対に触れない
  - "/home/vibe-pdca/**"
  - "/etc/vibe-pdca/**"
  - "/var/lib/docker/**"
  - "/root/**"

denied_network:
  # PDCAシステムの内部APIには接続しない
  - "localhost:8080"   # PDCAシステムWeb UI
  - "localhost:5432"   # 将来のDB
```

### 4.3 Tailscaleによるネットワーク分離

```
【ネットワーク設計】

インターネット
     │
     │（Tailscale VPN経由のみ・直接公開なし）
     ↓
 スマートフォン / 外部端末
     │
     │（Tailscale メッシュVPN）
     ↓
 ラックサーバー
  ├── OpenClaw（openclaw-monitor ユーザー）
  │   └── Tailscale経由でのみアクセス可能
  │   └── PDCAシステムの内部ポートにはiptablesでアクセス禁止
  │
  └── PDCAシステム（vibe-pdca ユーザー）
      └── 外部には露出しない（GitHub API・LLM APIへの送信のみ）

# iptablesルール（openclaw-monitorからPDCAへのアクセスをブロック）
iptables -A OUTPUT -m owner --uid-owner openclaw-monitor \
  -d 127.0.0.1 -p tcp --dport 8080 -j REJECT
iptables -A OUTPUT -m owner --uid-owner openclaw-monitor \
  -d 127.0.0.1 -p tcp --dport 5432 -j REJECT
```

## 5. OpenClawの定期実行スケジュール

```yaml
# OpenClaw cron設定（ハードウェア監視専用）

schedules:
  # 高頻度監視（クリティカル項目）
  - cron: "*/5 * * * *"      # 5分ごと
    task: monitor_gpu_temperature
    description: "GPU温度・VRAMの監視"
```

## 8. 設計の利点とトレードオフ

### 利点

```
1. 障害の波及防止
   → HW障害がSWシステムを誤動作させない
   → SW障害がHW監視を止めない
   → 両系統が独立して動き続ける

2. セキュリティの向上
   → OpenClawはPDCAシステムの機密情報（APIキー等）に触れない
   → PDCAシステムはHW制御コマンドを実行できない
   → プロンプト注入がHW操作に波及しない

3. 監査の明確化
   → HWログとSWログが完全に分離される
   → インシデント原因の特定が容易になる
   → 「どの層の問題か」がDiscordチャンネルで即座に判別できる

4. 段階的な対応が可能
   → HW問題はOpenClawが自律対処
   → SW問題はPDCAシステムが自律対処
   → 重大判断のみ人間に集約
```

### トレードオフ

```
1. 情報の断絶
   → PDCAシステムはGPU温度を知らないため、
     VRAMが実は物理的に不安定でも気づけない
   → 対策: 人間が両チャンネルを定期確認する

2. vLLMクラッシュの二重対処
   → OpenClawとPDCAシステムが独立して検知・対処するため
     タイミングによっては重複したアクションが発生しうる
   → 対策: OpenClawの再起動完了後にPDCAが復帰試行するため
     実害は少ない（ADR-008のHALF_OPEN待機が緩衝材になる）

3. 人間の確認負荷
   → 2系統のDiscordチャンネルを監視する必要がある
   → 対策: 重大通知はメンション（@Owner）を使い、
     軽微な通知はサマリチャンネルにまとめる
```
