### 追加提案 16: ロールバック戦略の体系化とワンクリックロールバック

**現状の課題**:
`intervention.py`にロールバック機能の基盤はありますが、ロールバックの粒度（タスク単位/サイクル単位/マイルストーン単位）、ロールバック後のステート整合性保証、ロールバック履歴の追跡が体系化されていません。ACTフェーズで`reject`判定された場合のリカバリフローも、具体的な自動化手順が不明確です。

**提案内容**:
- **多粒度ロールバック**: タスク単位（PRのrevert）、サイクル単位（ステートの巻き戻し）、マイルストーン単位（複数サイクルの一括取消）の3レベルを定義
- **ロールバックプレビュー**: 実行前に影響範囲（対象PR、変更ファイル、依存タスク）を一覧表示
- **ステート整合性チェッカー**: ロールバック後のPDCAステートマシン・監査ログ・GitHub Issue の整合性を自動検証
- **ロールバック連鎖の検出**: あるタスクのロールバックが他のタスクに波及する場合の自動警告
- **GUIワンクリックロールバック**: ダッシュボードから操作可能なロールバックUI

**期待効果**: 障害復旧時間の短縮、ロールバック操作のヒューマンエラー防止

### 追加提案 17: 設定変更のホットリロードとバージョン管理

**現状の課題**:
`config/loader.py`による階層的な設定読み込みは実装されていますが、設定変更時にシステム再起動が必要です。また、設定変更の履歴追跡、変更前後の差分表示、設定変更のロールバック機能がありません。運用中のコスト上限やLLMプロバイダの切替を、サイクル中断なく行う必要があります。

**提案内容**:
- **ホットリロード**: ファイルシステム監視（`watchdog`）またはシグナル（`SIGHUP`）による設定の動的再読み込み
- **設定バージョニング**: 変更ごとにバージョン番号を付与し、監査ログに記録
- **差分表示**: 変更前後の設定値をdiff形式で表示
- **段階的適用**: 次のサイクル開始時に新設定を適用する「遅延適用モード」
- **バリデーション**: 新設定の整合性チェック（コスト上限の最小値、プロバイダの有効性等）を適用前に実行
- **ロールバック**: 過去の設定バージョンへの即座の復帰

**期待効果**: 運用中のダウンタイムゼロでの設定調整、設定変更の監査追跡

### 追加提案 18: PDCAサイクルのタイムアウト戦略の精緻化

**現状の課題**:
現在、6時間の一律タイムアウトが設定されていますが（`stop_conditions.py`）、タスクの複雑度やフェーズごとの所要時間は大きく異なります。シンプルなバグ修正は30分で完了しうる一方、大規模リファクタリングでは6時間でも不足する可能性があります。

**提案内容**:
- **フェーズ別タイムアウト**: PLAN（30分）、DO（3時間）、CHECK（1時間）、ACT（30分）等、フェーズごとに個別設定
- **タスク複雑度ベースの動的タイムアウト**: PLANフェーズで推定される複雑度に基づいてタイムアウト値を自動調整
- **プログレスベースの延長**: 進捗が確認できる場合（例：CIジョブが実行中）は自動延長
- **段階的エスカレーション**: タイムアウト50%到達で警告通知、75%で介入要求、100%でストップ
- **タイムアウト統計**: 過去のサイクルの所要時間データに基づくタイムアウト値の自動最適化

**期待効果**: 不必要な早期中断の防止、長時間スタックの早期検出

### 追加提案 19: サンドボックスのリソース制限と監視の強化

**現状の課題**:
Docker rootlessコンテナによるサンドボックス（ADR-007）は設計されていますが、コンテナ内のリソース消費（CPU、メモリ、ディスクI/O）の上限設定と監視、および悪意のあるコードによるリソース枯渇攻撃への対策が具体化されていません。

**提案内容**:
- **リソース制限の明示的設定**: `--memory`, `--cpus`, `--pids-limit`, `--storage-opt` によるコンテナリソース上限
- **リアルタイムリソース監視**: コンテナ実行中のCPU/メモリ/ディスク使用率をメトリクスとして収集
- **リソース超過時の自動終了**: OOMKillの検出とグレースフルなエラーハンドリング
- **実行時間制限**: コンテナ単位のタイムアウト（`--stop-timeout`）
- **ネットワーク帯域制限**: 大量のパッケージダウンロードによる帯域枯渇の防止
- **ファイルシステムサイズ制限**: 一時ファイルの無制限生成の防止

**期待効果**: DOフェーズの安全性向上、ホストシステムの保護

### 追加提案 20: GitHub App のスコープ動的管理とトークンローテーション

**現状の課題**:
GitHub App認証は`github/__init__.py`にプレースホルダー実装されていますが（PyJWTスタブ）、本番実装に向けて以下が未整備です：インストールトークンの有効期限管理、最小権限の動的適用、トークン漏洩時の即座の無効化。

**提案内容**:
- **トークン自動ローテーション**: インストールトークンの有効期限（1時間）到達前の自動更新
- **スコープの動的縮小**: フェーズに応じた最小権限の適用（例：CHECKフェーズではContents Read-Only）
- **トークン漏洩検出**: ログ出力やエラーメッセージにトークンが含まれていないかの自動スキャン
- **緊急無効化フロー**: トークン漏洩検出時のインストール即時無効化とP0インシデント自動起票
- **アクセスログ**: GitHub API呼び出しの詳細ログ（エンドポイント、パラメータ、レスポンスコード）

**期待効果**: GitHub連携のセキュリティ強化、認証障害の防止

### 追加提案 21: PDCAサイクルの依存関係グラフとクリティカルパス分析

**現状の課題**:
マイルストーン内のタスクは最大7個まで生成されますが、タスク間の依存関係（先行タスク・後続タスク）は明示的に管理されていません。並列実行可能なタスクの特定や、ボトルネックとなるタスクの予測ができません。

**提案内容**:
- **依存関係グラフ**: PLANフェーズでタスク間の依存関係を有向非巡回グラフ（DAG）として生成
- **クリティカルパス分析**: 最長所要時間のパスを特定し、サイクル完了時間を予測
- **並列実行スケジューリング**: 依存関係のないタスクの同時実行（将来的な並列DOフェーズへの布石）
- **依存関係の可視化**: ダッシュボード上にDAGをインタラクティブに表示
- **ブロッカー検出**: 他の全タスクをブロックしているタスクの早期警告

**期待効果**: サイクル完了時間の短縮、ボトルネックの早期発見

### 追加提案 22: LLMレスポンスの品質スコアリングと自動リトライ

**現状の課題**:
LLMからのレスポンスに対する品質評価は、CHECKフェーズの5ペルソナレビューに依存しています。しかし、DOフェーズでのコード生成やPLANフェーズでのマイルストーン生成における即座の品質チェック（構文エラー、不完全な出力、ハルシネーション検出）がありません。

**提案内容**:
- **構造的バリデーション**: LLMレスポンスが期待するJSON/Markdownスキーマに準拠しているかの即座のチェック
- **完全性スコアリング**: 要求された項目（タスクリスト、レビューコメント等）が漏れなく含まれているかの定量評価
- **ハルシネーション検出**: 存在しないファイルパス、API、ライブラリへの参照を自動検出
- **自動リトライ**: 品質スコアが閾値以下の場合、エラーフィードバックを含めた自動再問い合わせ（最大3回）
- **品質メトリクス**: モデル別・ロール別の品質スコア推移を追跡

**期待効果**: LLM出力の信頼性向上、不良出力によるサイクル遅延の防止

### 追加提案 23: 変更影響分析（インパクトアナリシス）の自動化

**現状の課題**:
DOフェーズで生成されたコード変更が、既存コードベースに与える影響の範囲を事前に評価する仕組みがありません。CIテストで事後的に検出されるのみです。

**提案内容**:
- **静的依存関係分析**: 変更対象ファイルを参照/依存するファイルの自動特定（AST解析、import追跡）
- **影響スコア算出**: 変更の波及範囲の広さを定量化（高影響 = 多くのファイルに波及）
- **テスト対象の自動特定**: 変更に関連するテストファイルの自動抽出と優先実行
- **破壊的変更の検出**: パブリックAPI、データモデル、設定スキーマの互換性を壊す変更の警告
- **CHECKフェーズへのフィード**: 影響分析結果を5ペルソナレビューのコンテキストとして注入

**期待効果**: CI失敗率の低減、レビュー品質の向上

### 追加提案 24: ユーザーフィードバック収集と満足度トラッキング

**現状の課題**:
PDCAサイクルの品質は、CI結果と5ペルソナレビューで評価されますが、最終的なユーザー（開発者・プロジェクトオーナー）の満足度を体系的に収集する仕組みがありません。システムが生成した成果物が実際に有用だったかのフィードバックループが閉じていません。

**提案内容**:
- **サイクル完了後のフィードバックフォーム**: 成果物の品質（1-5点）、有用性、改善要望を構造化して収集
- **PR承認/拒否の追跡**: 人間レビュアーによるPR承認率を品質指標として監視
- **フィードバック→学習連携**: 収集したフィードバックを10サイクルごとの学習フィードバック（`learning.py`）に統合
- **NPS（Net Promoter Score）相当の満足度トラッキング**: 長期的なシステム改善の指標
- **フィードバックのカテゴリ分類**: コード品質/設計/テスト/ドキュメント別の満足度分析

**期待効果**: ユーザー視点でのシステム品質改善、実際の有用性の定量把握

### 追加提案 25: コンプライアンステンプレートとポリシーエンジン

**現状の課題**:
A/B/C操作分類によるガバナンスは優れていますが、業界固有のコンプライアンス要件（SOC 2、ISO 27001、GDPR、HIPAA等）への対応が考慮されていません。組織のセキュリティポリシーをシステムのガバナンスルールに反映する仕組みがありません。

**提案内容**:
- **コンプライアンステンプレート**: SOC 2 / ISO 27001 / GDPR / HIPAA の各要件に対応するチェックリスト
- **ポリシーエンジン**: 組織固有のルール（「本番環境への変更は必ずA操作」「個人情報を含むコードは追加レビュー必須」等）をYAML/JSONで定義・適用
- **自動コンプライアンスチェック**: 各PDCAサイクルのACTフェーズでポリシー違反を自動検出
- **コンプライアンス監査レポート**: 指定期間のポリシー準拠率を自動集計
- **ポリシーのバージョン管理**: 組織ポリシーの変更履歴と適用開始日の追跡

**期待効果**: 規制産業でのシステム採用の促進、コンプライアンス対応工数の削減

### 追加提案 26: マルチリポジトリ連携とモノレポ対応

**現状の課題**:
`multi_project.py`によるプロジェクト間の物理的隔離（ADR-006）は設計されていますが、関連する複数リポジトリにまたがる変更（マイクロサービス間のAPI変更、共有ライブラリの更新等）の協調的なPDCAサイクル管理がありません。モノレポ構成のプロジェクトへの対応も未定義です。

**提案内容**:
- **クロスリポジトリ依存関係マップ**: 複数リポジトリ間のAPI依存関係を自動検出・管理
- **協調的PDCAサイクル**: 関連リポジトリへの同時変更を一つのGoalで管理
- **モノレポ対応**: パッケージ/サービス単位のスコープ指定によるPDCAサイクル実行
- **統合テスト調整**: クロスリポジトリのインテグレーションテスト自動実行
- **リリース協調**: 関連リポジトリの同時リリース判定

**期待効果**: マイクロサービスアーキテクチャへの対応、大規模プロジェクトでの利用

### 追加提案 27: AIモデルのファインチューニングパイプライン

**現状の課題**:
学習フィードバック（`learning.py`）は10サイクルごとにPLANプロンプトを改善しますが、これはプロンプトエンジニアリングの範囲です。プロジェクト固有のコーディングスタイル、アーキテクチャパターン、レビュー基準に特化したモデルの微調整（ファインチューニング）は考慮されていません。

**提案内容**:
- **トレーニングデータ自動収集**: 承認されたPR、成功したPLAN、高評価レビューからファインチューニング用のデータセットを自動構築
- **LoRA/QLoRAファインチューニング**: ローカルLLM（Ollama経由）のプロジェクト特化型微調整
- **バリデーションパイプライン**: ファインチューニング前後のモデル品質比較
- **A/Bデプロイ**: ファインチューニング済みモデルと汎用モデルの並行運用・比較
- **定期的な再トレーニング**: 蓄積データの増加に伴う定期的なモデル更新

**期待効果**: プロジェクト固有の品質向上、ローカルLLMの実用性向上

### 追加提案 28: 自然言語によるシステム操作インターフェース（チャットオプス）

**現状の課題**:
システム操作はCLI/GUIまたはDiscordの定型メッセージに限定されています。「今のサイクルの進捗は？」「コストが高すぎるので上限を下げて」「次のサイクルはセキュリティ重視で」といった自然言語での対話的なシステム制御ができません。

**提案内容**:
- **チャットオプスインターフェース**: Discordボットまたは専用チャットUIで、自然言語のコマンドを解釈して実行
  - 「サイクル3の結果を見せて」→ レポート生成
  - 「コスト上限を$20に下げて」→ 設定変更（B操作として処理）
  - 「次のPLANではパフォーマンスを重視して」→ プロンプトのコンテキスト注入
  - 「今すぐ止めて」→ グレースフルストップ実行
- **意図分類エンジン**: ユーザー入力をクエリ/コマンド/フィードバックに自動分類
- **確認フロー**: 破壊的操作の前にユーザー確認を挟む
- **コンテキスト保持**: 会話の文脈を維持した連続的な対話

**期待効果**: 操作の直感性向上、非技術者でもシステム操作が可能

### 追加提案 29: PDCAサイクルのリプレイとデバッグモード

**現状の課題**:
ドライランモード（`dry_run.py`）はPLAN→CHECKのシミュレーションが可能ですが、過去のサイクルを完全に再現（リプレイ）する機能はありません。特定のサイクルで問題が発生した場合、原因調査のためにLLMレスポンスを含めた完全な再現が困難です。

**提案内容**:
- **サイクルスナップショット**: 各フェーズの入出力（LLMプロンプト、レスポンス、決定、CI結果等）を完全に記録
- **リプレイモード**: スナップショットを使って過去のサイクルを忠実に再現（LLMは記録済みレスポンスを返す）
- **ステップ実行**: フェーズごとに一時停止し、中間状態を確認できるデバッグモード
- **What-If分析**: 特定のLLMレスポンスやCI結果を差し替えて「もしこうだったら」のシミュレーション
- **比較ビュー**: 2つのサイクルの入出力を並べて比較し、差分を特定

**期待効果**: 問題の根本原因分析の効率化、システムの透明性向上

---

### 追加提案 30: エッジケースとコーナーケースの自動探索

**現状の課題**:
CHECKフェーズのレビューは5ペルソナによる定性的な評価が中心で、生成コードに対する体系的なエッジケース探索（境界値テスト、異常入力テスト、並行処理テスト等）が自動化されていません。

**提案内容**:
- **プロパティベーステスト生成**: 生成コードに対してHypothesis等を用いたプロパティベーステストを自動生成
- **ファジング**: 入力パラメータのランダム変異によるクラッシュ・例外の探索
- **境界値分析**: 数値パラメータの最小値・最大値・ゼロ・負数での動作確認
- **並行処理テスト**: 競合状態やデッドロックの検出（適用可能な場合）
- **エッジケースレポート**: 発見された問題をCHECKフェーズのレビュー結果に自動統合

**期待効果**: 生成コードの堅牢性向上、見落とされやすいバグの早期発見
