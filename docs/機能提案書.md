# Vibe-PDCA システム 機能提案書

**作成日**: 2026-03-04
**対象**: Vibe-coding PDCA System v0.1.0
**目的**: 現在の要件定義書（v6）・実装手順書（v3）・ADR・ソースコードを精査した上で、未カバーの機能や改善案を提案する

---

## 📋 提案概要

現在のシステム設計は非常に包括的であり、PDCAサイクルの自動化・ガバナンス・監査・コスト管理・マルチプロバイダLLM対応など、主要機能は網羅されています。以下は、現在の設計に**含まれていない**、または**強化が望ましい**機能の提案です。

---

## 🔴 優先度：高（運用品質に直結）

### 提案 1: Webhook / イベント駆動アーキテクチャの導入

**現状の課題**:
現在のシステムはCLI/GUI起動による同期的な処理フローが中心です。GitHub上のイベント（Issue作成、PR更新、CIステータス変更等）に対するリアルティムの反応機構が設計に含まれていません。

**提案内容**:
- GitHub Webhookの受信サーバー（FastAPIベース）を追加し、以下のイベントでPDCAサイクルを自動トリガーする仕組みを構築
  - `issues.opened` → 新規Goal検出 → PLANフェーズ自動開始
  - `pull_request.review_submitted` → CHECKフェーズへの外部レビュー取り込み
  - `check_suite.completed` → CI結果のリアルタイム反映
  - `issue_comment.created` → 人間からの介入指示の自動検出
- イベントキューイング（Redis/インメモリキュー）によるバックプレッシャー制御

**期待効果**: 手動トリガーの削減、リアルタイム性の向上、GitHubネイティブなワークフロー統合

---

### 提案 2: セッション永続化とクラッシュリカバリ機構

**現状の課題**:
PDCAステートマシンはインメモリで動作しており、プロセス再起動時にサイクルの途中状態が失われる可能性があります。GitHub Issueをステートストアとして使う設計は記載されていますが、ローカル状態の定期的なチェックポイント保存やクラッシュリカバリの具体的な実装方針が不明確です。

**提案内容**:
- PDCAステートの定期チェックポイント保存（JSON/YAML形式、設定可能な間隔）
- プロセス起動時の自動リカバリ：最新チェックポイントからのサイクル再開
- 「ダーティシャットダウン」検出フラグによる不正終了時の自動通知
- チェックポイントデータの整合性検証（既存のchain-hashメカニズムの活用）

**期待効果**: 運用安定性の大幅向上、長時間サイクル実行中の障害耐性

---

### 提案 3: レート制限・スロットリングの統合管理

**現状の課題**:
コスト管理（$30/日、80コール/サイクル）は実装されていますが、各LLMプロバイダ固有のAPIレート制限（RPM/TPM）への対応が設計に含まれていません。GitHub API のレート制限（5000リクエスト/時間、Search API 30リクエスト/分）への対応も不十分です。

**提案内容**:
- プロバイダごとのレート制限トラッカー（トークンバケットアルゴリズム）
- レート制限到達時の自動待機（exponential backoff）とフォールバック
- GitHub API呼び出しの集約とキャッシュ（特にRAG検索用）
- レート制限状況のダッシュボード表示

**期待効果**: API制限によるサイクル中断の防止、安定した長時間運用

---

### 提案 4: テスト戦略の拡充 — E2Eテストとシナリオテスト

**現状の課題**:
単体テスト（28ファイル）は充実していますが、PDCAサイクル全体を通したEnd-to-Endテスト、およびシナリオベースのテスト（異常系・境界値・マルチサイクル）が不足しています。

**提案内容**:
- **E2Eシナリオテスト**: Goal入力→Milestone生成→Task実行→Review→Decision の完全サイクルをモックLLMで自動テスト
- **異常系シナリオ**: LLM全プロバイダダウン、GitHub API障害、コスト上限到達、6時間スタック等のシミュレーション
- **マルチサイクルテスト**: 10サイクル連続実行による学習フィードバック・劣化検出の検証
- **GUI統合テスト**: Fletコンポーネントの状態遷移テスト（現在テストなし）
- テスト用のLLMモックサーバー（固定レスポンスの提供）

**期待効果**: リグレッション防止、リリース品質の保証

---

## 🟡 優先度：中（ユーザー体験・運用効率の向上）

### 提案 5: ダッシュボードの高度化 — 可視化とインタラクション

**現状の課題**:
現在のGUI（Flet）はステータス表示とログビューアが中心で、PDCAサイクルの進行状況やコスト推移を視覚的に把握することが困難です。

**提案内容**:
- **PDCAサイクルタイムライン**: 各フェーズ（PLAN/DO/CHECK/ACT）の進行をガントチャート風に表示
- **コスト推移グラフ**: 日別/サイクル別のLLMコスト推移をリアルタイムで可視化
- **トレーサビリティマップ**: Goal→Milestone→Task→PR→Decision の関連を対話的なグラフとして表示
- **5ペルソナレビュー結果のレーダーチャート**: 各ペルソナの評価スコアを視覚的に比較
- **アラート通知パネル**: コスト警告、ストップ条件接近、ヘルスチェック異常をリアルタイム表示

**期待効果**: 意思決定の迅速化、システム状態の直感的把握

---

### 提案 6: プロンプトバージョニングとA/Bテスト機能

**現状の課題**:
プロンプトテンプレートのバージョン管理は設計に含まれていますが、複数バージョンのプロンプトを同時に運用して効果を比較するA/Bテスト機能がありません。学習フィードバック（10サイクルごと）でプロンプトを改善する仕組みはありますが、改善の効果測定が定量的に行えません。

**提案内容**:
- プロンプトテンプレートの複数バージョン同時運用
- サイクル単位でのランダム割り当て（A/Bスプリット）
- 成功率・レビュー品質・コストなどのKPIによるバージョン比較レポート
- 統計的有意差検定による自動バージョン選択

**期待効果**: プロンプト品質の継続的・定量的な改善

---

### 提案 7: 通知チャネルの多様化

**現状の課題**:
現在、外部通知はDiscordのみが設計されています。チームのコミュニケーションツールがDiscord以外の場合や、複数チャネルへの同時通知が必要な場合に対応できません。

**提案内容**:
- **Slack統合**: Slack Webhookによる通知（Discord設計と同等の機能）
- **メール通知**: P0/P1インシデント時のSMTP/SES経由の緊急通知
- **Microsoft Teams統合**: Teams Webhookによる企業環境対応
- **通知ルーティングエンジン**: イベント種別×重要度でチャネルを自動選択
- プラグインシステム（既存の`PluginCategory.NOTIFICATION`）を活用した拡張可能な設計

**期待効果**: 多様なチーム環境への対応、重要通知の見逃し防止

---

### 提案 8: 国際化（i18n）対応

**現状の課題**:
システムの用語集・プロンプト・UIは日本語中心で設計されています。グローバルチームでの利用や、英語環境のLLMとの整合性確保に課題があります。

**提案内容**:
- UIテキストの外部リソースファイル化（JSON/YAML）
- プロンプトテンプレートの言語別バリアント管理
- 用語集（Glossary）の多言語対応
- LLMレスポンスの言語指定と自動翻訳パイプライン
- ロケール設定による自動切替

**期待効果**: グローバルチームでの利用可能性、LLMとの言語的整合性向上

---

### 提案 9: マルチユーザー同時アクセス制御

**現状の課題**:
RBACの権限モデルは定義されていますが、複数ユーザーが同時にシステムを操作した場合の排他制御・競合解決の仕組みが不明確です。

**提案内容**:
- PDCAサイクル単位の楽観的ロック（バージョン番号による競合検出）
- A操作承認の同時実行制御（二重承認の防止）
- 設定変更の排他ロック
- 競合発生時のユーザー通知と手動解決フロー

**期待効果**: チーム運用時のデータ整合性保証

---

## 🟢 優先度：低（将来的な拡張性・差別化）

### 提案 10: PDCAサイクルのテンプレート化とカスタマイズ

**現状の課題**:
PDCAサイクルのフェーズ構成・タスク上限・ペルソナ構成は固定的で、プロジェクトの性質に応じた柔軟なカスタマイズができません。

**提案内容**:
- プロジェクトタイプ別のPDCAテンプレート（例：ホットフィックス用の簡易サイクル、大規模リファクタリング用の拡張サイクル）
- ペルソナの追加・削除・重み付け変更のUI
- フェーズごとのカスタムゲート条件の設定
- テンプレートの共有・インポート/エクスポート

**期待効果**: 多様なプロジェクト形態への柔軟な対応

---

### 提案 11: 分析・レポーティング機能の強化

**現状の課題**:
メトリクス収集（cycle成功率、CI率、モデル別メトリクス）は実装されていますが、長期的なトレンド分析や定期レポートの自動生成機能がありません。

**提案内容**:
- **週次/月次サマリーレポート**: PDCAサイクル数、成功率推移、コスト推移、主要な意思決定の要約を自動生成
- **プロジェクト横断ダッシュボード**: マルチプロジェクト環境での全体俯瞰ビュー
- **ボトルネック分析**: フェーズ別滞留時間、失敗パターン、コスト集中箇所の自動検出
- **PDFエクスポート**: 監査・経営報告用のフォーマット済みレポート出力
- **カスタムメトリクス**: ユーザー定義のKPI追跡

**期待効果**: 経営層への報告、継続的改善のためのデータドリブンな意思決定

---

### 提案 12: 外部CI/CDサービスとの深い統合

**現状の課題**:
CI結果の取得はGitHub Actions前提で設計されていますが、他のCI/CDサービス（GitLab CI、CircleCI、Jenkins等）との統合や、カスタムCIパイプラインとの連携が考慮されていません。

**提案内容**:
- CI/CDプロバイダの抽象化レイヤー（プラグインシステム活用）
- GitHub Actions以外のCIサービスとのアダプター
  - GitLab CI/CD
  - CircleCI
  - Jenkins（Webhook経由）
- CI結果の標準化フォーマット（成功/失敗/スキップ + カテゴリ分類）
- カスタムCIステップの組み込み（例：セキュリティスキャン、パフォーマンステスト）

**期待効果**: CI/CD環境に依存しない汎用的なシステム

---

### 提案 13: ナレッジベースの蓄積と活用

**現状の課題**:
RAGコンテキストマネージャはGitHub Search APIを利用した短期的なコンテキスト取得に限定されています。過去のPDCAサイクルから得られた知識（成功パターン、失敗パターン、設計判断の理由等）を構造化して蓄積・再利用する仕組みがありません。

**提案内容**:
- **決定履歴DB**: 過去のACTフェーズの意思決定とその結果（成功/失敗）を構造化して蓄積
- **パターンライブラリ**: 成功したPLANテンプレート、効果的なレビュー指摘パターンの自動抽出
- **類似案件検索**: 新しいGoalに対して、過去の類似案件と結果を自動提示
- **ベストプラクティス自動生成**: 蓄積データから技術的なベストプラクティスを定期的に抽出
- **プラグインシステム**（既存の`PluginCategory.KNOWLEDGE`）を活用した拡張可能な設計

**期待効果**: 組織的な学習の加速、同じ失敗の繰り返し防止

---

### 提案 14: API / SDKの公開

**現状の課題**:
現在のシステムはCLI/GUIによる操作が前提で、外部システムからプログラム的にPDCAサイクルを制御するインターフェースがありません。

**提案内容**:
- **REST API**: FastAPIベースのHTTP APIサーバー
  - Goal作成・サイクル開始/停止/再開
  - ステータス照会・メトリクス取得
  - A操作の承認/却下
  - エクスポートデータの取得
- **Python SDK**: APIクライアントライブラリ
- **OpenAPI仕様書**: Swagger UIによるAPIドキュメント自動生成
- **APIキー認証**: 外部システムからの安全なアクセス

**期待効果**: 他ツール（CI/CD、プロジェクト管理ツール等）との自動連携

---

### 提案 15: セキュリティ強化 — 監査ログの外部転送と改ざん検知

**現状の課題**:
監査ログはchain-hashによる改ざん検知を備えていますが、ログ自体がローカルファイルに保存されるため、ホストOSレベルの改ざんには脆弱です。また、ADRにて「監査ログの単一障害点」がリスクとして認識されています。

**提案内容**:
- **外部ログ転送**: 監査ログを外部サービス（CloudWatch Logs、Datadog、Elasticsearch等）にリアルタイム転送
- **イミュータブルストレージ**: S3 Object Lock / Azure Immutable Blob による改ざん不可能なバックアップ
- **ログ署名**: デジタル署名による出所証明
- **定期的な整合性監査**: ローカルログとリモートログの自動照合
- **SIEM連携**: セキュリティイベントの自動相関分析

**期待効果**: コンプライアンス要件への対応、監査ログの信頼性向上

## 💡 補足：既存設計で特に優れている点

現在の設計には以下の優れた点があり、これらは維持・強化されるべきです：

- **5ペルソナレビュー統合**: 多角的な品質評価の仕組み
- **A/B/C操作分類とガバナンス**: リスクに応じた承認フロー
- **サーキットブレーカーとフォールバック**: LLM障害への自動対応
- **chain-hash監査ログ**: 改ざん検出の基盤
- **三層独立運用体制**: ハードウェア・ソフトウェア・人間の明確な責務分離
- **コスト管理**: 多層的な制限（日次/サイクル/コール数）
- **ストップ条件**: 7つの自動停止条件による安全性確保
