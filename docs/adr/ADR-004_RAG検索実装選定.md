## 背景・課題

PDCAサイクルの各フェーズにおいて、過去のドキュメントを参照するRAG機構が必要である（§16.5）。チャンク戦略はファイル単位と確定済みだが、検索の実装方式を選定する必要がある。

---

## 選択肢

### 案A：GitHub Search API（採用）

GitHubが提供するコード検索・テキスト検索APIを使用する。

- **利点**：追加インフラ不要、セットアップコストが極めて低い、ドキュメントの版管理と検索が同一プラットフォームで完結
- **欠点**：キーワード検索のみ（セマンティック検索ではない）、APIレート制限あり

### 案B：ローカルベクトルDB（例：ChromaDB / Qdrant）

ドキュメントをベクトル化してローカルDBに格納し、セマンティック検索を実現する。

- **利点**：意味的類似度に基づく高精度な検索
- **欠点**：追加インフラの運用が必要、エンベディングモデルのコスト、物理分離との整合性確保が複雑

---

## 採用理由

**案A（GitHub Search API）を採用する。**

本システムのドキュメントはすべてGitHubリポジトリ内に保存・版管理されており、GitHub Search APIで直接検索するのが最も自然。追加インフラが不要であり、マルチプロジェクト物理分離（§26.1）との整合性が自動的に確保される。

---

## 検索仕様（§16.5確定事項）

| 項目 | 仕様 |
|------|------|
| 検索対象 | ファイル名・ファイルパス・ファイル内容の全文検索 |
| チャンク単位 | ファイル単位（1ファイル = 1チャンク） |
| 取得件数 | 関連ファイル最大5件 |
| コンテキスト投入形式 | 各ファイルの先頭200トークン + 関連セクション抽出 |
| 合計トークン上限 | 2,000トークン以内 |

---

## トレードオフ

- キーワード検索はセマンティック検索に比べて「意図の理解」が弱いが、適切な命名規則で補完可能
- GitHub Search APIのレート制限（30リクエスト/分）を超過する場合、キャッシュ層の導入が必要

---

## 撤回条件

- 10サイクルごとの検索品質評価（§16.5参照）で、ヒット率が70%を下回る状態が3回連続した場合
- ドキュメント量が増大し、キーワード検索のノイズが許容範囲を超えた場合
