---

## 背景・課題

Claudeが生成したコードのビルド・テスト・静的解析を安全に実行するための隔離環境（サンドボックス）が必要である。悪意あるコードや予期しない副作用からホスト環境を保護しつつ、開発に必要な機能を提供する。

---

## 選択肢

### 案A：Dockerコンテナ + ホワイトリスト管理（採用）
- **利点**：成熟した技術基盤、rootlessで権限最小化、ネットワーク制御が柔軟
- **欠点**：完全なカーネル隔離ではない

### 案B：gVisor / Kata Containers
- **利点**：より強固な隔離
- **欠点**：パフォーマンスオーバーヘッドが大きい、一部システムコールが非対応

### 案C：Firecracker microVM
- **利点**：最も強固な隔離、起動が高速
- **欠点**：Linux KVM依存、運用管理が複雑

---

## 採用理由

**案A（Dockerコンテナ + ホワイトリスト管理）を採用する。** 主要リスクはrootlessコンテナ + ネットワークホワイトリストで十分に緩和できる。

---

## コンテナ設計要件

### 実行環境
- コンテナはプロジェクト・サイクルごとに起動し、終了後に破棄する（ステートレス）
- rootで実行しない（非rootユーザー必須）
- ホストファイルシステムへのマウントは読み取り専用かつ最小範囲

### ネットワーク制御（ホワイトリスト）

| 許可対象 | 用途 |
|---------|------|
| GitHub API | PR作成・CI取得 |
| パッケージレジストリ（npm / PyPI等） | 依存解決 |
| LLM API（Anthropic / OpenAI / Google / xAI） | モデル呼び出し |

上記以外の外部通信はすべて遮断（デフォルト拒否）。ホワイトリストの変更はA操作。

### 禁止事項
- 危険コマンドの実行を検知・ブロック
- 秘密情報のコンテナ外への出力を検知・ブロック
- ホストネットワークモード使用を禁止

---

## 撤回条件

- コンテナエスケープの脆弱性が発見された場合
- gVisor/Firecrackerのオーバーヘッドが許容範囲まで改善された場合
