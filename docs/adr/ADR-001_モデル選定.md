# ADR-001：マルチLLMモデル選定

**文書番号**：VCS-ADR-001  
**ステータス**：承認済み  
**作成日**：2026-02-27  
**決定者**：Owner  
**関連要件**：§1.2, §5.1, 実装手順書§11  
**関連ADR**：ADR-004（RAG検索で使用するコンテキスト管理がモデルの長文処理能力に依存）

---

## 背景・課題

本システムはPDCAサイクルを自動で回すために、複数のLLMを役割ごとに使い分ける（§5.1）。各役割には異なる能力軸が要求されるため、単一モデルでは全役割を最適にカバーできない。実装着手時点で利用可能な最新安定版APIの中から、能力軸に最も適合するモデルを選定する必要がある。

---

## 役割別の必須能力軸

| 役割 | 必須能力軸 | 重視事項 |
|------|-----------|---------|
| **PM** | 長文の構造化・タスク分解・リスク列挙 | 日本語での論理的説明 |
| **サブPM/書記** | ドキュメント間の矛盾検出・要約精度 | 長コンテキスト処理（ADR・仕様書全体の把握） |
| **プログラマ** | コードの欠陥検出・セキュリティ観点・型安全性 | 対象言語の精度 |
| **デザイナ** | UI導線の問題検出・アクセシビリティ・情報設計 | 非エンジニア視点での説明能力 |
| **ユーザ** | 混乱・不便・誤操作の発見 | 非エンジニア視点での指摘 |
| **DO担当（実装）** | 指示に忠実な実装・テスト生成・エラー修正 | 対象言語の精度・長いコードへの対応 |

---

## 選択肢

### 案A：単一モデル統一方式

すべての役割を1つの最高性能モデル（例：GPT-5.1）で賄う。

- **利点**：運用が単純、コスト管理が容易、障害点が少ない
- **欠点**：各役割に最適化されない、単一障害点、ベンダーロックイン

### 案B：役割別マルチモデル方式（採用）

役割ごとに最適なモデルを個別に選定する。

- **利点**：各役割で最高品質のレビューが可能、冗長性が高い、ベンダー分散
- **欠点**：運用複雑度が上がる、コスト管理が複雑、モデル間の出力形式統一が必要

---

## 採用理由

**案Bを採用する。**

レビュー品質の最大化がシステムの中核価値であり、各ペルソナが異なる観点で最大限の能力を発揮する必要があるため。ベンダー分散によるリスク低減効果も大きい。

| **PM** | GPT-5.1 | Claude Opus 4 → Gemini 3 Pro | Ollama llama3.3:70b → qwen2.5:32b |
| **サブPM/書記** | Gemini 3 Pro | Claude Opus 4 → GPT-5.1 | Ollama llama3.3:70b → qwen2.5:32b |
| **プログラマ** | Claude Opus 4 | GPT-5.1 → Gemini 3 Pro | Ollama llama3.3:70b |
| **デザイナ** | Gemini 3.1 Pro | GPT-5.1 → Claude Opus 4 | Ollama llama3.3:70b → qwen2.5:32b |

---

## トレードオフ

- マルチモデル方式は運用複雑度とコストが増大するが、LLMゲートウェイ（§4.2）による統一インターフェースで複雑度を吸収する
- フォールバック順は能力軸の類似度に基づき定義。フォールバック時はレビュー品質が低下する可能性がある
- DO担当はクラウドではClaude固定であり、クラウド間フォールバックを設けない（要件定義で確定済み）。ただしクラウド全滅時はローカルLLM（llama3.3:70b）へ退避する（ADR-008）

---

## 撤回条件

- 選定モデルが連続10サイクルでモデル劣化検知閾値（§5.3）を超えた場合、Ownerが切替を判断しADRに記録する
- モデル提供者がAPI提供を停止した場合、フォールバック順に従い即座に切替を実施する
- 新規モデルがベンチマーク上で明確に上回る場合、10サイクル評価の上で段階移行を検討する
