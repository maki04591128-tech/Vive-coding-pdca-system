## 📖 この文書について

> この ADR-008 では、**クラウドAI（インターネット経由）とローカルAI（自前サーバー）を自動で切り替える仕組み**の設計を記録しています。  
> たとえるなら、「商用電源と自家発電の自動切替装置」のようなものです。クラウドAIが使えなくなったら、自動でローカルAIに切り替わって作業を続けます。

## 背景・課題

ADR-001 において役割別マルチモデル方式とフォールバック順を定義した。しかし以下の課題が未解決であった。

- クラウドLLMのインターネット接続障害や API 障害発生時に、手動介入なしでシステムを継続運転させる仕組みが必要
- 開発時やオフライン環境ではコスト削減のためローカルLLMを優先使用したいが、切替の仕組みが未定義
- ADR-001 のフォールバック順はクラウド間の代替を想定しており、クラウド全滅時のローカルLLMへの退避が未定義

---

## 選択肢

### 案A：設定ファイルによる静的切替のみ

設定ファイル（`config/default.yml`）で `preferred_mode: local` に変更することでモードを切り替える。

- **利点**：実装が単純、状態管理が不要
- **欠点**：障害時に手動で設定変更→再起動が必要、自動復旧ができない

### 案B：サーキットブレーカー + ヘルスチェックによる動的切替（採用）

設定ファイルによる手動切替に加え、サーキットブレーカーパターンでクラウド障害を自動検知し、ローカルLLMへ動的にフォールバックする。ヘルスチェッカーが定期的にプロバイダの死活を監視する。

- **利点**：障害時の自動フォールバック、復旧後の自動復帰、無停止運転
- **欠点**：実装複雑度が上がる、サーキットブレーカーの閾値チューニングが必要

### 案C：リトライのみ（フォールバックなし）

クラウド障害時はリトライ（最大2回）のみ行い、失敗時はエラーとして停止する。

- **利点**：実装が最も単純
- **欠点**：ネットワーク障害時にシステム全停止、SLO（§13.2）を満たせない

---

## 採用理由

**案Bを採用する。**

- §13.2 の信頼性要件（縮退モードでの運用継続）を満たすには、自動フォールバック機構が必須
- ADR-001 のクラウド間フォールバック順と組み合わせ、クラウド全滅時はローカルLLMを最終退避先とする
- 開発時のコスト削減ニーズにも `preferred_mode: local` で対応可能
- サーキットブレーカーの状態は監査ログに記録され、障害の追跡可能性を維持する

---

## 設計概要

### フォールバック優先順位

```
1. 役割に割り当てられた主クラウドプロバイダ（ADR-001 選定結果）
2. ADR-001 のフォールバック順に従った代替クラウドプロバイダ
3. 役割に割り当てられたローカルLLMプロバイダ（最終フォールバック）
```

### ローカルLLMプロバイダ選定

ADR-001 のフォールバック順に加え、クラウド全滅時の最終退避先としてローカルLLMを以下の通り選定する。

| プロバイダ名 | モデル | 用途 | 対応役割 |
|-------------|--------|------|---------|
| ollama-llama3.3 | llama3.3:70b | 汎用フォールバック | PM, 書記, プログラマ, デザイナ, ユーザ, DO |
| ollama-qwen2.5 | qwen2.5:32b | 軽量応急用 | PM, 書記, デザイナ, ユーザ |

- **実行基盤**: Ollama（OpenAI 互換 API: `http://localhost:11434/v1`）
- **選定理由**: llama3.3:70b は全役割をカバーできる汎用性と高い日本語能力を持つ。qwen2.5:32b は軽量で応急対応に適するが、コード生成精度の制約からプログラマ・DO担当には割り当てない
- **コスト**: ローカル実行のため $0（ハードウェアコストは別途）

### サーキットブレーカー仕様

| パラメータ | デフォルト値 | 説明 |
|-----------|-------------|------|
| `failure_threshold` | 3 | OPEN遷移する連続失敗回数 |
| `recovery_timeout` | 60秒 | OPEN→HALF_OPEN待機時間 |
| `success_threshold` | 2 | HALF_OPEN→CLOSED に必要な連続成功 |

### ヘルスチェック仕様

| 項目 | 仕様 |
|------|------|
| チェック間隔 | 30秒（本番15秒 / 開発60秒） |
| チェック対象 | インターネット接続 + 各プロバイダAPI疎通 |
| 異常検知時 | サーキットブレーカーへ failure 通知 |

---

## トレードオフ

- ローカルLLMはクラウドLLMに比べてレビュー品質が低下する可能性がある。ローカルフォールバック中はレスポンスに `fallback_used: true` を付与し、品質低下の可能性をログに記録する
- サーキットブレーカーの `recovery_timeout` が短すぎると障害復旧前にクラウドへ再接続を試み無駄なエラーが増える。本番では120秒を推奨
- ローカルLLMが未セットアップの環境では自動フォールバックが機能しない。セットアップガイドとヘルスチェックで事前検知する

---

## 撤回条件

- サーキットブレーカーの誤判定（正常なクラウドLLMをOPENにする）が10サイクル中5回以上発生した場合、閾値パラメータの再調整を行う
- ローカルLLMの品質が許容範囲を下回り、フォールバック時のレビュー品質が実用に耐えない場合は、フォールバックを無効化しクラウド復旧まで停止する方式に変更する
