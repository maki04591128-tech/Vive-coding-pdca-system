## 📖 この文書について

> この文書は、本システムのローカル環境で実施すべき**テスト・デバッグの全検査項目**をまとめた要綱です。  
> たとえるなら、「自動車の車検チェックシート」のようなものです。どこを、なぜ、どう検査して、何が正常なのかを一覧にしています。  
> 各検査項目には具体的なコマンドと期待結果を記載しているので、初めてテストする方でも手順に沿って実施できます。

### 🔰 この文書で使われる主要な用語

| 用語 | かんたんな説明 |
|------|-------------|
| **lint** | コードの書き方のルール違反を自動検出すること（スペルチェックのコード版） |
| **mypy** | Pythonの型チェックツール。変数や関数の「型」の整合性を確認する |
| **ruff** | Pythonの高速lint/formatツール |
| **pytest** | Pythonの自動テスト実行フレームワーク |
| **ユニットテスト** | 関数やクラス単位で正しく動くか確認するテスト |
| **E2Eテスト** | 端から端まで通しで動くか確認するテスト（End-to-End） |
| **レッドチーム** | 攻撃者の視点でセキュリティを検証するテスト |
| **カバレッジ** | テストがソースコードのどれだけの割合を実行したかの指標 |
| **gitleaks** | ソースコードに秘密情報（APIキー等）が混入していないか検出するツール |
| **fixtures** | テストで使う共通のサンプルデータやセットアップ処理 |

## T1：環境構築確認

> **目的**: テストを実施するための前提環境が正しく構築されていることを確認する。  
> 環境に不備があると、後続のすべてのテストが正しく実行できないため、最初に必ず確認する。

### T1-1: Pythonバージョン確認

| 項目 | 内容 |
|------|------|
| **何を** | インストールされているPythonのバージョン |
| **何のために** | pyproject.tomlの `requires-python = ">=3.12"` を満たしているか確認 |
| **どのように** | `python --version` を実行 |
| **正常な結果** | `Python 3.12.x` 以上が表示される |
| **異常な場合** | 3.11以下 → Python 3.12以上をインストールする |

### T1-2: 開発依存パッケージのインストール

| 項目 | 内容 |
|------|------|
| **何を** | 開発用パッケージ一式（pytest, ruff, mypy等）のインストール |
| **何のために** | テスト・lint・型チェックの実行に必要なツールが揃っていることを確認 |
| **どのように** | `pip install -e ".[dev]"` を実行 |
| **正常な結果** | `Successfully installed ...` でエラーなく完了する |
| **異常な場合** | ネットワークエラーまたは依存関係の競合 → pip のアップグレードやvenv再作成 |

### T1-3: パッケージのインポート確認

| 項目 | 内容 |
|------|------|
| **何を** | vibe_pdca パッケージが正しくインポートできること |
| **何のために** | editable install が成功し、ソースパスが正しく設定されていることを確認 |
| **どのように** | `python -c "import vibe_pdca; print(vibe_pdca.__version__)"` を実行 |
| **正常な結果** | バージョン番号（例: `0.1.0`）が表示される |
| **異常な場合** | `ModuleNotFoundError` → `pip install -e ".[dev]"` を再実行 |

### T1-4: 設定ファイルの存在確認

| 項目 | 内容 |
|------|------|
| **何を** | 必要な設定ファイルが存在するか |

> 未使用のimport、長すぎる行、非推奨パターンなどを検出し、コード品質を維持する。

### T2-1: ソースコードのLint

| **何のために** | E（pycodestyle）, F（pyflakes）, W（警告）, I（import順序）, N（命名規約）, UP（Pythonバージョン）, B（バグ危険）, SIM（簡略化） の各ルールへの準拠を確認 |
| **どのように** | `ruff check src/` を実行 |
| **正常な結果** | `All checks passed!` と表示される（指摘件数0） |
| **異常な場合** | 指摘が表示される → `ruff check src/ --fix` で自動修正可能なものを修正後、手動修正 |

| **どのように** | `ruff check tests/` を実行 |
| **正常な結果** | `All checks passed!` と表示される（指摘件数0） |
| **異常な場合** | 同上 |

### T2-3: ソース＋テスト一括Lint（CI同等）

| 項目 | 内容 |
|------|------|
| **何を** | CI パイプラインと同等の lint 実行 |
| **何のために** | ローカルでCI失敗を事前検出するため |

| **正常な結果** | `All checks passed!` と表示される |
| **異常な場合** | 指摘件数と内容を記録し修正 |

| **何のために** | Issue本文/PR本文/README/CI出力/依存関係を経由した攻撃を防御できることを確認 |
| **どのように** | `pytest tests/test_security_redteam.py -v` を実行 |
| **正常な結果** | 全5パターンで注入が無効化され、安全な応答が返る |
| **異常な場合** | サニタイズ処理の不足またはプロンプトテンプレートの変更 |

### T6-2: セキュリティ全般

| 項目 | 内容 |
|------|------|
| **何を** | サンドボックスリソース制限、入力バリデーション、セキュリティ関連機能 |
| **何のために** | 不正入力やリソース枯渇攻撃に対する耐性を確認 |

| **正常な結果** | リソース制限が適切に適用され、不正入力がバリデーションで弾かれる |
| **異常な場合** | バリデーションルールまたはサンドボックス設定の変更 |

### T6-3: 秘密情報スキャン

| 項目 | 内容 |
|------|------|
| **何を** | リポジトリ内のAPIキー・トークン等の秘密情報の有無 |
| **何のために** | コードに秘密情報が混入していないことを確認 |
| **どのように** | `gitleaks detect --source . --config .gitleaks.toml` を実行（gitleaks要インストール） |
| **正常な結果** | `no leaks found` と表示される |
| **異常な場合** | 漏洩検出 → 即座に該当キーを無効化し、`.env` ファイルで管理するよう修正 |

## T8：ドキュメント整合性

> **目的**: ドキュメントの内容がコードの実態と一致していることを確認する。  
> コードを変更した際にドキュメントが更新されていないと、利用者に誤った情報を提供してしまう。

### T8-1: README.md のインストール手順

| 項目 | 内容 |
|------|------|
| **何を** | README.md に記載されたインストールコマンドの動作 |
| **何のために** | 新規参加者が README の手順に従って環境構築できることを確認 |
| **どのように** | README.md の「セットアップ」セクションの各コマンドを実際に実行 |
| **正常な結果** | `pip install -e ".[dev]"`, `pip install -e ".[dev,gui]"`, `pip install -e ".[dev,google]"` が全てエラーなく完了 |
| **異常な場合** | パッケージ名変更やオプション依存の更新漏れ |

### T8-2: .env.example の完全性

| 項目 | 内容 |
|------|------|
| **何を** | .env.example に記載された環境変数がコード内で参照されていること |
| **何のために** | 環境変数の追加・削除がドキュメントに反映されていることを確認 |
| **どのように** | `.env.example` 内の変数名をコード内で検索（`grep -r "VIBE_PDCA_" src/`）|
| **正常な結果** | 全変数名がソースコード内で参照されている |
| **異常な場合** | 環境変数の追加・リネームのドキュメント反映漏れ |

### T8-3: CIパイプラインとの一致

| 項目 | 内容 |
|------|------|
| **何を** | ローカルで実行したコマンドとCI定義（`.github/workflows/ci.yml`）の一致 |
| **何のために** | ローカルで成功してもCIで失敗する、またはその逆が起きないことを確認 |
| **どのように** | CIの4ステージ（lint / type-check / test / secrets-scan）をローカルで再現 |
| **正常な結果** | CIの各ステージと同じコマンド・結果になる |
| **異常な場合** | Pythonバージョン差異、依存パッケージ差異、OS差異 |

**CI再現コマンド一覧**:

```bash
# Stage 1: Lint
ruff check src/ tests/

# Stage 2: Type Check
mypy src/ --exclude 'src/vibe_pdca/gui/'

# Stage 3: Test
pytest --tb=short -q --ignore=tests/test_gui.py

# Stage 4: Secrets Scan（gitleaks要インストール）
gitleaks detect --source . --config .gitleaks.toml
```

## 検査実施のフロー

```
T1 環境構築確認
    ↓ （全てOKの場合のみ次へ）
T2 静的解析（Lint）    ←── 並行実行可 ──→    T3 型チェック
    ↓                                         ↓
T4 ユニットテスト（まず T4-1 の一括テスト、失敗時に個別実行）
    ↓
T5 統合・E2Eテスト
    ↓
T6 セキュリティテスト
    ↓
T7 受入テスト
    ↓
T8 ドキュメント整合性
    ↓
【完了】→ 検査結果報告テンプレートに結果を記入
```

## クイック検査（最小限の確認）

時間がない場合、以下の3コマンドで最小限の検査が可能です（CI同等）:

```bash
# 1. Lint（約5秒）
ruff check src/ tests/

# 2. 型チェック（約30秒）
mypy src/ --exclude 'src/vibe_pdca/gui/'

# 3. 全テスト（約10秒）
pytest --tb=short -q --ignore=tests/test_gui.py
```

**クイック検査の正常結果**:
- ruff: `All checks passed!`
- mypy: `Success: no issues found in N source files`
- pytest: `1573 passed`
