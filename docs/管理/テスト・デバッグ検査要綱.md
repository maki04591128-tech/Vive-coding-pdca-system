## 📖 この文書について

> この文書は、本システムのローカル環境で実施すべき**テスト・デバッグの全検査項目**をまとめた要綱です。  
> たとえるなら、「自動車の車検チェックシート」のようなものです。どこを、なぜ、どう検査して、何が正常なのかを一覧にしています。  
> 各検査項目には具体的なコマンドと期待結果を記載しているので、初めてテストする方でも手順に沿って実施できます。

### 🔰 この文書で使われる主要な用語

| 用語 | かんたんな説明 |
|------|-------------|
| **lint** | コードの書き方のルール違反を自動検出すること（スペルチェックのコード版） |
| **mypy** | Pythonの型チェックツール。変数や関数の「型」の整合性を確認する |
| **ruff** | Pythonの高速lint/formatツール |
| **pytest** | Pythonの自動テスト実行フレームワーク |
| **ユニットテスト** | 関数やクラス単位で正しく動くか確認するテスト |
| **E2Eテスト** | 端から端まで通しで動くか確認するテスト（End-to-End） |
| **レッドチーム** | 攻撃者の視点でセキュリティを検証するテスト |
| **カバレッジ** | テストがソースコードのどれだけの割合を実行したかの指標 |
| **gitleaks** | ソースコードに秘密情報（APIキー等）が混入していないか検出するツール |
| **pip-audit** | Pythonパッケージの既知脆弱性を検出するツール |
| **fixtures** | テストで使う共通のサンプルデータやセットアップ処理 |

## 検査実施のフロー

```
T1 環境構築確認
    ↓ （全てOKの場合のみ次へ）
T2 静的解析（Lint・Format）  ←── 並行実行可 ──→  T3 型チェック
    ↓                                              ↓
T4 ユニットテスト（まず T4-1 の一括テスト、失敗時に個別実行）
    ↓
T5 統合・E2Eテスト
    ↓
T6 セキュリティテスト
    ↓
T7 受入テスト
    ↓
T8 ドキュメント整合性
    ↓
T9 カバレッジ・パフォーマンス
    ↓
T10 GUIテスト（オプション ─ flet環境がある場合のみ）
    ↓
【完了】→ 検査結果報告テンプレートに結果を記入
```

## クイック検査（最小限の確認）

時間がない場合、以下の3コマンドで最小限の検査が可能です（CI同等）:

```bash
# 1. Lint（約5秒）
ruff check src/ tests/

# 2. 型チェック（約30秒）
mypy src/ --exclude 'src/vibe_pdca/gui/'

# 3. 全テスト（約10秒）
pytest --tb=short -q --ignore=tests/test_gui.py
```

**クイック検査の正常結果**:
- ruff: `All checks passed!`
- mypy: `Success: no issues found in N source files`
- pytest: `1573 passed`
