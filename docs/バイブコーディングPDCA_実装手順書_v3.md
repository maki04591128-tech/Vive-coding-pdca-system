## 4. 横断設計方針

### 4.1 エラーハンドリング方針 [C5]

エラー分類（LLM一時/恒久障害・GitHub一時/恒久障害・サンドボックス障害・バリデーションエラー・セキュリティ違反・コスト超過・タイムアウト・監査ログ不整合）ごとにアクション（RETRY/FALLBACK/DEGRADE/STOP/STOP_IMMEDIATE/ESCALATE_HUMAN）をマッピングする。リトライは指数バックオフ、同一失敗は2回で人間承認待ちとする。

### 4.2 GitHub ステートストア設計

PDCAの全状態をIssue/PR/Milestoneで表現（§4.2）。ラベル体系: phase/, status/, priority/, type/, severity/, change-type/, governance/ プレフィクス。

### 4.3 ポリシー設定の階層マージ

`default.yml → 環境別.yml → プロジェクト別.yml` でマージ。緩和変更はA操作承認必須。

### 4.4 JSONスキーマ運用ポリシー [C3]

スキーマは `config/prompts/schemas/*.json` に配置。重要出力は `response_format: "json"` 指定。Pydanticで検証し失敗時は再提出要求（最大1回）。スキーマの版管理・後方互換を維持。

### 4.5 トレーサビリティ設計

TraceLinkモデルでGoal→Milestone→Task→PR→Review→Decisionの双方向追跡を実現。

### 4.6 環境分離方針

dev/stg/prodでGitHub App・LLM APIキー・コスト上限・Discord・監査ログ保管を分離。

### 4.7 RBAC設計 [A1]

4ロール（Owner/Maintainer/Reviewer/Auditor）体系。各ロールの権限をRolePermissionモデルで定義。Web UIミドルウェアとDiscord承認4/4検証に適用。

### 4.8 マルチプロジェクトプロセス分離 [A2]

プロジェクト単位でプロセス起動（共有プロセス禁止）。プロジェクト固有の環境変数注入。

### 4.9 プラグイン設計 [A6]

PluginInterface基底クラスでNotification/CI/Knowledgeプラグインを交換可能・無効化可能に。

### 4.10 再現性・説明可能性設計 [B9]

重要出力にOutputMetadata（モデル名・版・プロンプト版・入力参照・LLMパラメータ: temperature=0.0, seed=42）を記録。
